## 問題 201

Azure SQL Managed Instance でホストされているデータベースがあります。このデータベースには機密情報を格納する Column1 という名前の列があります。Column1 へのアクセスを監査する必要があります。ソリューションは次の要件を満たす必要があります：監査データを少なくとも 18 年間保持すること、管理作業を最小限に抑えること。実行すべき 3 つのアクションはどれですか？（三つ選択）

- A. Azure SQL Auditing を有効にする
- B. 列に対して Always Encrypted を有効にする
- C. 列に機密性分類を適用する
- D. Storage 監査ログ送信先を構成する
- E. Log Analytics 監査ログ送信先を構成する
- F. Transparent Data Encryption (TDE) を有効にする

<details>
<summary>解答・解説</summary>

**解答:** A, C, D

このシナリオでは、Azure SQL Managed Instance の特定の列（Column1）へのアクセスを 18 年間監査するソリューションを構築する必要があります。

正解の組み合わせとその理由：

1. **Azure SQL Auditing を有効にする** - 監査機能の基盤となるもので、データベース操作の記録を開始するために必須です。
2. **列に機密性分類を適用する** - 機密性分類を適用することで、特定の列（Column1）を機密データとして明示的にマークし、監査の対象として特定できます。
3. **Storage 監査ログ送信先を構成する** - Azure Storage は長期保存に適しており、18 年という長期間のデータ保持要件を満たすことができます。また、管理オーバーヘッドが少なく、「管理作業を最小限に抑える」という要件も満たします。

不正解の選択肢：
- **Log Analytics 監査ログ送信先** - 通常、短期間の分析に使用され、18 年間という長期保存には適していません。
- **Always Encrypted** - データの暗号化に関するもので、監査機能ではありません。
- **Transparent Data Encryption (TDE)** - データベース全体の保存データを暗号化するもので、特定の列へのアクセス監査には直接関係しません。

| 機能 | 目的 | 長期監査に適切か |
|---|---|---|
| Azure SQL Auditing | データベース操作の監査を有効化 | はい（基盤機能） |
| 機密性分類 | 特定の列を機密データとしてマーク | はい（監査対象の特定） |
| Storage 監査ログ | 長期間のログ保存 | はい（18年の保持に対応） |
| Log Analytics 監査ログ | 短期間の分析と可視化 | いいえ（長期保存に不向き） |
| Always Encrypted | データの暗号化 | いいえ（監査機能ではない） |
| TDE | データベース全体の暗号化 | いいえ（監査機能ではない） |

</details>

## 問題 202

Azure SQL Database エラスティック プールに 10 個のデータベースがあります。次のアラートを受け取りました。Msg 1132, Level 16, State 1, Line 1 エラスティック プールがストレージ制限に達しました。エラスティック プールに使用されるストレージは (76800) MB を超えることができません。このアラートを解決する必要があります。ソリューションは管理作業を最小限に抑える必要があります。どの 3 つのアクションを実行できますか？それぞれの正解は完全なソリューションを示します。注：各正解の選択は 1 ポイントの値があります。（該当するものをすべて選択）

- A. データベースからデータを削除する
- B. プールからデータベースを削除する
- C. データ圧縮を有効にする
- D. 個々のデータベースを縮小する
- E. エラスティック プールの最大ストレージを増やす

<details>
<summary>解答・解説</summary>

**解答:** A, C, E

このアラートは、Azure SQL Database エラスティック プールがストレージ制限（76800 MB）に達したことを示しています。管理作業を最小限に抑えながらこの問題を解決するには、以下の方法があります：

1. **データベースからデータを削除する** - 不要なデータを削除することで、使用されているストレージ容量を減らすことができます。
2. **データ圧縮を有効にする** - データ圧縮を使用すると、データベース内のデータが占めるスペースを減らすことができます。これは既存のデータに対して実行でき、管理作業を最小限に抑えながらストレージ使用量を削減できます。
3. **エラスティック プールの最大ストレージを増やす** - Azure ポータルまたは PowerShell を使用してエラスティック プールのストレージ制限を増やすことができます。これは簡単な操作で、即時に効果があります。

「プールからデータベースを削除する」は有効な解決策ですが、データベースの移行や再構成が必要になるため、管理作業が増えます。「個々のデータベースを縮小する」は一時的な解決策ですが、データベースの断片化を引き起こす可能性があり、長期的には管理作業が増える可能性があります。

| 解決策 | 管理作業の量 | 即時性 | 長期的効果 |
|---|---|---|---|
| データベースからデータを削除する | 低〜中 | 即時 | 良好 |
| データ圧縮を有効にする | 低 | 実装に時間がかかる場合あり | 良好 |
| エラスティック プールの最大ストレージを増やす | 非常に低い | 即時 | 良好 |
| プールからデータベースを削除する | 高い | 即時 | データベース再配置が必要 |
| 個々のデータベースを縮小する | 中 | 即時 | 断片化の可能性あり |

</details>

## 問題 203

Azure サブスクリプションがあります。Always On 可用性グループを使用する高可用性構成で、Azure 仮想マシン上に SQL Server の 2 つのインスタンスをデプロイする必要があります。ソリューションは次の要件を満たす必要があります：

- フェールオーバーにかかる時間を最小限にする
- フェールオーバー中にプライマリ レプリカへの既存の接続を維持する

どのように構成すべきですか？（一つ選択）

- A. 各仮想マシンを仮想ネットワーク上の同一サブネットに接続し、Standard Azure Load Balancer をデプロイする
- B. 各仮想マシンを仮想ネットワーク上の異なるサブネットに接続し、Basic Azure Load Balancer をデプロイする
- C. 各仮想マシンを仮想ネットワーク上の同一サブネットに接続する
- D. 各仮想マシンを単一の仮想ネットワーク上の異なるサブネットに接続する

<details>
<summary>解答・解説</summary>

**解答:** A

この問題では、Always On 可用性グループを使用する SQL Server 高可用性構成において、フェールオーバー時間の最小化と既存接続の維持という要件を満たすソリューションを選択する必要があります。

Standard Azure Load Balancer は、SQL Server Always On 可用性グループのリスナーとして機能し、クライアント接続をプライマリ レプリカに透過的にリダイレクトします。これにより、フェールオーバー発生時に既存の接続を維持することができます。また、Standard Load Balancer は高速なフェールオーバー検出と切り替えをサポートしているため、フェールオーバー時間を最小限に抑えることができます。

SQL Server Always On 可用性グループの構成では、仮想マシンを同じサブネットに配置することが一般的なプラクティスです。これにより、ネットワーク構成がシンプルになり、レプリカ間の通信が効率化されます。

| 構成 | Load Balancer | サブネット配置 | フェールオーバー時間 | 接続維持 |
|---|---|---|---|---|
| 選択肢 A | Standard | 同一サブネット | 最小化 | 対応 |
| 選択肢 B | Basic | 異なるサブネット | 長い | 部分的対応 |
| 選択肢 C | なし | 同一サブネット | 長い | 非対応 |
| 選択肢 D | なし | 異なるサブネット | 長い | 非対応 |

Basic Load Balancer は、Standard Load Balancer と比較して機能が制限されており、可用性グループのリスナーとしての機能が不十分です。

</details>

## 問題 204

販売データのストリーミング集計を実行するために、どのウィンドウ関数を使用すべきですか？（一つ選択）

- A. Tumbling
- B. Session
- C. Sliding
- D. Hopping

<details>
<summary>解答・解説</summary>

**解答:** A

ストリーミングデータの集計処理では、時間ベースのウィンドウ関数が重要な役割を果たします。Azure SQL Database や Azure Synapse Analytics などでストリーミングデータを処理する際、適切なウィンドウ関数の選択が重要です。

Tumbling ウィンドウは、固定サイズの時間枠で区切られた、重複のない連続したウィンドウです。各イベントは正確に 1 つのウィンドウに属します。これは販売データの集計に最適で、「1 時間ごとの売上合計」や「日次売上集計」などの分析に適しています。Tumbling ウィンドウは実装が単純で、各時間枠のデータが一度だけ処理されるため、販売データの集計処理において最も効率的かつ直感的な選択肢です。

| ウィンドウ関数 | 特徴 | 適用例 |
|---|---|---|
| Tumbling | 固定サイズ、重複なし、連続 | 時間単位の集計（時間別/日別/月別売上） |
| Hopping | 固定サイズ、重複あり、一定間隔でジャンプ | 移動平均、重複期間の分析 |
| Sliding | 固定サイズ、イベント発生ごとに移動 | リアルタイムの移動平均、傾向分析 |
| Session | 可変サイズ、非アクティブ期間で区切る | ユーザーセッション分析、アクティビティ分析 |

</details>

## 問題 205

複数の Azure SQL Database を含むデータ資産があります。データ資産全体でデータのライフサイクルを監査する必要があります。管理作業を最小限に抑える解決策が必要です。何を使用すべきですか？（一つ選択）

- A. Azure Data Factory
- B. Microsoft Purview
- C. Azure SQL Database 用の監査機能
- D. Azure Monitor

<details>
<summary>解答・解説</summary>

**解答:** B

この問題では、複数の Azure SQL Database にまたがるデータのライフサイクルを監査する最適な方法を選ぶ必要があります。

Microsoft Purview は、データガバナンスとデータカタログのためのサービスで、組織全体のデータ資産を自動的に検出し、分類、マッピングすることができます。データのライフサイクルを追跡し、データの系統（データリネージ）を可視化する機能を持っています。複数のデータソースにまたがるデータ資産を一元的に管理できるため、管理作業を最小限に抑えることができます。

他の選択肢について：
- **Azure Data Factory** - データ統合サービスであり、主に ETL（抽出・変換・読み込み）プロセスの自動化に使用されます。データの移動と変換には適していますが、データのライフサイクル全体を監査するための包括的なソリューションではありません。
- **Azure SQL Database 用の監査機能** - 単一のデータベースまたはサーバーレベルでのデータベースイベントを記録するのに役立ちますが、複数のデータソースにまたがるデータのライフサイクル全体を監査するための包括的なソリューションではありません。
- **Azure Monitor** - Azure リソースのパフォーマンスと可用性を監視するためのサービスですが、データのライフサイクルを追跡するための専用機能は提供していません。

| サービス | 主な機能 | データライフサイクル監査に適しているか | 管理作業の負荷 |
|---|---|---|---|
| Microsoft Purview | データガバナンス、カタログ化、分類、データリネージ | はい（複数データソース対応） | 低（一元管理） |
| Azure Data Factory | データ統合、ETL 処理 | 部分的（データ移動のみ） | 中（パイプライン管理が必要） |
| Azure SQL Database 監査 | データベースイベントの記録 | 部分的（単一 DB 内のみ） | 高（DB 毎に設定が必要） |
| Azure Monitor | リソース監視、アラート | いいえ | 中（監視設定が必要） |

</details>

## 問題 206

Azure SQL データベースに対して Azure Monitor から多数のアラートを受信しています。アラートの数を減らす必要があります。使用パターンに長時間にわたって重大な変化がある場合にのみアラートを受信する必要があります。どの 2 つのアクションを実行すべきですか？（二つ選択）

- A. Azure Monitor アラートの評価頻度を 30 分に設定する
- B. アラートロジックのしきい値を動的に設定する
- C. しきい値の感度を低に設定する
- D. アラートロジックのしきい値を静的に設定する
- E. しきい値の感度を高に設定する

<details>
<summary>解答・解説</summary>

**解答:** B, C

この問題では、Azure SQL データベースのアラート数を減らし、長時間にわたる重大な使用パターンの変化に対してのみアラートを受信するための設定を選択する必要があります。

**動的しきい値**は、過去のデータに基づいて自動的にしきい値を調整します。これにより、通常の使用パターンの変動に対してはアラートが発生せず、実際に異常と見なされるパターンに対してのみアラートが発生します。

**しきい値の感度を低に設定する**ことで、小さな変動や一時的な変化ではなく、より大きな偏差が発生した場合にのみアラートがトリガーされます。感度を低くすると、アラートがトリガーされる前により大きな変動が必要になるため、アラート数が減少します。

静的しきい値は固定値であり、使用パターンの変化に適応しないため、この場合は適切ではありません。感度を高に設定すると、より小さな変動でもアラートが発生するため、アラート数が増加してしまいます。

| 設定 | 効果 | 適切か |
|---|---|---|
| 動的しきい値 | 過去のパターンに基づいて自動調整され、通常の変動を無視 | ✓ |
| 静的しきい値 | 固定値で変化に適応しない | ✗ |
| 感度：低 | 大きな変動のみを検出し、アラート数を減らす | ✓ |
| 感度：高 | 小さな変動も検出し、アラート数が増加する | ✗ |
| 評価頻度：30分 | アラート評価の間隔を設定するだけで、パターン認識には影響しない | ✗ |

</details>

## 問題 207

SQL Server のデータベースレベル保護を使用したフェイルオーバー後に利用可能になるのはどれですか？（一つ選択）

- A. データベースと SQL Server Agent ジョブ
- B. データベース内のデータのみ（データベース外の要素は手動で対処する必要がある）
- C. ログイン、データベース、および SQL Server Agent ジョブ
- D. データベースとログインのみ（SQL Server Agent ジョブは手動で再構成する必要がある）

<details>
<summary>解答・解説</summary>

**解答:** B

SQL Server のデータベースレベル保護（データベースミラーリングや Always On 可用性グループなど）を使用したフェイルオーバー後に自動的に利用可能になるのは、基本的にデータベース内のデータのみです。データベース外の要素（ログイン、SQL Server Agent ジョブ、リンクサーバー、証明書など）は自動的には転送されず、手動で対処する必要があります。

フェイルオーバー後にシステムを完全に機能させるためには、データベース外の要素を事前に準備しておくか、フェイルオーバー後に手動で構成する必要があります。これらの要素を自動的に同期するには、サードパーティのツールを使用するか、カスタムスクリプトを作成して定期的に同期する方法があります。

なお、インスタンスレベルの保護（フェイルオーバークラスタリングなど）を使用した場合は、インスタンス全体が別のノードに移動するため、ログインやジョブなどのインスタンスレベルのオブジェクトも保持されます。

| 要素 | フェイルオーバー後の状態 | 対処方法 |
|---|---|---|
| データベース内のデータ | 利用可能 | 自動的に転送される |
| ログイン | 利用不可 | 手動で転送または同期が必要 |
| SQL Server Agent ジョブ | 利用不可 | 手動で転送または同期が必要 |
| リンクサーバー | 利用不可 | 手動で転送または同期が必要 |
| 証明書 | 利用不可 | 手動で転送または同期が必要 |
| サーバーレベルのトリガー | 利用不可 | 手動で転送または同期が必要 |

</details>

## 問題 208

Azure SQL Managed Instance に新しいデータベースが追加されたときのイベントをキャプチャするために、Extended Events モニタリングを構成する必要があります。どのイベントカテゴリを使用すべきですか？（一つ選択）

- A. security
- B. operational
- C. admin
- D. connectivity

<details>
<summary>解答・解説</summary>

**解答:** B

Azure SQL Managed Instance では、Extended Events（XEvents）を使用してさまざまなデータベースイベントをモニタリングできます。新しいデータベースの追加などのインスタンスレベルの管理操作は「operational」カテゴリに分類されます。

「operational」カテゴリは、データベースの作成や削除などのデータベースライフサイクル操作に関連するイベントを含んでいるため、新しいデータベースが追加されたときのイベントをキャプチャするのに最適です。

Extended Events セッションを作成する際は、目的に合ったイベントカテゴリを選択することで、必要なイベントを効率的にキャプチャし、不要なイベントによるオーバーヘッドを避けることができます。

| カテゴリ | 説明 | 使用例 |
|---|---|---|
| operational | インスタンスの操作に関連するイベント | データベースの作成/削除、バックアップ/復元操作 |
| admin | 管理タスクに関連するイベント | 構成変更、管理者アクション |
| security | セキュリティ関連のイベント | ログイン試行、権限変更、監査イベント |
| connectivity | 接続に関連するイベント | 接続の確立/終了、ネットワークイベント |

</details>

## 問題 209

あなたは 2 つの Azure VM を管理しています。VM1 は米国東部リージョンにあり、データベース A を含む SQL Server インスタンスをホストしています。VM2 は米国西部リージョンにあります。VM1 では大量のデータ変更が発生し、データ書き込みの低レイテンシ要件があります。データベース A の可用性グループを構成し、VM2 に二次レプリカを配置する必要があります。どのように構成すべきですか？（一つ選択）

- A. プライマリエンドポイントを TCP://VM1:5022、セカンダリエンドポイントを TCP://VM2:5022 として、可用性モードを非同期に設定する
- B. プライマリエンドポイントを TCP://VM1:445、セカンダリエンドポイントを TCP://VM2:445 として、可用性モードを同期に設定する
- C. プライマリエンドポイントを TCP://VM1:5022、セカンダリエンドポイントを TCP://VM2:5022 として、可用性モードを同期に設定する
- D. プライマリエンドポイントを TCP://VM1:1433、セカンダリエンドポイントを TCP://VM2:1433 として、可用性モードを非同期に設定する

<details>
<summary>解答・解説</summary>

**解答:** A

この問題では、地理的に離れた 2 つのリージョン（米国東部と米国西部）間で SQL Server 可用性グループを構成する必要があります。重要なポイントは以下の通りです：

1. **ポート選択** - SQL Server Always On 可用性グループの標準エンドポイントポートは 5022 です。ポート 445 は SMB ファイル共有用で、ポート 1433 は SQL Server の標準接続ポートですが、可用性グループのエンドポイントには使用しません。

2. **可用性モード** - 問題文では「高いデータ変更量」と「低レイテンシ要件」が指定されています。地理的に離れたリージョン間で同期レプリケーションを使用すると、プライマリレプリカでのトランザクションのコミットが遅延し、パフォーマンスに影響します。

- D. 動的データマスキング (Dynamic Data Masking)

<details>
<summary>解答・解説</summary>

**解答:** A

所有権チェーン（Ownership Chaining）は、SQL Server のセキュリティ機能で、ユーザーがストアド プロシージャを実行する権限を持っていれば、そのプロシージャ内で参照されているテーブルに対する直接的なアクセス権限がなくてもプロシージャを実行できるようにします。

所有権チェーンの仕組みは、ストアド プロシージャとそれが参照するテーブルが同じ所有者（スキーマ）によって所有されている場合、SQL Server はテーブルへのアクセス権限チェックをスキップします。これにより、データベース設計者はユーザーに必要最小限の権限だけを付与しながら、必要な操作を実行できる環境を構築できます。

- **行レベルセキュリティ** - テーブル内の特定の行へのアクセスを制限する機能ですが、テーブル全体へのアクセス権限がない場合のストアド プロシージャ実行には関係ありません。
- **動的データマスキング** - 権限のないユーザーに対して機密データを自動的にマスクする機能ですが、テーブルアクセス権限の迂回には使用できません。

|---|---|---|
| 所有権チェーン | 同一所有者のオブジェクト間でのアクセス権限チェックをスキップ | 可能 |
| 最小権限の原則 | セキュリティ設計の考え方 | 不可能 |
| 行レベルセキュリティ | テーブル内の特定行へのアクセス制限 | 不可能 |
| 動的データマスキング | 機密データの表示制限 | 不可能 |

- これにより、単一のデータセンター障害が発生しても、データベースは引き続き利用可能です

要件との対応：
- 「少なくとも 2 つの読み取り可能なコピー」：Business Critical は通常 3 つの読み取り可能なセカンダリレプリカを提供するため、この要件を満たします
- 「データセンターに障害が発生しても利用可能」：可用性ゾーンによりレプリカが異なるデータセンターに配置されるため、この要件も満たします

| サービスレベル | 読み取り可能なレプリカ | データセンター障害からの保護 | 要件を満たすか |
|---|---|---|---|
| Business Critical + 可用性ゾーン | あり（通常 3 つ） | あり（異なる可用性ゾーンに配置） | はい |
| Business Critical（可用性ゾーンなし） | あり（通常 3 つ） | 限定的（同一データセンター内） | いいえ |
| General Purpose | なし | 限定的 | いいえ |
| Hyperscale | あり（最大 4 つ） | Geo-replication と組み合わせる必要あり | 構成による |

なお、より広範囲の災害に対応するには、異なるリージョンへの geo-replication などの追加対策も検討する必要がありますが、設問の要件は Business Critical + 可用性ゾーンで満たすことができます。

</details>

<details>
<summary>解答・解説</summary>

**解答:** A

- **CPU utilization**：全体的な CPU 使用率を示しますが、キャッシュの問題を特定するには適していません

| Local tempdb percentage | 一時的なストレージの使用率 | クエリの複雑さに依存するが、頻度との直接的な関連性は低い | 同左 |

OPTIMIZE FOR ヒントは、特定のパラメータ値または UNKNOWN 値に対してクエリを最適化するよう最適化プログラムに指示します。これにより、パラメータスニッフィングの問題を解決できます。

OPTIMIZE FOR ヒントを使用すると、特定のパラメータ値または UNKNOWN 値に対して最適化されたプランを生成し、それを再利用できるため、最も適切な解決策です。

RECOMPILE ヒントも効果的ですが、これは毎回クエリプランを再コンパイルするため、「プロシージャ実行後にクエリプランを削除したくない」という要件に反します。

| クエリヒント | 説明 | パラメータスニッフィング問題への適合性 |
|---|---|---|
| OPTIMIZE FOR | 特定のパラメータ値または UNKNOWN 値に対してクエリを最適化する | 高い - パラメータ値に依存せず一貫したプランを生成できる |
| RECOMPILE | 実行ごとにプランを再コンパイルする | 高いが、要件に反する（プラン再利用不可） |
| FORCESEEK | インデックスシークを強制する | 低い - パラメータスニッフィング自体には対処しない |
| OPTION (FAST n) | 最初の n 行を高速で返すことを優先する | 低い - パラメータスニッフィングには対処しない |

</details>

オンライン注文の記録を含むデータセットのスタースキーマを設計しています。各レコードには注文日、納期日、出荷日が含まれています。任意の日付範囲でレコードを照会し、会計カレンダー属性で集計する際に最速のクエリ時間を提供するように設計を確保する必要があります。実行すべき 2 つのアクションはどれですか？（二つ選択）

- E. 日付フィールドに DateTime 型の列を使用する

<summary>解答・解説</summary>

**解答:** B, D

スタースキーマにおける日付関連のクエリパフォーマンスを最適化するためには、適切な日付ディメンションテーブルの設計と日付データの格納方法が重要です。

1. **YYYYMMDD 形式の整数キーを持つ日付ディメンションテーブル**：整数型のキーは DateTime 型よりも比較演算が高速です。YYYYMMDD 形式（例：20250423）は自然な日付順序を維持しながら、整数として効率的に処理できます。

- **DateTime 型のキー**は、比較や結合操作が整数型より計算コストが高くなります。

- **DateTime 型の列**は整数型よりもストレージ要件が大きく、処理も遅くなります。

| 設計アプローチ | パフォーマンスへの影響 | 推奨 |
|---|---|---|
| 整数型キー（YYYYMMDD） | 高速な比較・結合、効率的なインデックス | ✓ |
| DateTime 型キー | 比較・結合が遅い、インデックス効率が低い | ✗ |
| 整数型日付列 | ストレージ効率が良い、高速な処理 | ✓ |
| DateTime 型日付列 | ストレージ要件が大きい、処理が遅い | ✗ |
| 実行時の日付関数 | クエリごとに計算が必要、パフォーマンス低下 | ✗ |

スタースキーマでは、事前計算された値をディメンションテーブルに格納し、ファクトテーブルとの結合を最適化することが重要です。整数型の日付表現はこの目的に最適であり、特に大量データを扱う分析クエリで顕著なパフォーマンス向上をもたらします。

</details>

SQL Server Agent ジョブの失敗通知を設定するには、Database Mail を構成して有効化する必要があります。正しい手順は以下の通りです：

1. **Database Mail を有効化する** - これはメール機能の基盤となるシステムを有効にします
2. **Database Mail アカウントを作成する** - これは SMTP サーバー情報を含む送信メールアカウントです
3. **Database Mail プロファイルを作成する** - これはメール送信に使用するアカウントのコレクションです

これらの設定が完了したら、SQL Server Agent のジョブ通知を構成できます。ジョブのプロパティで通知設定を行い、失敗時にメールを送信するよう指定します。

**SQL Server Agent アラート**は別の機能で、特定のイベントやパフォーマンス条件に基づいて通知を設定するものです。ジョブ失敗の通知には直接必要ありません。

**SQL Server Agent オペレーター**は通知の受信者を定義するもので、通知設定の一部として作成することがありますが、基本的なメール機能の設定には含まれません。

| Database Mail 設定コンポーネント | 目的 | 必須か |
|---|---|---|
| Database Mail 有効化 | メール機能のシステム全体での有効化 | はい |
| Database Mail アカウント | SMTP サーバー接続情報の定義 | はい |
| Database Mail プロファイル | 使用するメールアカウントのグループ化 | はい |
| SQL Server Agent オペレーター | 通知の受信者の定義 | いいえ（ジョブで直接メールアドレスを指定可能） |
| SQL Server Agent アラート | 特定のイベントに対する通知設定 | いいえ（ジョブ失敗通知には不要） |

</details>

<details>
<summary>解答・解説</summary>

## 問題 247

Azure サブスクリプションに West US Azure リージョンでホストされているリソースグループ RG1 があります。新しい Azure SQL サーバー srv1 を作成する必要があります。どの Azure CLI コマンドを実行すべきですか？（一つ選択）

- A. `az sql server create -l westus -g RG1 -n srv1 -u adminuser -p adminpassword`
- B. `az sql db create -g RG1 -s srv1 -n mydb --service-objective S0`
- C. `az sql mi create -g RG1 -n srv1 -l westus -u adminuser -p adminpassword`
- D. `az sql elastic-pool create -g RG1 -s srv1 -n pool1 --capacity 50`

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL サーバーを作成するには、`az sql server create` コマンドを使用します。このコマンドには以下のパラメータが必要です：

- `-l` または `--location`：サーバーを作成する Azure リージョン（この場合は westus）
- `-g` または `--resource-group`：サーバーを作成するリソースグループ（この場合は RG1）
- `-n` または `--name`：作成するサーバーの名前（この場合は srv1）
- `-u` または `--admin-user`：管理者ユーザー名
- `-p` または `--admin-password`：管理者パスワード

他の選択肢が不適切な理由：
- `az sql db create` は Azure SQL データベースを作成するコマンドであり、サーバーではなくデータベースを作成します
- `az sql mi create` は Azure SQL Managed Instance を作成するコマンドであり、SQL サーバーとは異なります
- `az sql elastic-pool create` は Azure SQL Elastic Pool を作成するコマンドであり、これもサーバーではなく既存のサーバー上にリソースを作成します

| Azure SQL リソース | 作成コマンド | 用途 |
|---|---|---|
| SQL サーバー | az sql server create | SQL データベースをホストするための論理サーバー |
| SQL データベース | az sql db create | 単一のデータベースを作成 |
| SQL Managed Instance | az sql mi create | フル機能の SQL Server インスタンスと互換性のあるマネージドインスタンス |
| SQL Elastic Pool | az sql elastic-pool create | 複数のデータベースで共有するリソースプール |

</details>

- B. Elastic Job エージェントを作成する
- C. Elastic Job ターゲットを作成する
- D. Azure SQL Managed Instance を作成する

<details>
<summary>解答・解説</summary>

**解答:** B

Elastic Jobs を使用して複数の Azure SQL Database に対して T-SQL ベースのメンテナンスタスクを実行するには、最初に Elastic Job エージェントを作成する必要があります。

Elastic Job エージェントは、ジョブの実行を調整し、スケジュールを管理するコンポーネントです。エージェントを作成した後、ジョブターゲットを定義し、実行するジョブを作成します。

実装の正しい順序は以下の通りです：
1. Elastic Job エージェント用の Azure SQL Database を作成する（ジョブデータベース）

Elastic Jobs は、複数のデータベースに対して一貫したメンテナンス操作（インデックスの再構築、統計の更新など）を実行する際に非常に便利なサービスです。

| Elastic Jobs の実装ステップ | 説明 |
|---|---|
| Elastic Job エージェントの作成 | ジョブの実行を調整するコンポーネント |

| ターゲットグループの定義 | ジョブを実行する対象のデータベースグループ |

## 問題 249

Azure Data Lake Storage Gen 2 を使用するアプリケーションを開発しています。特定のアプリケーションに対して、限られた期間のみアクセス権を付与するソリューションを推奨する必要があります。推奨事項に何を含めるべきですか？（一つ選択）

- A. 共有アクセス署名（SAS）
- B. Azure Active Directory（Azure AD）ID
- C. ストレージアカウントのファイアウォールルール
- D. アクセスキー

<details>
<summary>解答・解説</summary>

**解答:** A

共有アクセス署名（SAS）は、Azure Storage リソースへの制限付きアクセスを提供するための最適な方法です。SAS を使用すると、特定のアプリケーションに対して、限定された期間だけ特定のアクセス許可を持つトークンを生成できます。

SAS の主な特徴：
1. 時間制限付きアクセス - 有効期限を設定できる
2. 特定の操作のみを許可 - 読み取り、書き込み、削除などの権限を個別に制御
3. 特定のリソースへのアクセスを制限
4. クライアント側の実装が容易

他の選択肢が適切でない理由：
- **Azure AD ID**は長期的なアクセス管理に適していますが、一時的なアクセス付与には複雑すぎます
- **ストレージアカウントのファイアウォールルール**は IP ベースのアクセス制限であり、特定のアプリケーションへの時間制限付きアクセスには適していません
- **アクセスキー**はストレージアカウント全体へのフルアクセスを提供するため、限定的なアクセス権の付与には適していません

| アクセス制御方法 | 時間制限 | 権限の粒度 | セキュリティレベル | 実装の複雑さ |
|---|---|---|---|---|
| 共有アクセス署名（SAS） | あり（設定可能） | 高（操作レベル） | 中〜高 | 低 |
| Azure AD ID | なし（別途管理必要） | 高（RBAC） | 高 | 中〜高 |
| ファイアウォールルール | なし | 低（IP ベース） | 中 | 低 |
| アクセスキー | なし | 低（フルアクセス） | 低 | 低 |

</details>

## 問題 250

Azure サブスクリプションに以下のリソースがあります：10 個の Azure SQL データベース、5 つの Azure SQL マネージドインスタンス、5 つの Azure Virtual Machines 上の SQL Server インスタンス。すべての Azure SQL リソースに対して一元的な監視ソリューションを実装する必要があります。このソリューションは管理作業を最小限に抑える必要があります。このソリューションに含めるべきものは何ですか？（一つ選択）

- A. Azure Monitor for SQL
- B. Log Analytics ワークスペース
- C. Azure SQL Analytics
- D. Azure Database Advisor

<details>
<summary>解答・解説</summary>

**解答:** C

Azure SQL Analytics は、Azure Monitor の一部として提供される監視ソリューションで、複数の Azure SQL リソース（Azure SQL Database、Azure SQL Managed Instance、SQL Server on Azure VM）を一元的に監視するために設計されています。

このソリューションを使用すると、パフォーマンス、可用性、セキュリティなどの重要な指標を単一のダッシュボードで確認でき、管理作業を最小限に抑えることができます。

- **Azure Database Advisor**：Azure SQL Database のパフォーマンス推奨事項を提供するサービスですが、複数の SQL リソースタイプを一元的に監視するためのソリューションではありません

| 監視ソリューション | 対応するリソース | 一元管理 | 管理作業の負荷 |
|---|---|---|---|
| Azure SQL Analytics | Azure SQL DB、SQL MI、SQL on VM | あり | 低い |
| Log Analytics ワークスペース | 多様なリソース | あり | 中程度（カスタム設定が必要） |
| Azure Database Advisor | Azure SQL DB のみ | 限定的 | 中程度 |
| Azure Monitor for SQL | 存在しない独立サービス | - | - |
