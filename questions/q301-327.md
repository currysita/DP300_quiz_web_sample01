## 問題 301

SQL Server on Azure Virtual Machines インスタンス上の可用性グループがあります。既存のデータベースを可用性グループに追加する必要があります。可用性グループのプライマリノードでデータベースの完全バックアップを実行しました。データベースを可用性グループに参加させるために、セカンダリレプリカで実行すべき適切なリストアコマンドはどれですか？（一つ選択）

- A. RESTORE WITH STANDBY
- B. RESTORE WITH RECOVERY
- C. RESTORE WITH NORECOVERY
- D. RESTORE WITH PARTIAL

<details>
<summary>解答・解説</summary>

**解答:** C

可用性グループにデータベースを追加する場合、セカンダリレプリカでは必ず `RESTORE WITH NORECOVERY` オプションを使用してデータベースをリストアする必要があります。

このオプションを使用すると、データベースはリカバリ保留状態（非復旧状態）になり、追加のトランザクションログのリストアが可能になります。可用性グループでは、プライマリレプリカからセカンダリレプリカへのトランザクションの継続的な転送が必要なため、この状態が必須です。

他のオプションが適切でない理由：
- `RESTORE WITH RECOVERY`：データベースを完全に復旧させてオンラインにしますが、追加のログ適用ができません
- `RESTORE WITH STANDBY`：読み取り専用状態でデータベースをリストアしますが、可用性グループの要件を満たしません
- `RESTORE WITH PARTIAL`：部分的なデータベースリストアを行うオプションで可用性グループの要件とは無関係です

| リストアオプション | 状態 | 可用性グループへの参加 | 追加のログ適用 |
|---|---|---|---|
| NORECOVERY | リカバリ保留 | 可能 | 可能 |
| RECOVERY | オンライン | 不可 | 不可 |
| STANDBY | 読み取り専用 | 不可 | 制限付き |
| PARTIAL | 部分復元 | 不可 | 該当なし |

</details>

## 問題 302

Azure SQL Managed Instance の SQL1 と、2 つの Azure Web Apps（App1 と App2）があります。SQL1 上で App2 のクエリが生成する IOPS 数を制限する必要があります。SQL1 で実行すべき操作はどれですか？（二つ選択）

- A. Resource Governor を有効にする
- B. ワークロードグループを作成する
- C. クエリオプティマイザの修正を有効にする
- D. インメモリ OLTP を構成する
- E. パラメータスニッフィングを有効にする
- F. 並列処理の最大次数（MAXDOP）の値を減らす
- G. Database Engine Tuning Advisor を実行する

<details>
<summary>解答・解説</summary>

**解答:** A、B

Resource Governor は Azure SQL Managed Instance で利用可能な機能で、データベースリソースの使用を制限し管理するために使用されます。特定のワークロードやアプリケーションに対して CPU、メモリ、IOPS などのリソース使用量に制限を設けることができます。

Resource Governor を使用するためには、以下の 2 つのステップが必要です：
1. Resource Governor を有効にする
2. ワークロードグループを作成して、特定のアプリケーション（この場合は App2）からのクエリを分類し、リソース制限を適用する

他の選択肢が不適切な理由：
- クエリオプティマイザの修正やパラメータスニッフィングは、クエリのパフォーマンスに影響しますが、IOPS の制限には直接関係しません
- インメモリ OLTP はパフォーマンス向上のための機能であり、IOPS 制限のためのものではありません
- MAXDOP の値を減らすことは並列処理を制限しますが、IOPS を直接制御するものではありません
- Database Engine Tuning Advisor はパフォーマンス向上のための推奨事項を提供しますが、IOPS 制限機能ではありません

| 機能 | 目的 | IOPS 制限に適切か |
|---|---|---|
| Resource Governor | リソース使用量（CPU、メモリ、IOPS）の制限と管理 | はい |
| ワークロードグループ | 特定のワークロードに対するリソース制限の適用 | はい |
| クエリオプティマイザ修正 | クエリプラン生成の最適化 | いいえ |
| パラメータスニッフィング | クエリ最適化のためのパラメータ値の使用 | いいえ |
| インメモリ OLTP | メモリ内処理によるパフォーマンス向上 | いいえ |
| MAXDOP 調整 | 並列処理の制御 | いいえ |
| Database Engine Tuning Advisor | パフォーマンス向上のための推奨事項提供 | いいえ |

</details>

## 問題 303

Always On 可用性グループに 3 つの Microsoft SQL レプリカがあります。各レプリカで 1 時間ごとに実行されるビジネスプロセスがありますが、このプロセスはプライマリレプリカでのみ開始する必要があります。各レプリカの状態に関する必要な情報を取得するには、どのビューをクエリすべきですか？（一つ選択）

- A. sys.dm_hadr_availability_replica_states
- B. sys.dm_hadr_database_replica_cluster_states
- C. sys.availability_groups
- D. sys.dm_hadr_availability_group_states

<details>
<summary>解答・解説</summary>

**解答:** A

`sys.dm_hadr_availability_replica_states` ビューは、可用性グループ内の各レプリカの動的状態情報を提供します。このビューには `role` や `role_desc` 列が含まれており、レプリカがプライマリ（PRIMARY）かセカンダリ（SECONDARY）かを示します。ビジネスプロセスをプライマリレプリカでのみ開始するには、このビューをクエリして現在のプライマリレプリカを特定する必要があります。

他の選択肢について：
- `sys.dm_hadr_database_replica_cluster_states`：データベースレベルのレプリケーション状態に関する情報を提供しますが、レプリカのロール（プライマリかセカンダリか）は含まれません
- `sys.availability_groups`：可用性グループの静的構成情報を提供しますが、どのレプリカがプライマリかという動的な状態情報は含まれません
- `sys.dm_hadr_availability_group_states`：可用性グループ全体の状態情報を提供しますが、個々のレプリカのロール情報は含まれません

| ビュー名 | 提供する情報 | プライマリ/セカンダリ情報 |
|---|---|---|
| sys.dm_hadr_availability_replica_states | レプリカの動的状態情報 | あり（role, role_desc 列） |
| sys.dm_hadr_database_replica_cluster_states | データベースレベルのレプリケーション状態 | なし |
| sys.availability_groups | 可用性グループの静的構成情報 | なし |
| sys.dm_hadr_availability_group_states | 可用性グループ全体の状態情報 | なし |

プライマリレプリカを特定するための典型的なクエリ例：
```sql
SELECT ar.replica_server_name, ars.role_desc
FROM sys.availability_replicas ar
JOIN sys.dm_hadr_availability_replica_states ars
ON ar.replica_id = ars.replica_id
WHERE ars.role_desc = 'PRIMARY';
```

</details>

## 問題 304

Azure SQL Database 論理サーバー Instance1 があります。Instance1 のマスターデータベースに User1 というユーザーを作成しました。User1 が Instance1 に接続されているデータベースを作成および削除できるようにする必要があります。ソリューションは最小権限の原則に従う必要があります。User1 にどのロールを割り当てるべきですか？（一つ選択）

- A. dbmanager
- B. db_owner
- C. loginmanager
- D. db_ddladmin

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database 論理サーバーでは、マスターデータベースに特定のロールが用意されており、サーバーレベルの権限を管理するために使用されます。

`dbmanager` ロールは、Azure SQL Database 論理サーバーのマスターデータベースに存在するサーバーレベルのロールで、データベースの作成と削除の権限を持っています。これは要件を満たす最小権限のロールです。

他の選択肢が不適切な理由：
- `db_owner`：これはデータベースレベルのロールであり、特定のデータベース内のすべての権限を持ちますが、新しいデータベースの作成や削除の権限はありません
- `loginmanager`：これはサーバーレベルのロールですが、ログイン（認証）の管理に関する権限を持ち、データベースの作成や削除の権限はありません
- `db_ddladmin`：これはデータベースレベルのロールで、特定のデータベース内で DDL（Data Definition Language）コマンドを実行する権限を持ちますが新しいデータベースの作成や削除はできません

| ロール | レベル | 主な権限 | データベース作成/削除 |
|---|---|---|---|
| dbmanager | サーバー | データベースの作成と削除 | はい |
| loginmanager | サーバー | ログインの作成と管理 | いいえ |
| db_owner | データベース | 特定 DB の完全な権限 | いいえ |
| db_ddladmin | データベース | 特定 DB 内の DDL 実行 | いいえ |

Azure SQL Database では、サーバーレベルの権限とデータベースレベルの権限を明確に区別することが重要です。データベースの作成と削除はサーバーレベルの操作であるため、サーバーレベルのロールである「dbmanager」が適切な選択肢となります。

</details>

## 問題 305

データベース内のインデックス使用状況を確認するために、どの Dynamic Management View (DMV) を使用すべきですか？（一つ選択）

- A. sys.dm_db_index_operational_stats
- B. sys.dm_db_index_usage_stats
- C. sys.dm_db_missing_index_details
- D. sys.dm_db_index_physical_stats

<details>
<summary>解答・解説</summary>

**解答:** B

`sys.dm_db_index_usage_stats` は、データベース内のインデックスがどのように使用されているかに関する統計情報を提供する Dynamic Management View（DMV）です。このビューを使用すると、各インデックスの検索操作、更新操作、スキャン操作などの使用状況を確認できます。

`sys.dm_db_index_usage_stats` は以下の重要な情報を提供します：
- `user_seeks`：ユーザークエリによるインデックスシークの回数
- `user_scans`：ユーザークエリによるインデックススキャンの回数
- `user_lookups`：ユーザークエリによるインデックスルックアップの回数
- `user_updates`：ユーザークエリによるインデックス更新の回数

この情報を分析することで、頻繁に使用されるインデックスと使用されていないインデックスを特定し、パフォーマンスチューニングやインデックス管理の意思決定に役立てることができます。

各選択肢の機能：

| DMV | 主な用途 |
|---|---|
| sys.dm_db_index_usage_stats | インデックスの使用頻度や種類（検索、更新、スキャンなど）の統計情報を提供 |
| sys.dm_db_index_operational_stats | インデックスの I/O、ロック、ラッチなどの低レベル操作の統計情報を提供 |
| sys.dm_db_missing_index_details | クエリオプティマイザが推奨する作成すべき不足インデックスの詳細情報を提供 |
| sys.dm_db_index_physical_stats | インデックスの物理的な特性（断片化レベルなど）に関する情報を提供 |

</details>

## 問題 306

既存の Microsoft SQL Server データベースからの bacpac ファイルがあります。このファイルを Azure Storage アカウントにコピーしました。既存の Azure SQL Database にこの bacpac ファイルを適用する必要があります。どのコマンドを実行すべきですか？（一つ選択）

- A. az sql db export
- B. az sql db restore
- C. az sql db update
- D. az sql db import

<details>
<summary>解答・解説</summary>

**解答:** D

bacpac ファイルは、データベーススキーマとデータを含む SQL データベースの論理バックアップファイルです。Azure SQL Database に既存の bacpac ファイルを適用（インポート）するには、`az sql db import` コマンドを使用します。

各選択肢の説明：

- `az sql db import`：bacpac ファイルから Azure SQL Database にデータベースをインポートするためのコマンドです。Azure Storage アカウントに保存された bacpac ファイルを指定して実行します。
- `az sql db export`：逆の操作で、Azure SQL Database から bacpac ファイルを作成してエクスポートするコマンドです。
- `az sql db restore`：バックアップからデータベースを復元するコマンドですが、これは bacpac ファイルではなく、自動バックアップまたは長期保存バックアップから復元する場合に使用します。
- `az sql db update`：既存の Azure SQL Database の設定（価格レベル、最大サイズなど）を更新するコマンドであり bacpac ファイルの適用には使用できません。

| コマンド | 用途 | bacpac ファイルの適用に適切か |
|---|---|---|
| az sql db import | bacpac ファイルからデータベースをインポート | はい |
| az sql db export | データベースから bacpac ファイルを作成 | いいえ |
| az sql db restore | バックアップからデータベースを復元 | いいえ |
| az sql db update | データベース設定の更新 | いいえ |

bacpac ファイルのインポートプロセスでは、スキーマが最初に作成され、その後データがロードされます。このプロセスはデータベースのサイズによっては時間がかかる場合があります。

</details>

## 問題 307

PowerShell を使用して、リソースグループ RG1 内に Azure SQL Managed Instance を作成する必要があります。どのコマンドレットを実行すべきですか？（一つ選択）

- A. New-AzSqlServer
- B. New-AzSqlInstance
- C. Set-AzSqlInstance
- D. New-AzSqlDatabase

<details>
<summary>解答・解説</summary>

**解答:** B

Azure SQL Managed Instance を作成するための PowerShell コマンドレットは「New-AzSqlInstance」です。このコマンドレットは、指定したリソースグループ内に Azure SQL Managed Instance を新規作成するために使用します。

各選択肢の機能：

| コマンドレット | 機能 |
|---|---|
| New-AzSqlInstance | Azure SQL Managed Instance を新規作成する |
| New-AzSqlServer | Azure SQL Server を新規作成する（Managed Instance ではない） |
| New-AzSqlDatabase | 既存の Azure SQL Server または Managed Instance 上にデータベースを作成する |
| Set-AzSqlInstance | 既存の Azure SQL Managed Instance の設定を変更する |

「New-AzSqlServer」は Azure SQL Server を作成するためのコマンドレットであり、Managed Instance とは異なるサービスです。「New-AzSqlDatabase」は既存のサーバーまたは Managed Instance 上にデータベースを作成するためのコマンドレットです。「Set-AzSqlInstance」は既存の Managed Instance の設定を変更するためのコマンドレットであり、新規作成には使用できません。

なお、このコマンドレットの実行にはサブネット ID の指定が必要です。これは Azure SQL Managed Instance が仮想ネットワーク内に展開されるためです。

</details>

## 問題 308

インデックスの統計情報を再計算する操作はどれですか？（一つ選択）

- A. インデックスの再構築（Rebuild）
- B. インデックスの再編成（Reorganize）
- C. ファイルグループの圧縮（Shrink）
- D. インデックスの無効化（Disable）

<details>
<summary>解答・解説</summary>

**解答:** A

インデックスの統計情報を完全に再計算するのは「インデックスの再構築（Rebuild）」操作です。インデックスの再構築は、インデックスを完全に削除して再作成するプロセスであり、その過程で統計情報も完全に更新されます。

SQL Server では、インデックスの再構築は `ALTER INDEX REBUILD` コマンドで実行でき、これによりインデックスの統計情報が完全に更新されます。

各操作と統計情報の関係：

| 操作 | 統計情報の再計算 | フラグメンテーション解消 | 説明 |
|---|---|---|---|
| インデックスの再構築 | 完全に再計算 | 完全に解消 | インデックスを削除して再作成 |
| インデックスの再編成 | 部分的な更新のみ | 部分的に解消 | ページの物理的な再整理 |
| ファイルグループの圧縮 | 影響なし | 悪化する可能性あり | 未使用領域の削減 |
| インデックスの無効化 | 影響なし | 影響なし | インデックスを無効化 |

「インデックスの再編成（Reorganize）」は、インデックスのフラグメンテーションを解消するために物理的なページを再整理しますが、統計情報の完全な再計算は行いません。「ファイルグループの圧縮（Shrink）」は、データベースファイルやファイルグループの未使用領域を削減するための操作であり、インデックスの統計情報とは直接関係ありません。「インデックスの無効化（Disable）」は、インデックスを無効にする操作であり、統計情報の再計算は行いません。

</details>

## 問題 309

ResearchDB1 の認証を実装する必要があります。このソリューションはセキュリティとコンプライアンス要件を満たす必要があります。実装の一部として何を実行すべきですか？（一つ選択）

- A. CREATE USER と FROM EXTERNAL PROVIDER 句を実行する
- B. CREATE LOGIN と FROM WINDOWS 句を実行する
- C. CREATE USER と FROM CERTIFICATE 句を実行する
- D. CREATE USER と ASYMMETRIC KEY 句を実行する

<details>
<summary>解答・解説</summary>

**解答:** B

Windows 認証（統合認証）は、SQL Server で最も安全な認証方法の一つとされています。この方法では、Active Directory の資格情報を使用してデータベースにアクセスするため、パスワードがネットワーク上で送信されることはありません。

`CREATE LOGIN` ステートメントと `FROM WINDOWS` 句を組み合わせることで、Windows 認証を使用するログインを作成できます。これにより、Windows のセキュリティ機能（パスワードポリシー、アカウントロックアウトなど）を活用できます。

他の選択肢が不適切な理由：
- `FROM EXTERNAL PROVIDER` 句は Azure AD 認証に使用されますが、オンプレミス環境では適していません
- `FROM CERTIFICATE` 句は証明書ベースの認証に使用されますが、一般的なユーザー認証には複雑すぎます
- `ASYMMETRIC KEY` 句は非対称キーを使用した認証に使用されますが、通常のユーザー認証には向いていません

| 認証方法 | 使用するコマンド | 主な用途 | セキュリティレベル |
|---|---|---|---|
| Windows 認証 | CREATE LOGIN FROM WINDOWS | 一般的なユーザー認証 | 高 |
| SQL 認証 | CREATE LOGIN WITH PASSWORD | レガシーアプリケーション | 中 |
| 証明書認証 | CREATE USER FROM CERTIFICATE | サービス間認証 | 高 |
| 非対称キー認証 | CREATE USER FROM ASYMMETRIC KEY | 暗号化操作 | 高 |
| Azure AD 認証 | CREATE USER FROM EXTERNAL PROVIDER | クラウド環境 | 高 |

Windows 認証は、パスワードハッシュがネットワーク上で送信されないため、セキュリティとコンプライアンス要件を満たす最適な選択肢です。

</details>

## 問題 310

オンプレミスの Microsoft SQL Server 2019 インスタンスに DB1 という名前のデータベースがあります。Azure サブスクリプションには SQLDB1 という名前の Azure SQL データベースがあります。DB1 を SQLDB1 にレプリケーションする必要があります。どのタイプのレプリケーションを使用すべきですか？（一つ選択）

- A. スナップショット
- B. トランザクション
- C. マージ
- D. ピアツーピア

<details>
<summary>解答・解説</summary>

**解答:** B

オンプレミスの SQL Server から Azure SQL Database へのレプリケーションを行う場合、トランザクションレプリケーションが最適な選択肢です。

トランザクションレプリケーションは、SQL Server から Azure SQL Database へのデータ移行やハイブリッド環境での継続的なデータ同期に対応しています。このレプリケーション方式では、パブリッシャー（オンプレミス SQL Server）で発生したデータ変更をキャプチャし、それをサブスクライバー（Azure SQL Database）に適用します。

Azure SQL Database はパブリッシャーとしては機能しませんが、サブスクライバーとしては機能します。そのため、オンプレミス SQL Server から Azure SQL Database への一方向のトランザクションレプリケーションが可能です。

| レプリケーション方式 | 特徴 | オンプレミスから Azure SQL への適合性 |
|---|---|---|
| トランザクション | 変更データのみを継続的に転送、低遅延、高い整合性 | 最適（Azure SQL Database への公式サポート有り） |

- B. ホストキャッシュを「読み取り専用」に設定する
- C. Storage Spaces を使用して複数のディスクをストライプ化する
- D. Premium SSD ディスクを使用する
- E. Write Accelerator ディスクを使用する

<details>
<summary>解答・解説</summary>

**解答:** A、C

SQL Server のデータファイルに最高のパフォーマンスを確保するには、ホストキャッシュを「なし」に設定し、Storage Spaces で複数ディスクをストライプ化する組み合わせが最適です。

- **ホストキャッシュを「なし」に設定する**：SQL Server のデータファイルはキャッシュなしに設定することが推奨される。データファイルは書き込み操作が多いため、キャッシュを使用するとクラッシュ時にデータ損失のリスクが高まる。キャッシュなしにすることでデータの整合性を確保できる。
- **Storage Spaces を使用して複数のディスクをストライプ化する**：複数ディスクを束ねてストライプ構成にすることで、I/O 操作を複数ディスクに分散させ、スループットを向上させる。

- ホストキャッシュを「読み取り専用」に設定する：ログファイルには適しているが、データファイルには最適ではない
- Premium SSD ディスクを使用する：高 IOPS を提供するが、ストライプ化との組み合わせに比べてパフォーマンス向上の観点では補助的
- Write Accelerator ディスク：プレミアムストレージを使用する SQL ログファイルに適しているが、データファイルには最適化されていない

| ストレージ設定 | データファイル | ログファイル | 理由 |
|---|---|---|---|
| ホストキャッシュ | なし | 読み取り専用 | データファイルはキャッシュなしで整合性確保、ログファイルは読み取りパフォーマンス向上 |
| ディスク構成 | ストライプ化 | 単一ディスク | データファイルは I/O 分散のためストライプ化、ログは順次書き込みのため単一で十分 |
| Write Accelerator | 不要 | 推奨 | ログファイルの書き込みレイテンシ削減に効果的 |
| Premium SSD | 推奨 | 推奨 | 高い IOPS、低レイテンシを提供 |

オンプレミスに 2 つの Microsoft SQL Server 2019 インスタンス（SQL1 と SQL2）があります。SQL1 上のデータベースを Azure に移行する必要があります。移行ソリューションは次の要件を満たす必要があります：移行先のサービスはリンクサーバー接続を使用して SQL2 と通信できること、管理作業を最小限に抑えること。データベースのホスト先として何を使用すべきですか？（一つ選択）

- B. Azure SQL Managed Instance
- C. Azure SQL Database シングルデータベース
- D. Azure Virtual Machines 上の SQL Server

<details>
<summary>解答・解説</summary>

**解答:** B

この問題では、オンプレミス SQL Server 環境から Azure への移行において、リンクサーバー接続のサポートと管理作業の最小化が要件となっています。

Azure SQL Managed Instance は、オンプレミスの SQL Server とほぼ 100% の互換性を持ち、リンクサーバー接続をネイティブにサポートしています。これにより、移行したデータベースから SQL2 へのリンクサーバー接続を維持できます。また、PaaS サービスとして提供されるため、基盤となるインフラストラクチャの管理が不要で、管理作業を最小限に抑えることができます。

| Azure SQL サービス | リンクサーバーサポート | 管理作業レベル | 移行の容易さ |
|---|---|---|---|

- E. データベースの総数 × データベースごとの平均 CPU 使用率

SQL Server のパフォーマンス問題を調査しています。どの待機タイプが過剰な CPU 消費を示す指標となりますか？（一つ選択）

- A. CXPACKET
- B. SOS_SCHEDULER_YIELD
- C. PAGEIOLATCH_SH
- D. RESOURCE_SEMAPHORE

<details>
<summary>解答・解説</summary>

**解答:** B

SQL Server の待機タイプは、データベースエンジンがリソースを待機している理由を示す重要な診断情報です。

SOS_SCHEDULER_YIELD は、スレッドが自発的に CPU を放棄し、他のスレッドが実行できるようにしていることを示します。これは通常、CPU リソースの競合が発生している場合に見られ、過剰な CPU 消費の明確な指標です。スレッドが CPU 量子（4 ミリ秒）を使い切ると、スケジューラは他のスレッドに実行機会を与えるためにこのスレッドを一時停止します。

他の選択肢の待機タイプは異なるリソース問題を示します：

| 待機タイプ | 意味 | 関連するリソース問題 |
|---|---|---|
| SOS_SCHEDULER_YIELD | スレッドが CPU を自発的に放棄 | CPU 競合/過剰消費 |
| CXPACKET | 並列クエリ実行における同期待機 | 並列処理の非効率性 |
| PAGEIOLATCH_SH | データページの物理的な読み取り待機 | ディスク I/O のボトルネック |
| RESOURCE_SEMAPHORE | メモリ許可の待機 | メモリ不足 |

CXPACKET は並列クエリ実行に関連し、必ずしも CPU 問題を示すわけではありません。PAGEIOLATCH_SH はディスク I/O のボトルネックを示し、RESOURCE_SEMAPHORE はメモリ許可の問題を示します。

SQL Server のパフォーマンス問題を診断する際は、`sys.dm_os_wait_stats` DMV を使用して待機統計を分析し、どのリソースがボトルネックになっているかを特定することが重要です。
