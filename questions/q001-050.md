| D. Column5: decimal(10,2) | decimal は実際にはサポート対象。ただし問題文は「二つ選択」であり、A・C が最も一般的な正解として提示されている | ❌（試験想定の正解外） |

## 問題 7

Azure Synapse Analytics でエンタープライズデータウェアハウスに Customers というテーブルがあります。このデータウェアハウスには、クレジットカード情報を含む Customers テーブルへの特定ユーザーからのアクセスを制限するリソーションを提供する必要があります。このソリューションを実装するために何が必要ですか？（一つ選択）

- A. データマスキング
- B. 行レベルセキュリティ
- C. 列レベルセキュリティ
- D. Always Encrypted

<details>
<summary>解答・解説</summary>

**解答:** A

データマスキングは、特定のユーザーが機密データ（クレジットカード番号など）にアクセスする際に、データを完全に制限せず、マスク（隠す）した形で見せる機能。これにより「すべての列へのアクセスを制限」せず閲覧できるようにしながら保護できる。

| 機能 | 特定列へのアクセス制限 | DBA が閲覧可 | 管理者のアクセスを制限 |
|---|---|---|---|
| データマスキング | はい（マスク） | はい（ランダム値で表示） | いいえ（管理者は制限不可） |
| 行レベルセキュリティ | 行単位でアクセスを制限 | はい（特定行を除く） | はい |
| 列レベルセキュリティ | 特定列へのアクセスを制限 | はい（列全体へのアクセス制限） | はい |
| Always Encrypted | データを暗号化して保存 | はい（復号化権限があれば） | 部分的（復号化権限がなければ保護） |

</details>

## 問題 10

SQL Server の Resource Governor において、システムリソースに対する制限を構成できるコンポーネントはどれですか？（一つ選択）

- A. Resource Pool
- B. Classifier Function
- C. Workload Group
- D. Resource Monitor

<details>
<summary>解答・解説</summary>

**解答:** A

SQL Server Resource Governor は、SQL Server インスタンス上のリソース消費を管理するための機能。主に3つのコンポーネントで構成される。

1. **Resource Pool**: CPU やメモリなどのシステムリソース（CPU使用率・メモリ使用量など）の割り当て制限を定義するコンポーネント。各リソースプールには最小/最大 CPU 使用率やメモリ使用量などの制限を設定できる
2. **Workload Group**: ユーザーリクエストをグループ化し、特定のリソースプールに割り当てるコンポーネント。リソース制限を直接設定するのではなく、リソースプールに関連付けられる
3. **Classifier Function**: 着信リクエストを適切なワークロードグループに振り分けるためのユーザー定義関数

| コンポーネント | 主な役割 | リソース制限の設定 |
|---|---|---|
| Resource Pool | CPU やメモリなどのリソース割り当て制限を定義 | 可能（直接設定） |
| Workload Group | ユーザーリクエストのグループ化とリソースプールへの割り当て | 不可（リソースプールに関連付け） |
| Classifier Function | リクエストの分類とワークロードグループへの割り当て | 不可 |
| Resource Monitor | リソース使用状況の監視（監視のみ） | 不可 |

</details>

## 問題 11

SQL Server on Azure Virtual Machine があります。SQL Server インスタンスに障害が発生した場合でも、以下の要件を満たす必要があります。

- システムデータベースを含むすべてのデータベースを保護する
- SQL Server Agent ジョブを保護する
- ダウンタイムを最小限に抑える
- フェールオーバープロセスを自動的に実行する

推奨事項に何を含めるべきですか？（一つ選択）

- A. トランザクションレプリケーション
- B. Always On フェールオーバークラスターインスタンス
- C. ログ配布
- D. データベースミラーリング

<details>
<summary>解答・解説</summary>

**解答:** B

Always On フェールオーバークラスターインスタンス（FCI）が正解。以下の理由から要件を満たす最適なソリューション：

1. **システムデータベースを含むすべてのデータベースの保護**: FCI はインスタンスレベルの保護を提供するため、システムデータベース（master、msdb、model、tempdb）を含むすべてのデータベースが自動的に保護される
2. **SQL Server Agent ジョブの保護**: FCI はインスタンス全体を保護するため、SQL Server Agent ジョブも自動的に保護される
3. **最小限のダウンタイム**: FCI は自動フェールオーバーをサポートし、クラスターサービスがノードの障害を検出すると自動的にフェールオーバーを開始する
4. **自動フェールオーバー**: FCI はクラスターサービスを使用して自動フェールオーバーを提供する

| ソリューション | システム DB の保護 | SQL Agent ジョブの保護 | 自動フェールオーバー | 備考 |
|---|---|---|---|---|
| Always On フェールオーバークラスターインスタンス | はい | はい | はい | インスタンスレベルの保護を提供 |
| トランザクションレプリケーション | いいえ | いいえ | いいえ | データ配布に適しているが、HA/DR ソリューションとしては不十分 |
| ログ配布 | いいえ | いいえ | 手動フェールオーバーが必要 | — |
| データベースミラーリング | いいえ | いいえ | 条件付き | システム DB をサポートせず、SQL Server 2022 では非推奨 |

Always On 可用性グループ（AG）は選択肢にありませんが、これはデータベースレベルの保護を提供し、システムデータベースは保護しないため、この要件には適していません。

FCI の実装には共有ストレージが必要であり、Azure では Azure 共有ディスク、Premium ファイル共有、または Azure Storage Space Direct（S2D）などのオプションが利用できます。

</details>

## 問題 12

2 つの 100GB データベースを Azure に移行し、ワークロードに基づいて動的にスケーリングを行いながらダウンタイムを最小限にする必要があります何を使用すべきですか（一つ選択）

- A. Azure SQL Database エラスティック プール
- B. Azure SQL Database シングルデータベース
- C. Azure SQL Managed Instance
- D. Azure Virtual Machines 上の SQL Server

<details>
<summary>解答・解説</summary>

**解答:** A

複数のデータベースを効率的に管理し、動的スケーリングを行いながらダウンタイムを最小限に抑えるのは、Azure SQL Database エラスティック プールが最適。

| オプション | 動的スケーリング | ダウンタイム | 複数 DB 管理 | 備考 |
|---|---|---|---|---|
| Azure SQL Database エラスティック プール | 自動 | なし | 最適 | 複数 DB で共有リソースを効率的に利用 |
| Azure SQL Database シングルデータベース | 自動 | なし | 個別管理 | 各 DB を個別に管理するため非効率 |
| Azure SQL Managed Instance | 手動 | 発生する場合あり | 可能 | インスタンスレベルでのスケーリング |
| VM 上の SQL Server | 手動 | 発生する | 可能 | インフラ管理の負担が大きい |

Azure SQL Database シングルデータベースも動的スケーリングとダウンタイムなしのスケーリングをサポートしていますが、複数のデータベースを個別に管理する必要があり、リソースの共有ができないため効率的ではありません。

</details>

## 問題 13

SQL Managed Instance の地理的フェイルオーバーを実装するために使用すべきオプションはどれですか？（一つ選択）

- A. Distributed Availability Groups
- B. Auto-failover groups
- C. Always On 可用性グループ
- D. Active geo-replication

<details>
<summary>解答・解説</summary>

**解答:** B

SQL Managed Instance の地理的フェイルオーバーを実装するための正しいオプションは「Auto-failover groups（自動フェールオーバーグループ）」。

Auto-failover groups は、Azure SQL Managed Instance の複数のインスタンス間でデータベースをレプリケートし、自動的に地理的フェイルオーバーを行うための機能。これにより、リージョン全体の障害が発生した場合でも、セカンダリリージョンへの自動フェールオーバーが可能になる。

他の選択肢が不適切な理由：
- **Distributed Availability Groups**: オンプレミスの SQL Server で使用される機能であり、SQL Managed Instance には適用されない
- **Always On 可用性グループ**: 主に SQL Server の機能であり、SQL Managed Instance では直接この形式では使用できない
- **Active geo-replication**: Azure SQL Database で使用される機能だが、SQL Managed Instance では使用できない

| 機能 | 適用対象 | 地理的フェイルオーバー対応 | SQL Managed Instance 対応 |
|---|---|---|---|
| Auto-failover groups | SQL Database、SQL Managed Instance | はい | はい |
| Active geo-replication | SQL Database | はい | いいえ |
| Always On 可用性グループ | SQL Server | はい | いいえ（直接的には） |
| Distributed Availability Groups | SQL Server | はい | いいえ |

SQL Managed Instance では、Auto-failover groups を使用することで、プライマリインスタンスとセカンダリインスタンス間の自動同期を設定し、障害発生時に自動的にフェールオーバーするポリシーを定義できます。これにより、地理的に分散した DR（ディザスタリカバリ）構成を実現できます。

| インデックスタイプ | 最適なユースケース | データ圧縮 | 分析クエリのパフォーマンス |

| クラスター化列ストア | データウェアハウス、大規模分析 | 非常に高い | 優れている |

## 問題 18

Microsoft SQL Server 2019 インスタンスがあります。バックアップメンテナンスソリューションを構成しています。データベースを任意の時点に復元できるようにする必要があります。何を構成すべきですか？（一つ選択）

**解答:** A

データベースを任意の時点（Point-in-Time）に復元するには、完全復旧モデル（Full Recovery Model）の構成が必要。

完全復旧モデルでは、すべてのトランザクションがログに記録されるため、任意の時点への復元が可能になる。完全復旧モデルを使用する場合：

2. 定期的に差分バックアップを取る（任意）
3. トランザクションログバックアップを定期的に取る

| 単純復旧モデル | 最小限（操作間のみ） | 不可 | 完全・差分バックアップのみ |

SQL Server で任意の時点への復元を実現するには、完全復旧モデルの構成と適切なバックアップ戦略（完全バックアップ、差分バックアップ、トランザクションログバックアップ）の組み合わせが必要。

SQL Server 2019 でデータ分類情報はどこに保存されますか？（一つ選択）

- B. 各オブジェクトの拡張プロパティに保存される
- C. sys.all_objects カタログビューに保存される

<details>
<summary>解答・解説</summary>

**解答:** A

SQL Server 2019 では、データ分類情報は `sys.sensitivity_classifications` カタログビューに保存される。このカタログビューは、データベース内の列に適用されたデータ分類ラベルとその情報を格納するために使用される。

データ分類機能は、組織のデータガバナンスポリシーに従ってデータベース内の機密データを識別し、ラベル付けするために使用される。これにより、機密データの発見、追跡、保護が容易になる。

| カタログビュー / 保存場所 | 説明 |
|---|---|
| sys.sensitivity_classifications | データ分類情報を格納する正しいカタログビュー。列のラベル、情報タイプ、ランクなどの分類メタデータを含む |

データ分類情報を確認するには、以下のようなクエリを実行できる：

```sql
SELECT s.name AS schema_name,
       o.name AS table_name,
       c.name AS column_name,
       sc.label,
       sc.information_type,
       sc.rank
FROM sys.sensitivity_classifications sc
JOIN sys.objects o ON sc.major_id = o.object_id
JOIN sys.schemas s ON o.schema_id = s.schema_id
JOIN sys.columns c ON sc.major_id = c.object_id
AND sc.minor_id = c.column_id;
```

- A. インデックスを使用するクエリは、より多くの CPU リソースを消費します
- B. インデックスを使用するクエリは、より少ないデータページを取得します
- C. インデックスはより少ないディスク容量を消費します
- D. インデックスを使用するクエリは、より多くのディスク I/O を消費します
- E. インデックスはより多くのメモリを消費します

<details>

**解答:** A、B

1. **より高い圧縮率**: アーカイブ圧縮は、標準の列ストア圧縮よりも高い圧縮率を実現する。これによりインデックスはより少ないディスク容量を消費する（選択肢 C の逆が正しい）

4. **I/O 効率の向上**: データが高度に圧縮されているため、ディスクからの読み取り I/O は減少する（選択肢 D の逆が正しい）

|---|---|---|
| 圧縮率 | 標準 | より高い |
| ディスク使用量 | 標準 | より少ない |
| CPU 使用率 | 標準 | より高い |
| データページ数 | 標準 | より少ない |
| 適用シナリオ | 一般的なワークロード | アクセス頻度の低いデータ |

アーカイブ圧縮を有効にすると、クエリはより多くの CPU リソースを消費し、より少ないデータページを取得することになる。

## 問題 28

SQL Server 2019 を実行している Azure 仮想マシンがあります。SQL Server インスタンス内のクエリのパフォーマンス問題をトラブルシューティングしています。詳細情報を収集するために、sys.dm_exec_requests に対してクエリを実行したところ、wait_type が PAGELATCH_UP で、wait_resource が 2:3:905856 であることがわかりました。システムパフォーマンスを向上させる必要があります。解決策：追加の tempdb ファイルを作成します。この解決策は目標を達成しますか？（一つ選択）

- A. はい
- B. いいえ

<details>
<summary>解答・解説</summary>

**解答:** A

wait_resource の値「2:3:905856」の意味：
- 2: データベース ID（この場合は tempdb）
- 3: ファイル ID
- 905856: ページ ID

PAGELATCH_UP は、メモリ内のデータページに対する更新ラッチの競合を示している。特に tempdb（データベース ID=2）でこの問題が発生している場合、多くの同時実行セッションが tempdb の同じページに対して競合していることを意味する。

tempdb ファイルを追加することでこの問題を解決できる理由：
1. 複数の tempdb ファイルを作成すると SQL Server はそれらのファイル間でページの割り当てを分散する
2. これにより、同じページに対する競合（PAGELATCH）が減少する
3. 特に tempdb は多くのワークロードで共有されるため、ファイルを追加することでホットスポットを減らせる

| 待機タイプ | 意味 | 一般的な解決策 |
|---|---|---|
| PAGELATCH_UP | メモリ内ページの更新ラッチ競合 | tempdb ファイルの追加、ファイルの均等なサイズ設定 |
| PAGELATCH_SH | メモリ内ページの共有ラッチ競合 | tempdb ファイルの追加、ファイルの均等なサイズ設定 |
| PAGEIOLATCH | ディスク I/O 関連のラッチ | ストレージパフォーマンスの向上 |

Microsoft 推奨のベストプラクティスでは、tempdb ファイルの数は CPU コア数と同じにするか（8コアまで）、8コア以上の場合は 8 ファイルから始めて必要に応じて調整することが推奨されている。また、すべての tempdb ファイルは同じサイズで作成し、自動成長設定も同じにすることが重要。

</details>

<summary>解答・解説</summary>

**解答:** A

ARM テンプレートでは、リソースのデプロイ順序を制御するために依存関係を明示的に定義する必要がある。`dependsOn` プロパティは、あるリソースが別のリソースに依存していることを宣言するために使用される。

| Azure SQL Auditing | セキュリティ監査とコンプライアンス | SQL Insights とは別の機能 |

## 問題 49

オンプレミスネットワークには、db1 という名前のデータベースをホストする Microsoft SQL Server 2016 サーバーがあります。Azure サブスクリプションを持っており、db1 を Azure SQL Managed Instance に移行する計画です。SQL Managed Instance を作成する必要があります。ディスクレイテンシーを最小限に抑える必要があります。どのサービスティアを使用すべきですか？（一つ選択）

- A. General Purpose
- B. Business Critical
- C. Standard
- D. Hyperscale

<details>
<summary>解答・解説</summary>

**解答:** B

Business Critical ティアは、ディスクレイテンシーを最小限に抑えるために設計されている。データベースエンジン、そのファイル（データとログ）、tempdb がローカル SSD に直接配置されるため、I/O 操作のレイテンシーが大幅に削減される。また、インメモリ OLTP テクノロジーもサポートしており、メモリ内処理によってさらにパフォーマンスが向上する。

| サービスティア | ストレージタイプ | レイテンシー | 用途 |
|---|---|---|---|
| Business Critical | ローカル SSD | 最小 | 低レイテンシーが必要なミッションクリティカルなワークロード |
| General Purpose | リモートストレージ | 中程度 | 一般的なビジネスワークロード |
| Hyperscale | — | — | Azure SQL Managed Instance では利用不可 |
| Standard | — | — | Azure SQL Managed Instance のサービスティアとしては存在しない |

General Purpose ティアはリモートストレージを使用するため、Business Critical と比較するとレイテンシーが高くなる。

</details>

## 問題 50

Azure Synapse Analytics の専用 SQL プールに Table1 というテーブルがあります。Azure Data Lake Storage Gen2 の container1 というコンテナ内のファイルのデータは Table1 に格納され、Table1 はそのファイルのサービングレイヤーになっています。container1 にリンクされたファイルの DateTime を Table1 の追加列として格納する必要があります。解決策：Azure Data Factory パイプラインに、DateTime を持つファイルの Get Metadata アクティビティを追加し、このアクティビティの実行後に Table1 へ格納する。この解決策は目標を達成しますか？（一つ選択）

- A. はい、Get Metadata アクティビティでファイルのメタデータを取得し Table1 に格納できます
- B. いいえ、代わりに Copy Activity を使用し、ソース設定で DateTime を追加列として指定する必要があります
- C. いいえ、代わりに Get Metadata アクティビティの結果をパラメータとして使用し、Stored Procedure アクティビティと組み合わせる必要があります

<details>
<summary>解答・解説</summary>

**解答:** B

Get Metadata アクティビティはファイルのメタデータ（作成日時、最終更新日時、サイズなど）を取得できるが、単独ではテーブルにデータを挿入する際に自動的にその情報を列として追加することはできない。Get Metadata アクティビティは単にメタデータを取得するだけで、データの移動や変換は行わない。

正しい解決策は、Copy Activity を使用し、そのソース設定で追加列として DateTime を指定すること。Copy Activity では「Additional columns」設定を使用して、ソースデータに存在しない列を追加できる。この設定では、固定値、ファイルのメタデータ（`@filePath`、`@timestamp` 等）、またはパイプラインのパラメータや変数を値として使用できる。

| アクティビティ | 機能 | データ移動/変換 | メタデータを列として追加可能 |
|---|---|---|---|
| Get Metadata | ファイルのメタデータを取得 | いいえ | いいえ（単独では不可） |
| Copy Activity | データのコピーと変換 | はい | はい（Additional columns 設定で可能） |
| Stored Procedure | SQL ストアドプロシージャの実行 | いいえ | 条件付き（SP 内でロジック実装が必要） |

</details>
