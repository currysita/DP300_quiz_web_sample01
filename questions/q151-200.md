## 問題 151

SalesSQLDb1 のフェイルオーバー後、SalesSQLDb1App1 からデータベースへのアクセスを継続するために何をすべきですか？（一つ選択）

- A. SalesSQLDb1App1 の接続文字列を更新する
- B. SalesSQLDb1 のファイアウォールルールを更新する
- C. SalesSQLDb1 を書き込み可能として構成する
- D. フェイルオーバー後にデータベースの同期設定を確認する

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database でフェイルオーバーが発生すると、プライマリサーバーとセカンダリサーバーの役割が入れ替わります。この場合、データベースへの接続に使用するエンドポイント（サーバー名）が変更されるため、アプリケーションの接続文字列を更新する必要があります。

接続文字列を更新しない場合、アプリケーションは古いエンドポイントに接続しようとして失敗します。フェイルオーバー後もアプリケーションからデータベースへのアクセスを継続するためには、新しいプライマリサーバーを指すように接続文字列を変更することが必要です。

| フェイルオーバー後の対応 | 目的 | 必要性 |
|---|---|---|
| 接続文字列の更新 | 新しいプライマリサーバーへの接続を確立 | 必須 |
| ファイアウォールルール更新 | クライアント IP からのアクセス許可 | クライアント IP 変更時のみ |
| 書き込み設定の構成 | データベースへの書き込み許可 | 自動適用されるため不要 |
| 同期設定の確認 | レプリケーション状態の確認 | 推奨だが直接的なアクセス継続には不要 |

</details>

## 問題 152

複数の企業をサポートする Azure Synapse Analytics 専用 SQL プールのセキュリティモデルを設計しています。各企業のユーザーが自社のデータのみを閲覧できるようにする必要があります。このソリューションに含めるべきオブジェクトはどれですか？（二つ選択）

- A. セキュリティポリシー
- B. 関数
- C. カスタムロールベースのアクセス制御（RBAC）ロール
- D. 列暗号化キー
- E. 非対称キー

<details>
<summary>解答・解説</summary>

**解答:** A、B

この問題では、Azure Synapse Analytics 専用 SQL プールで行レベルのセキュリティを実装して、各企業のユーザーが自社のデータのみにアクセスできるようにする方法を問われています。

行レベルのセキュリティ（RLS）を実装するには、以下の 2 つの主要コンポーネントが必要です：

1. **セキュリティポリシー**：テーブルに適用され、どの行にアクセスできるかを制御するフィルター述語を定義します。
2. **関数**：セキュリティポリシーで使用される述語関数で、ユーザーコンテキスト（例：ユーザー名やロール）に基づいて、アクセス可能な行を決定するロジックを含みます。

| コンポーネント | 目的 | 行レベルセキュリティでの役割 |
|---|---|---|
| セキュリティポリシー | テーブルに適用されるフィルター述語を定義 | 行のフィルタリングルールを適用 |
| 関数 | ユーザーコンテキストに基づく条件評価 | どの行にアクセスできるかを決定するロジック |
| 列暗号化キー | 列データの暗号化 | 行レベルセキュリティには不適切 |
| 非対称キー | 暗号化や署名に使用 | 行レベルセキュリティには不適切 |
| カスタム RBAC ロール | Azure リソースレベルのアクセス制御 | データベース内の行レベルセキュリティには不適切 |

</details>

## 問題 153

Azure Virtual Machines 上の SQL Server インスタンスがあります。週次メンテナンス作業中に、db1 という名前のデータベースに整合性エラーがあることを発見しました。バックアップからデータベースを復元する予定です。RESTORE コマンドを実行する前に、どのコマンドを実行する必要がありますか？（一つ選択）

- A. ALTER DATABASE db1 SET SINGLE_USER
- B. ALTER DATABASE db1 SET RESTRICTED_USER
- C. ALTER DATABASE db1 SET EMERGENCY
- D. ALTER DATABASE db1 SET OFFLINE

<details>
<summary>解答・解説</summary>

**解答:** A

データベースをバックアップから復元する前に、そのデータベースに接続しているユーザーがいないことを確認する必要があります。`ALTER DATABASE db1 SET SINGLE_USER` コマンドは、現在のすべての接続を終了し、データベースへのアクセスを単一ユーザーモードに制限します。これにより、復元プロセス中に他のユーザーが接続できなくなり、復元操作が妨げられることなく完了できます。

| データベース状態 | 用途 | 接続制限 | 復元操作に適しているか |
|---|---|---|---|
| SINGLE_USER | 管理操作（復元など） | 1 ユーザーのみ接続可能 | はい |
| RESTRICTED_USER | 限定的な管理操作 | 特定のロールメンバーのみ接続可能 | いいえ |
| EMERGENCY | 診断・修復 | システム管理者のみ読み取り可能 | いいえ |
| OFFLINE | 完全アクセス遮断 | 接続不可 | いいえ |

</details>

## 問題 154

動的 SQL コードを実行して「Sales」テーブルからデータを選択するストアドプロシージャを作成しました。すべてのユーザーにはこのストアドプロシージャに対する実行権限があります。ユーザーがエラーなくプロシージャを実行できるようにするには、どうすべきですか？（一つ選択）

- A. ストアドプロシージャに EXECUTE AS [owner] 句を追加する
- B. ユーザーに Sales テーブルに対する SELECT 権限を付与する
- C. ストアドプロシージャに WITH ENCRYPTION 句を追加する
- D. ストアドプロシージャに WITH RECOMPILE 句を追加する

<details>
<summary>解答・解説</summary>

**解答:** A

この問題は、動的 SQL を実行するストアドプロシージャの権限管理に関するものです。

ストアドプロシージャに「EXECUTE AS [owner]」句を追加することが正解です。この句を使用すると、プロシージャは所有者のセキュリティコンテキストで実行されるため、ユーザー自身がテーブルに対する直接的な権限を持っていなくても、プロシージャを通じてデータにアクセスできるようになります。

動的 SQL を使用する場合、通常のストアドプロシージャの権限チェックモデル（コンパイル時にチェック）ではなく、実行時に権限チェックが行われます。そのため、ユーザーが直接テーブルにアクセスする権限がなければエラーが発生します。

| 方法 | 目的 | 権限問題の解決 |
|---|---|---|
| EXECUTE AS [owner] | プロシージャを所有者のコンテキストで実行 | はい |
| テーブル直接権限付与 | ユーザーにテーブルアクセス権を与える | はい（ただしセキュリティリスク） |
| WITH ENCRYPTION | プロシージャコードの暗号化 | いいえ |
| WITH RECOMPILE | 毎回プロシージャを再コンパイル | いいえ |

</details>

## 問題 155

PaaS ソリューションの災害復旧要件を満たすために、何を実装すべきですか？（一つ選択）

- A. geo- レプリケーション
- B. フェールオーバーグループ
- C. 可用性ゾーン
- D. Azure Site Recovery

<details>
<summary>解答・解説</summary>

**解答:** B

Azure SQL Database や Azure SQL Managed Instance などの PaaS データベースサービスにおいて、災害復旧（DR）要件を満たすための最適な選択肢は「フェールオーバーグループ」です。

フェールオーバーグループは、Azure SQL Database や Azure SQL Managed Instance の自動フェールオーバー機能を提供し、プライマリデータベースとセカンダリデータベース間の関係を管理します。地理的に離れたリージョン間でのデータベースレプリケーションを設定し、リージョン全体の障害が発生した場合でも、ビジネス継続性を確保できます。

フェールオーバーグループを使用すると、自動フェールオーバーポリシーの設定、読み取りスケールの提供、エンドポイントの自動更新など、包括的な災害復旧ソリューションを実装できます。

| 選択肢 | 説明 | 災害復旧への適合性 |
|---|---|---|
| フェールオーバーグループ | 複数のデータベースを同時にフェールオーバーし、読み取りスケールを提供。リージョン間 DR 対応 | 最適 |
| geo- レプリケーション | 単一データベースのレプリケーション機能。フェールオーバーグループの基盤技術 | 部分的に適合 |
| 可用性ゾーン | 単一リージョン内の物理的に分離されたデータセンター。リージョン障害には対応不可 | 不適合 |
| Azure Site Recovery | 主に IaaS ワークロードの DR。SQL PaaS には最適でない | 不適合 |

</details>

## 問題 156

Azure SQL Database にテーブル「Table1」をデプロイする計画があります。このテーブルには、StockItemTransactionID、StockItemID、TransactionTypeID、CustomerID、InvoiceID、SupplierID、PurchaseOrderID、TransactionOccurredWhen、Quantity、LastEditedBy、LastEditedWhen のフィールドが含まれます。テーブルのパーティション分割方法を推奨する必要があります。ソリューションは以下の要件を満たす必要があります：データは 2 年間保持すること、2 年以上経過したデータは月に 1 回削除すること、データ削除操作のパフォーマンスを最大化すること。実行すべき 3 つのアクションはどれですか？（三つ選択）

- A. 各月ごとにパーティション関数を作成する
- B. 各年ごとにパーティション関数を作成する
- C. 各年のファイルグループ用のパーティションスキーマを作成する
- D. PRIMARY ファイルグループ用のパーティションスキーマを作成する
- E. StockItemTransactionID 列にパーティションスキーマを使用してテーブルを作成する
- F. TransactionOccurredWhen 列にパーティションスキーマを使用してテーブルを作成する

<details>
<summary>解答・解説</summary>

**解答:** A、C、F

この問題は、Azure SQL Database でのテーブルパーティショニングに関するものです。2 年間のデータ保持と古いデータの効率的な削除という要件を満たすためには、時間ベースのパーティショニング戦略が最適です。

1. **各月ごとにパーティション関数を作成する**：月単位でパーティションを作成することで、2 年以上経過したデータを月単位で簡単に削除できます。「スイッチアウト」操作を使用して高速に実行できます。
2. **各年のファイルグループ用のパーティションスキーマを作成する**：パーティションスキーマは、パーティションをどのファイルグループに配置するかを定義します。年ごとのファイルグループを使用することで、古いデータの管理が容易になります。
3. **TransactionOccurredWhen 列にパーティションスキーマを使用してテーブルを作成する**：日付列（TransactionOccurredWhen）でパーティション分割することで、時間ベースの要件（2 年以上経過したデータの削除）を効率的に実装できます。

| パーティショニング戦略 | 利点 | 欠点 |
|---|---|---|
| 月単位パーティション | 細かい粒度での管理が可能、月次削除に最適 | パーティション数が多くなる |
| 年単位パーティション | パーティション数が少なく管理が容易 | 月次削除には不向き |
| 日付列でのパーティション | 時間ベースの操作に最適 | 非時間ベースのクエリでは効率が下がる可能性あり |
| ID 列でのパーティション | 均等なデータ分散 | 時間ベースの要件を満たせない |

</details>

- B. Azure Data Factory
- C. Azure Synapse Analytics

Azure Databricks は以下の理由から最適な選択肢となります：

- セキュリティ分析のためのノートブック環境と高性能なクラスターを提供します

セキュリティ分析のユースケースでは、リアルタイム処理能力と柔軟な分析環境が重要です。Azure Databricks はこれらの要件を満たし、管理作業を最小限に抑えながら Apache Spark の機能を最大限に活用できるため、最適な選択肢となります。

| サービス | 主な機能 | Apache Spark 対応 | 管理作業の量 |

## 問題 159

Azure SQL データベース DB1 が General Purpose サービスレベルで稼働しています。DB1 のパフォーマンスメトリクスによると、Log IO パーセンテージが高い状態です。コストを最小限に抑えながら Log IO パーセンテージを削減する必要があります。どのような対策を取るべきですか？（一つ選択）

- A. チェックポイント操作を実行する
- B. Recovery モデルを Simple に変更する
- C. データベースのバックアップ頻度を減らす
- D. サービスレベルを Business Critical に変更する

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL データベースの Log IO パーセンテージが高い場合、トランザクションログの書き込みアクティビティが多いことを示しています。この問題を解決するための最も費用対効果の高い方法はチェックポイント操作を実行することです。

チェックポイント操作は、メモリ内のダーティページ（変更されたデータページ）をディスクに書き込み、トランザクションログからそれらの変更をマークします。これにより、トランザクションログの使用量が減少し、Log IO パーセンテージが低下します。

チェックポイント操作は、Azure SQL Database で `CHECKPOINT` コマンドを実行するか、データベースのワークロードパターンに基づいて自動チェックポイントの頻度を調整することで実装できます。これにより、追加コストなしでログ IO の使用率を効果的に削減できます。

| 選択肢 | コスト影響 | 効果 | 実装の容易さ |
|---|---|---|---|
| チェックポイント操作 | なし | 高 | 簡単 |
| Recovery モデルの変更 | なし | 不可能 | 不可能 |
| バックアップ頻度の削減 | なし | 制御不可 | 不可能 |
| Business Critical への変更 | 大幅増加 | 高 | 簡単 |

</details>

## 問題 160

SQL Server がメモリ内にデータを保持すると予想される時間を反映する Performance Monitor カウンターはどれですか？（一つ選択）

- A. Buffer Cache Hit Ratio
- B. Page Life Expectancy
- C. Memory Grants Pending
- D. Processor Queue Length

<details>
<summary>解答・解説</summary>

**解答:** B

Page Life Expectancy は、SQL サーバーのバッファプール内のページがメモリに残ると予想される秒数を示す Performance Monitor カウンターです。このカウンターの値が高いほど、SQL サーバーがディスク I/O を実行する必要性が低くなり、パフォーマンスが向上します。

一般的に、Page Life Expectancy の値は 300 秒（5 分）以上であることが推奨されています。値が低い場合は、メモリ不足の可能性があり、SQL サーバーのパフォーマンスに影響を与える可能性があります。

| パフォーマンスカウンター | 説明 | 関連リソース |
|---|---|---|
| Page Life Expectancy | バッファプール内のページがメモリに残ると予想される秒数 | メモリ |
| Buffer Cache Hit Ratio | バッファキャッシュから見つかったページの割合 | メモリ / ディスク I/O |
| Memory Grants Pending | メモリ許可を待機しているプロセスの数 | メモリ |
| Processor Queue Length | プロセッサ時間を待機しているスレッドの数 | CPU |

</details>

## 問題 161

Azure SQL Database を使用しています。以下の要件を満たす災害復旧ソリューションを実装する必要があります：
- データセンターに障害が発生した場合のデータベース復旧時間を最小化する
- 管理作業を最小限に抑える

このソリューションに含めるべきものは何ですか？（一つ選択）

- A. Azure Site Recovery
- B. 自動フェールオーバーグループ
- C. Azure Backup
- D. アクティブ geo レプリケーション

<details>
<summary>解答・解説</summary>

**解答:** B

自動フェールオーバーグループ（Auto-failover groups）は、Azure SQL Database の災害復旧ソリューションとして最適な選択肢です。このソリューションは以下の理由から要件を満たします：

1. **復旧時間の最小化**：自動フェールオーバーグループは、プライマリデータセンターに障害が発生した場合に自動的に検出し、セカンダリリージョンへのフェールオーバーを実行します。これにより、手動介入なしで迅速な復旧が可能になります。
2. **管理作業の最小化**：フェールオーバーグループは一度設定すれば、その後の管理作業はほとんど必要ありません。フェールオーバーの条件やポリシーを事前に定義でき、障害発生時に自動的に対応します。

| 機能 | 自動フェールオーバー | 管理作業 | 復旧時間 | SQL Database 向け最適化 |
|---|---|---|---|---|

| 暗号化コンポーネント | 作成場所 | 目的 |
|---|---|---|
| 証明書 | マスターデータベース | 対称キーの保護 |
| 対称キー | ユーザーデータベース (db1) | データの暗号化 |
| 非対称キー | - | SQL Server での列暗号化には通常使用しない |

この暗号化方法は、Transparent Data Encryption（TDE）ではなく、列レベルの暗号化に関するものです。TDE はデータベース全体を暗号化しますが、この場合は特定の列（CreditCard）のみを暗号化する必要があります。

## 問題 183

Azure SQL Database でカラムストア インデックスにバルク挿入する際、最小限必要な行数はいくつですか？（一つ選択）

- A. 102,400 行
- B. 1,000,000 行
- C. 1,000 行
- D. 10,000 行

<details>
<summary>解答・解説</summary>

**解答:** A

カラムストア インデックスは、大量のデータを効率的に圧縮して格納し、分析クエリのパフォーマンスを向上させるために設計されています。カラムストア インデックスにデータをバルク挿入する際、最適なパフォーマンスを得るためには最小限の行数が必要です。

Azure SQL Database のカラムストア インデックスでは、行グループと呼ばれる単位でデータが格納されます。1 つの行グループは最大約 1,048,576 行（2^20）を格納できますが、最適な圧縮と効率を得るためには、少なくとも行グループの約 10% にあたる 102,400 行（約 10 万行）をバルク挿入することが推奨されています。

この行数未満でバルク挿入を行うと、データは「デルタストア」と呼ばれる領域に一時的に格納され、後で行グループに圧縮されるため、即時のパフォーマンス向上が得られない可能性があります。

| カラムストア インデックスの重要な閾値 | 行数 | 説明 |
|---|---|---|
| 最小推奨バルク挿入行数 | 102,400 | 最適な圧縮と効率のための最小行数 |
| 行グループの最大容量 | 1,048,576 | 1 つの行グループが格納できる最大行数 |
| デルタストア閾値 | 102,400 未満 | この行数未満の挿入はデルタストアに格納される |

1,000 行や 10,000 行では少なすぎて効率的な圧縮が行われず、1,000,000 行は行グループの最大容量を超えるため、複数の行グループに分割されます。

</details>

Azure SQL Database のテーブルに、クレジットカードの CVV データを格納する Column1 という列があります。Column1 に対して以下の要件を満たす構成が必要です：保存時にデータを保護する、使用時にデータを保護する、データベース管理者が Column1 の内容を閲覧できないようにする。どの 2 つの機能を使用できますか？（二つ選択）

- D. 動的データマスキング
- E. 行レベルセキュリティ

<details>
<summary>解答・解説</summary>

**解答:** A、B

Always Encrypted は、データベース管理者やクラウド管理者からもデータを保護できる暗号化技術です。この機能は以下の要件を満たします：

1. 保存時のデータ保護（暗号化されたデータがディスクに保存される）
2. 使用時のデータ保護（データはクライアントアプリケーションでのみ復号される）

- 動的データマスキング：表示時にデータをマスクしますが、特権ユーザー（管理者）は元のデータを見ることができます
- 行レベルセキュリティ：行へのアクセスを制限しますが、暗号化は提供しません

Azure SQL 論理サーバー SQL1 があり、5 つのデータベースをホストしています。SQL1 が削除された場合でもデータベースを復元できるようにする必要があります。どのソリューションを含めるべきですか？（一つ選択）

- A. Azure Backup
- B. 長期バックアップ保持
- C. geo レプリケーション
- D. Azure Site Recovery

<details>
<summary>解答・解説</summary>

**解答:** B

> **注記（精査日: 2026-04-21）**: 元の解答は A（Azure Backup）でしたが、誤りです。Azure Backup は Azure SQL Database の論理サーバーには対応していません（SQL Server on Azure VM 向けのサービス）。論理サーバーが削除された場合に復元できるのは、事前に設定済みの長期バックアップ保持（LTR）バックアップのみです。正解は B（長期バックアップ保持）です。

SQL1 論理サーバーが削除された場合でもデータベースを復元できるようにするには、**長期バックアップ保持（LTR）**を設定する必要があります。

Microsoft の公式ドキュメントに明記されている通り、「論理サーバーを削除すると、そのサーバー上のすべてのデータベースとその PITR バックアップも削除されます。削除されたサーバーを復元することも、削除されたデータベースを PITR バックアップから復元することもできません。ただし、LTR バックアップが設定されていた場合は、それらのバックアップを使用して、同じサブスクリプション内の別のサーバーにデータベースを復元できます。」

長期バックアップ保持（LTR）は、バックアップを Azure Blob Storage の独立したコンテナーに最大 10 年間保存する機能で、論理サーバーとは独立して管理されます。これにより、サーバーが削除されても LTR バックアップからの復元が可能です。

他の選択肢について：

- Azure Backup：Azure SQL Database の論理サーバーには対応していません（SQL Server on Azure VM 向け）。
- geo レプリケーション：プライマリサーバーが削除された場合、レプリケーションは機能せず、データ復元には使用できません。
- Azure Site Recovery：主に VM やオンプレミスワークロードのディザスタリカバリに使用され、Azure SQL Database の論理サーバーには適用されません。

| ソリューション | サーバー削除時の復元可能性 | バックアップ保存場所 | 主な用途 |
|---|---|---|---|
| 長期バックアップ保持（LTR） | 可能（LTR バックアップから別サーバーへ復元） | Azure Blob Storage（サーバーと独立） | 長期データ保持・サーバー削除時の復元 |
| Azure Backup | 不可（Azure SQL DB 論理サーバー非対応） | Recovery Services コンテナー | SQL on Azure VM のバックアップ |
| geo レプリケーション | 不可（レプリケーションはサーバー削除で機能停止） | セカンダリリージョン | 地理的冗長性・DR |
| Azure Site Recovery | 適用外 | Recovery Services コンテナー | VM/オンプレミス DR |

参照: https://learn.microsoft.com/azure/azure-sql/database/recovery-using-backups#long-term-backup-restore

Resource Governor で制限できる主なリソースは以下の通りです：

- **CPU 使用率**：ワークロードグループごとに CPU 使用率の上限を設定可能
- **メモリ使用率**：リソースプールごとにメモリ使用量の上限を設定可能

| リソースタイプ | 制限可能 | 説明 |
|---|---|---|

| メモリ使用率 | はい | リソースプールごとにメモリ使用量の上限を設定可能 |
| ディスク読み取り IOPS | いいえ | I/O 操作の直接的な制限は Resource Governor では不可 |
| ディスク書き込み IOPS | いいえ | I/O 操作の直接的な制限は Resource Governor では不可 |
| 同時実行セッション数 | はい | ワークロードグループごとの同時実行セッション数を制限可能 |

したがって、SQL Server Resource Governor で制限できるリソースは「CPU 使用率」と「メモリ使用率」が正解です。

- ユーザープロパティ：パイプラインの実行時に追加できるカスタムプロパティですが、主に実行インスタンスの追跡に使用され、監視エクスペリエンスでのグループ化には最適ではありません
- 相関 ID：複数のパイプライン実行間の関連付けに使用されますが、パイプラインの目的によるグループ化には使用されません
- 注釈：パイプラインの説明やドキュメント目的で使用されますが、監視やフィルタリングには使用できません
- 実行グループ ID：パイプラインの実行をグループ化するために使用されますが、パイプラインの目的別分類には適していません

| 選択肢 | 主な用途 | 監視でのグループ化/フィルタリング | パイプライン目的のラベル付けに適切か |
|---|---|---|---|
| リソースタグ | リソースの分類と管理 | 可能 | はい |
| ユーザープロパティ | 実行インスタンスの追跡 | 限定的 | いいえ |
| 相関 ID | パイプライン実行間の関連付け | いいえ | いいえ |
| 注釈 | ドキュメント目的 | いいえ | いいえ |
| 実行グループ ID | 実行のグループ化 | 限定的 | いいえ |

</details>
