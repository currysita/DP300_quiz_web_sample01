## 問題 101

Azure SQL Database の db1 があり、その MAXDOP 設定を変更する必要があります。どのように操作すべきですか？（一つ選択）

- A. Azure SQL Database のリソース制限設定から MAXDOP パラメータを変更する
- B. db1 に接続し、ALTER DATABASE SCOPED CONFIGURATION コマンドを実行する
- C. db1 のクエリストアから MAXDOP 設定を構成する
- D. db1 に接続し、sp_configure コマンドを実行する

<details>
<summary>解答・解説</summary>

**解答:** B

Azure SQL Database では、データベースレベルの MAXDOP（最大並列度）設定を変更するには、データベーススコープ構成（Database Scoped Configuration）を使用します。これは SQL Server 2016 以降で導入された機能で、Azure SQL Database でも利用可能です。

正しい方法は、対象のデータベース（この場合は db1）に接続し、ALTER DATABASE SCOPED CONFIGURATION コマンドを実行することです。具体的には以下のようなコマンドを使用します：

```sql
ALTER DATABASE SCOPED CONFIGURATION SET MAXDOP = 値;
```

他の選択肢が不適切な理由：
- sp_configure コマンドは Azure SQL Database ではサポートされていません。これはオンプレミスの SQL Server でのみ使用できるインスタンスレベルの構成コマンドです。
- リソース制限設定から MAXDOP を直接変更する機能は Azure SQL Database には存在しません。
- クエリストアはクエリのパフォーマンスデータを収集・表示するための機能であり、MAXDOP 設定を構成するためのものではありません。

| 構成方法 | 適用範囲 | Azure SQL Database 対応 | 使用シナリオ |
|---|---|---|---|
| ALTER DATABASE SCOPED CONFIGURATION | データベースレベル | ✓ | 特定のデータベースの MAXDOP 設定変更 |
| sp_configure | インスタンスレベル | ✗ | オンプレミス SQL Server のインスタンス全体の設定 |
| クエリヒント | クエリレベル | ✓ | 特定のクエリに対する MAXDOP 設定のオーバーライド |
| リソースガバナー | リソースプールレベル | ✗ | オンプレミス SQL Server のリソース管理 |

</details>

## 問題 102

オンプレミスのデータベースバックアップを Azure に復元する場合、どの SQL Server PaaS サービスを選択すべきですか？（一つ選択）

- A. Azure SQL Database
- B. Azure SQL Managed Instance
- C. Azure SQL Elastic Pool
- D. Azure SQL Edge

<details>
<summary>解答・解説</summary>

**解答:** B

オンプレミスの SQL Server データベースバックアップ（.bak 形式）を Azure に直接復元する場合、Azure SQL Managed Instance が唯一の選択肢です。

Azure SQL Managed Instance は、オンプレミス SQL Server との高い互換性を持ち、ネイティブの復元機能をサポートしています。.bak ファイルからの直接復元が可能で、移行作業を簡素化できます。

一方、Azure SQL Database は .bak ファイルからの直接復元をサポートしていません。オンプレミスデータベースを Azure SQL Database に移行するには、以下のような別の方法を使用する必要があります：
- Data Migration Assistant (DMA)
- Azure Database Migration Service
- SQL パッケージ（BACPAC）ファイルのインポート

Azure SQL Elastic Pool は、複数の Azure SQL Database をプールしてリソースを共有するための機能であり、バックアップ復元の選択肢ではありません。

Azure SQL Edge は、IoT やエッジデバイス向けの軽量 SQL Server エンジンであり、オンプレミスバックアップの復元先としては設計されていません。

| サービス | .bak ファイルの直接復元 | 移行方法 |
|---|---|---|
| Azure SQL Managed Instance | サポート | ネイティブ復元、Azure Database Migration Service |
| Azure SQL Database | 非サポート | BACPAC、Data Migration Assistant、Azure Database Migration Service |
| Azure SQL Elastic Pool | 非サポート | Azure SQL Database と同様 |
| Azure SQL Edge | 非サポート | エッジデバイス向けのサービス |

</details>

## 問題 103

Azure SQL Managed Instance で実行されているデータベースの手動バックアップを実行する必要があります。このバックアップ操作を実行するために、BACKUP DATABASE 文に含めるべきオプションはどれですか？（一つ選択）

- A. WITH COPY_ONLY
- B. WITH CHECKSUM
- C. WITH COMPRESSION
- D. WITH INIT

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Managed Instance では、データベースの手動バックアップを実行する際に、通常のバックアップシーケンスを妨げないようにするために「WITH COPY_ONLY」オプションを使用する必要があります。

「WITH COPY_ONLY」オプションを使用すると、バックアップはログチェーンに影響を与えず、自動バックアップのスケジュールも中断しません。これは特に重要で、Azure SQL Managed Instance の管理バックアップシステムとの競合を避けるために必須です。

他のオプションについて：
- WITH CHECKSUM：バックアップ中にチェックサムを生成しますが、バックアップシーケンスへの影響は制御しません。
- WITH COMPRESSION：バックアップファイルを圧縮しますが、バックアップシーケンスには影響しません。
- WITH INIT：既存のバックアップセットを上書きするオプションで、ログチェーンを破壊する可能性があります。

| バックアップオプション | 目的 | Azure SQL Managed Instance での手動バックアップに適切か |
|---|---|---|
| WITH COPY_ONLY | 通常のバックアップシーケンスに影響を与えないバックアップを作成 | はい（推奨） |
| WITH CHECKSUM | バックアップ中にチェックサムを生成して検証 | いいえ（主目的に合わない） |
| WITH COMPRESSION | バックアップファイルを圧縮 | いいえ（主目的に合わない） |
| WITH INIT | 既存のバックアップセットを上書き | いいえ（ログチェーンを破壊する可能性あり） |

Azure SQL Managed Instance では、自動バックアップが既定で有効になっているため、手動バックアップを実行する際は、この自動バックアップシステムとの整合性を保つことが重要です。WITH COPY_ONLY オプションを使用することで、既存のバックアップチェーンを中断せずに、特定の時点でのバックアップを安全に作成できます。

</details>

## 問題 104

SQL Server の行レベルセキュリティ（RLS）において、ユーザーが行レベルセキュリティの述語に違反する値への行の更新を防止するブロック述語はどれですか？（一つ選択）

- A. BEFORE UPDATE
- B. AFTER UPDATE
- C. AFTER INSERT
- D. INSTEAD OF UPDATE

<details>
<summary>解答・解説</summary>

**解答:** A

SQL Server の行レベルセキュリティ（Row-Level Security、RLS）では、テーブルへのアクセスを行レベルで制限するためのセキュリティ機能を提供しています。RLS には主に 2 種類の述語があります：

1. フィルター述語（Filter Predicate）：SELECT、UPDATE、DELETE 操作時に、ユーザーがアクセスできる行をフィルタリングします。
2. ブロック述語（Block Predicate）：INSERT、UPDATE 操作時に、セキュリティポリシーに違反する変更をブロックします。

ブロック述語のタイプには以下があります：
- BEFORE UPDATE：更新操作が実行される前に評価され、述語に違反する更新をブロックします。
- AFTER UPDATE：更新後の状態を評価しますが、実際に更新をブロックするのは BEFORE UPDATE です。
- AFTER INSERT：挿入後の状態を評価します。

| 述語タイプ | 実行タイミング | 主な用途 |
|---|---|---|
| BEFORE UPDATE | 更新操作前 | 行レベルセキュリティ述語に違反する更新をブロック |
| AFTER UPDATE | 更新操作後 | 更新後の状態を評価（ブロックには使用されない） |
| AFTER INSERT | 挿入操作後 | 挿入後の状態を評価 |
| INSTEAD OF UPDATE | 更新操作の代わり | トリガーの一種で、RLS のブロック述語ではない |

したがって、行レベルセキュリティの述語に違反する値への行の更新を防止するブロック述語は BEFORE UPDATE です。これは更新操作が実行される前に評価され、セキュリティポリシーに違反する変更を事前にブロックする機能を持っています。

</details>

## 問題 105

Azure 仮想マシンのブロックレベルレプリケーションを実行するサービスはどれですか？（一つ選択）

- A. Azure Site Recovery
- B. Azure Backup
- C. Azure SQL Database Managed Instance
- D. Azure Disk Encryption

<details>
<summary>解答・解説</summary>

**解答:** A

Azure Site Recovery は、オンプレミスおよび Azure 仮想マシンのディザスタリカバリを提供するサービスです。このサービスは、仮想マシンのブロックレベルレプリケーションを実行し、プライマリサイトで障害が発生した場合にセカンダリロケーションへのフェールオーバーを可能にします。

Azure Backup は、データのバックアップと復元を行うサービスですが、ブロックレベルレプリケーションではなく、バックアップの作成と保存を行います。

Azure SQL Database Managed Instance は、SQL Server の完全に管理されたインスタンスを提供するサービスですが、VM 全体のレプリケーションは行いません。

Azure Disk Encryption は、Azure VM のディスクを暗号化するサービスであり、レプリケーション機能は提供していません。

</details>

## 問題 106

Azure サブスクリプションがあります。Azure Virtual Machines 上に SQL Server のインスタンスをデプロイする必要があります。ソリューションは次の要件を満たす必要があります：

- IOPS、容量、スループットなどのカスタムパフォーマンス構成をサポートする
- コストを最小限に抑える

どのタイプのディスクをソリューションに含めるべきですか？（一つ選択）

- A. Standard SSD
- B. Ultra SSD
- C. Premium SSD v2
- D. Premium SSD

<details>
<summary>解答・解説</summary>

**解答:** C

> **【2026-04-21 精査注記】** 元の解答は A（Standard SSD）でしたが、**誤りです**。Standard SSD は IOPS・スループットをカスタム設定できません（ディスクサイズに応じた固定上限値のみ）。IOPS・容量・スループットを個別にカスタム設定できるのは **Premium SSD v2** と Ultra SSD のみ。コスト最小化の要件も加味すると、Ultra SSD より安価な **Premium SSD v2** が正解です。

この問題では、Azure Virtual Machines 上の SQL Server インスタンスに適したディスクタイプを選択する必要があります。要件は「IOPS、容量、スループットなどのカスタムパフォーマンス構成のサポート」と「コスト最小化」です。

**Premium SSD v2** は、IOPS・容量・スループットを個別に設定できる唯一のディスクタイプです（Ultra SSD も可能だがより高価）。容量・IOPS・スループットをそれぞれ独立してカスタマイズでき、ワークロードに合わせた最適なコスト効率が実現できます。

各ディスクタイプのカスタム設定可否：
- **Ultra SSD**: IOPS・スループット・容量をカスタム設定可能。最高性能だが最高コスト。
- **Premium SSD v2**: IOPS・スループット・容量をカスタム設定可能。Ultra SSD より低コストで SQL Server に推奨される。
- **Premium SSD**: ディスクサイズごとに固定されたIOPS/スループット上限のみ。カスタム設定不可。
- **Standard SSD**: ディスクサイズごとに固定された上限のみ。カスタム設定不可。

| ディスクタイプ | カスタム IOPS/スループット設定 | コスト | SQL Server への適合性 |
|---|---|---|---|
| Premium SSD v2 | 可能（容量・IOPS・スループット独立設定） | 中 | 高（Microsoft 推奨） |
| Ultra SSD | 可能（容量・IOPS・スループット独立設定） | 高 | 高（最高性能・高コスト） |
| Premium SSD | 不可（サイズ固定上限のみ） | 中 | 中 |
| Standard SSD | 不可（サイズ固定上限のみ） | 低 | 低（本番 SQL Server 非推奨） |

</details>

## 問題 107

Azure SQL データベース DB3 があります。Microsoft SQL Server Management Studio (SSMS) から、ユーザー DevUser に DB3 のプロパティを表示する権限を付与する必要があります。どの Transact-SQL コマンドを実行すべきですか？（一つ選択）

- A. GRANT VIEW DATABASE STATE TO DevUser
- B. GRANT VIEW DEFINITION TO DevUser
- C. GRANT CONNECT TO DevUser
- D. GRANT SELECT TO DevUser

<details>
<summary>解答・解説</summary>

**解答:** B

この問題では、ユーザーがデータベースのプロパティを表示できるようにするための適切な権限を選択する必要があります。

VIEW DEFINITION 権限は、ユーザーにデータベースオブジェクトのメタデータを表示する権限を付与します。これにより、ユーザーはオブジェクトの定義や構造を確認できますが、データ自体にはアクセスできません。SSMS でデータベースのプロパティを表示するには、この権限が必要です。

他の選択肢について：
- VIEW DATABASE STATE：データベースの動的管理ビュー（DMV）やパフォーマンス関連の情報へのアクセスを許可しますが、プロパティの表示には直接関係しません。
- CONNECT：データベースへの接続のみを許可します。メタデータの表示権限は含まれません。
- SELECT：テーブルやビューからデータを読み取る権限を付与しますが、データベースのプロパティ表示には関係ありません。

| 権限 | 目的 | データベースプロパティ表示に適切か |
|---|---|---|
| VIEW DEFINITION | メタデータやオブジェクト定義の表示 | はい |
| VIEW DATABASE STATE | DMV やパフォーマンス情報へのアクセス | いいえ |
| CONNECT | データベースへの接続 | いいえ |
| SELECT | データの読み取り | いいえ |

したがって、ユーザーがデータベースのプロパティを表示できるようにするには、GRANT VIEW DEFINITION TO DevUser コマンドを実行する必要があります。

</details>

## 問題 108

Azure SQL Database のアクティブなジオレプリケーション構成で 2 つのデータベースを運用しています。ジオレプリケーションのラグを監視する必要があります。どのビューをクエリすべきですか？（一つ選択）

- A. sys.dm_database_replica_states
- B. sys.dm_geo_replication_link_status
- C. sys.dm_operation_status
- D. sys.dm_hadr_database_replica_states

<details>
<summary>解答・解説</summary>

**解答:** B

Azure SQL Database のジオレプリケーション構成におけるレプリケーションラグを監視するには、`sys.dm_geo_replication_link_status` 動的管理ビューを使用します。このビューは、プライマリデータベースとセカンダリデータベース間のレプリケーションの状態と遅延（ラグ）に関する情報を提供します。

`sys.dm_geo_replication_link_status` ビューには以下の重要な列が含まれています：
- `last_replication_time`：最後にレプリケーションが行われた時刻
- `replication_lag_sec`：セカンダリがプライマリより遅れている秒数（ラグ）
- `replication_state`：レプリケーションリンクの状態（0= プライマリ、1= セカンダリ）

他の選択肢について：
- `sys.dm_database_replica_states`：Always On 可用性グループのレプリカ状態を監視するためのビューで、Azure SQL Database のジオレプリケーションには適していません。
- `sys.dm_operation_status`：非同期操作の状態を確認するためのビューで、レプリケーションラグの監視には使用しません。
- `sys.dm_hadr_database_replica_states`：SQL Server（オンプレミス）の Always On 可用性グループ用のビューで、Azure SQL Database には存在しません。

| ビュー名 | 用途 | ジオレプリケーションラグ監視に適切か |
|---|---|---|
| sys.dm_geo_replication_link_status | ジオレプリケーションリンクの状態とラグ情報 | はい |
| sys.dm_database_replica_states | Always On 可用性グループのレプリカ状態 | いいえ |
| sys.dm_operation_status | 非同期操作の状態確認 | いいえ |
| sys.dm_hadr_database_replica_states | オンプレミス SQL Server の Always On 状態 | いいえ |

</details>

## 問題 109

Azure SQL Database の db1 があります。最新の TLS バージョンをサポートする App1 というアプリケーションがあります。App1 が最も安全な方法で db1 にアクセスできるようにする必要があります。どうすべきですか？（一つ選択）

- A. 最小 TLS バージョン設定を 1.2 に設定する
- B. 既定の TLS バージョン設定を受け入れる
- C. 最小 TLS バージョン設定を 1.0 に設定する
- D. 接続文字列に Encrypt=True パラメータを追加する

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database への接続を最も安全にするには、最小 TLS バージョン設定を 1.2 に設定するのが最適です。TLS（Transport Layer Security）はデータ転送時の暗号化プロトコルであり、バージョンが上がるごとにセキュリティが強化されています。

TLS 1.2 は、TLS 1.0 や 1.1 と比較して、より強力な暗号化アルゴリズムとセキュリティ機能を提供します。Microsoft 自身も、セキュリティ上の理由から TLS 1.0 および 1.1 の使用を非推奨としており、TLS 1.2 以上の使用を推奨しています。

他の選択肢について：
- 既定の TLS バージョン設定を受け入れる選択肢は、将来的に Microsoft がデフォルト設定を変更する可能性があるため、明示的に最新の安全なバージョンを指定する方が確実です。
- TLS 1.0 に設定する選択肢は、既知の脆弱性があるため、セキュリティリスクが高まります。
- 接続文字列に Encrypt=True を追加するだけでは、使用される TLS バージョンは指定されないため、最も安全な方法とは言えません。

| TLS バージョン | セキュリティレベル | Microsoft 推奨 | 主な特徴 |
|---|---|---|---|
| TLS 1.0 | 低 | 非推奨 | 既知の脆弱性あり、古いプロトコル |
| TLS 1.1 | 中 | 非推奨 | TLS 1.0 より改善されているが、まだ脆弱性あり |
| TLS 1.2 | 高 | 推奨 | 強力な暗号スイート、より安全なハッシュアルゴリズム |
| TLS 1.3 | 最高 | 推奨 | 最新バージョン、ハンドシェイクの改善、より高速 |

Azure SQL Database では、データベースサーバーレベルで最小 TLS バージョンを設定することで、指定したバージョン未満の TLS を使用した接続を拒否できます。これにより、古い TLS バージョンによるセキュリティリスクを排除できます。

</details>

## 問題 110

データベース管理者として、障害発生時のリカバリ計画を策定しています。データ損失を最小限に抑えるために、RPO（Recovery Point Objective）を定義する必要があります。RPO とは何ですか？（一つ選択）

- A. クラスター内のノード数
- B. 障害発生後にデータを復旧させる必要がある時点
- C. データベースの部分復元プロセス
- D. システムが完全に復旧するまでの許容時間

<details>
<summary>解答・解説</summary>

**解答:** B

RPO（Recovery Point Objective）は、災害復旧計画における重要な指標の一つで、「障害発生後にデータを復旧させる必要がある時点」を指します。具体的には、障害発生時にどの時点までのデータを復元する必要があるか、つまり許容できるデータ損失の最大量を時間で表したものです。

例えば、RPO が 4 時間の場合、最大で 4 時間分のデータ損失が許容されることを意味します。これに基づいて、バックアップの頻度やレプリケーション方式を決定します。

他の選択肢について：
- 「クラスター内のノード数」は RPO とは無関係で、高可用性クラスターの構成要素です。
- 「データベースの部分復元プロセス」は特定のデータのみを復元する手法を指しますが、RPO の定義ではありません。
- 「システムが完全に復旧するまでの許容時間」は RTO（Recovery Time Objective）の定義であり、RPO とは異なります。

| 復旧指標 | 定義 | 例 |
|---|---|---|
| RPO (Recovery Point Objective) | 障害発生後にデータを復旧させる必要がある時点 | 4 時間前のデータまで復元可能 |
| RTO (Recovery Time Objective) | システムが復旧するまでの許容時間 | 2 時間以内にシステムを復旧 |
| MTTR (Mean Time To Recover) | 障害からの平均復旧時間 | 平均 90 分で復旧完了 |
| MTBF (Mean Time Between Failures) | 障害間の平均時間 | 平均 6000 時間の安定稼働 |

</details>

## 問題 111

## 問題 115

Azure SQL Database の db1 には、db1_readers と db1_writers という 2 つのカスタムロールがあります。Microsoft Entra グループ dbadmin のメンバーが、db1_readers と db1_writers のメンバーシップを管理できるようにする必要があります。最小権限の原則に従った解決策を実装するには、どの 2 つのアクションを実行すべきですか？（二つ選択）

- A. db1 データベースのデータベースユーザーに db_securityadmin ロールを割り当てる
- B. master データベースのデータベースユーザーに sysadmin ロールを割り当てる
- C. dbadmin グループのログインと db1 データベースユーザーを作成する
- D. master データベースのデータベースユーザーに loginmanager ロールを割り当てる
- E. db1 データベースのデータベースユーザーに db_accessadmin ロールを割り当てる
- F. dbadmin グループのログインと master データベースユーザーを作成する

<details>
<summary>解答・解説</summary>

**解答:** A、C

この問題では、Microsoft Entra グループ dbadmin のメンバーがカスタムロール db1_readers と db1_writers のメンバーシップを管理できるようにするための最小権限の設定が求められています。

正解の組み合わせは以下の理由によります：

1. **dbadmin グループのログインと db1 データベースユーザーを作成する（C）**：Microsoft Entra グループをデータベースにマッピングするには、まずそのグループのログインを作成し、次にデータベースユーザーを作成する必要があります。これにより、グループメンバーがデータベースにアクセスできるようになります。

2. **db1 データベースのデータベースユーザーに db_securityadmin ロールを割り当てる（A）**：db_securityadmin ロールは、データベース内のロールメンバーシップを管理する権限を持っています。このロールを持つユーザーは、カスタムロールを含むデータベースロールのメンバーシップを変更できます。

不正解の選択肢について：
- db_accessadmin ロールは、データベースへのアクセス権を管理するためのものであり、ロールメンバーシップの管理には適していません。
- sysadmin と loginmanager は master データベースのロールであり、必要以上の権限を付与することになるため、最小権限の原則に反します。
- master データベースにユーザーを作成することは、db1 のカスタムロールを管理するという目的には直接関係ありません。

| ロール | 権限 | 適切な使用場面 |
|---|---|---|
| db_securityadmin | データベース内のロールメンバーシップと権限の管理 | ロールメンバーシップの管理が必要な場合 |
| db_accessadmin | データベースユーザーアクセスの管理 | データベースへのアクセス権の管理が必要な場合 |
| sysadmin | サーバーレベルでの完全な管理権限 | サーバー全体の管理が必要な場合（過剰な権限） |
| loginmanager | ログインの作成と管理 | サーバーレベルでのログイン管理が必要な場合 |

最小権限の原則に従うと、必要な操作（カスタムロールのメンバーシップ管理）に対して必要最小限の権限を付与するべきです。そのため、db1 データベース内での db_securityadmin ロールの割り当てと、dbadmin グループのログインと db1 データベースユーザーの作成が最適な解決策となります。

<summary>解答・解説</summary>

1. **可用性セット** - 単一の可用性セットに両方の VM をデプロイすることで、Azure は異なる障害ドメインと更新ドメインに VM を分散させ、99.95% の SLA を提供します。
2. **近接配置グループ** - 単一の近接配置グループに両方の VM を配置することで VM が物理的に近い場所に配置され、ネットワークレイテンシが最小化されます。これは同期レプリケーションとデータベース書き込みの遅延最小化の要件を満たすために重要です。

- 複数の可用性セットに分散すると、VM が異なるクラスターに配置される可能性があり、レイテンシが増加します。
- 複数の近接配置グループを使用すると VM が物理的に離れた場所に配置される可能性があり、同期レプリケーションのパフォーマンスに悪影響を与えます。

| 構成要素 | 目的 | 高可用性への影響 |
|---|---|---|
| 可用性セット | 計画的 / 計画外のメンテナンス中に VM の可用性を確保 | 99.95% の SLA を提供 |

| Always On 可用性グループ | データベースレベルの高可用性を提供 | 同期レプリケーションによるデータ保護を実現 |

SQL Server Always On 可用性グループでは、同期レプリケーションを使用する場合、レイテンシが重要な要素となります。VM が物理的に近い場所にあることで、レプリケーションのパフォーマンスが向上し、トランザクションの遅延が最小限に抑えられます。

</details>

## 問題 123

Azure Synapse Analytics の専用 SQL プールにて、Table1 という名前のテーブルがあります。Azure Data Lake Storage Gen2 の container1 にデータがあり、ロードされたデータを Table1 にサービングレイヤーとして格納したいと考えています。ロード時の DateTime 値を追加の列として格納したいと考えています。専用 SQL ツールを使用して、追加 DateTime 列を含む外部テーブルを作成することは可能ですか？（一つ選択）

- A. いいえ、外部テーブルはソースデータの構造を反映するだけで、追加の列を自動的に格納することはできません
- B. いいえ、この目的には PolyBase を使用する必要があります
- C. はい、DEFAULT GETDATE() 制約を追加する必要があります

<details>
<summary>解答・解説</summary>

**解答:** A

この問題では、Azure Data Lake Storage Gen2 から Azure Synapse Analytics の専用 SQL プールのテーブルにデータをロードする際に、ロード時間を示す DateTime 値を追加の列として格納する方法を問うています。

外部テーブルは、Azure Data Lake Storage Gen2 などの外部ソースに存在するデータの構造を定義するためのものです。外部テーブルを作成しても、それだけではロード時のタイムスタンプなどの追加情報を自動的に格納することはできません。外部テーブルはソースデータの構造をそのまま反映するだけです。

この要件を満たすためには、以下のような方法が考えられます：

1. CETAS（Create External Table As Select）ステートメントを使用して、クエリ結果を外部テーブルとして保存する際に GETDATE() などの関数を使用して日時を追加する
2. COPY 文を使用してデータをロードする際に、変換クエリで日時列を追加する
3. データフローや Azure Data Factory のパイプラインを使用して、ロード処理中に日時情報を追加する

| データロード方法 | 追加列の対応 | 使用シナリオ |
|---|---|---|
| 外部テーブルのみ | 不可（ソースデータ構造のみ反映） | データ探索、クエリのみ |
| CETAS | 可能（SELECT 句で追加可能） | データ変換と保存を同時に行う |
| COPY 文 | 可能（WITH 句で変換指定可能） | 高パフォーマンスデータロード |
| データフロー | 可能（派生列で追加可能） | 複雑な ETL 処理 |

したがって、単に外部テーブルを作成するだけでは、ロード時の DateTime を追加列として格納するという要件を満たすことはできません。

</details>

## 問題 124

Azure SQL Managed Instance で、ピーク時間帯に多数のブロッキングクエリが発生していることを発見しました。ブロッキングクエリの根本原因を調査する必要があります。どの動的管理ビューを照会すべきですか？（二つ選択）

- A. sys.dm_os_waiting_tasks
- B. sys.dm_exec_requests
- C. sys.dm_tran_locks
- D. sys.dm_exec_query_stats

<details>
<summary>解答・解説</summary>

**解答:** A、B

ブロッキングクエリの根本原因を調査するには、現在実行中のクエリとそれらの待機状態を確認する必要があります。

`sys.dm_os_waiting_tasks` は現在待機中のタスクに関する情報を提供します。このビューを使用すると、どのセッションが他のセッションをブロックしているかを特定できます。待機タイプ、待機時間、ブロッキングセッション ID などの重要な情報が含まれています。

`sys.dm_exec_requests` は現在実行中のリクエストに関する情報を提供します。このビューには、各リクエストの実行状態、待機タイプ、リソース使用状況などが含まれており、ブロッキングの状況を把握するのに役立ちます。

`sys.dm_tran_locks` もブロッキング分析に役立ちますが、設問では完全な解決策を求めているため、より直接的にブロッキング情報を提供する上記 2 つのビューが最適です。

`sys.dm_exec_query_stats` はクエリのパフォーマンス統計情報を提供しますが、現在のブロッキング状態を直接示すものではありません。

| 動的管理ビュー | 主な用途 | ブロッキング分析での役割 |
|---|---|---|
| sys.dm_os_waiting_tasks | 待機中のタスク情報 | ブロッキングチェーンの特定 |
| sys.dm_exec_requests | 実行中リクエスト情報 | 現在の実行状態と待機状態の確認 |
| sys.dm_tran_locks | ロック情報 | リソースに対するロック競合の確認 |
| sys.dm_exec_query_stats | クエリパフォーマンス統計 | 長時間実行クエリの特定 |

ブロッキング問題の調査では、通常これらのビューを組み合わせて使用し、問題のあるクエリやセッションを特定します。

</details>

## 問題 125

Azure 仮想マシン上でホストされている SQL Server データベースのバックアップソリューションを構築しています。Azure リージョンの停止が発生した場合でも、データベースバックアップを復元できる必要があります。このソリューションはコストを最小限に抑える必要があります。バックアップに使用すべきストレージアカウントの種類はどれですか？（一つ選択）

- A. geo-redundant storage (GRS)
- B. zone-redundant storage (ZRS)
- C. read-access geo-redundant storage (RA-GRS)
- D. geo-zone-redundant storage (GZRS)

<details>
<summary>解答・解説</summary>

**解答:** A

この問題では、Azure リージョンの停止時にもデータベースバックアップを復元できる必要があり、かつコストを最小限に抑えるソリューションを選ぶ必要があります。

GRS（geo-redundant storage）は、プライマリリージョンのデータを別のセカンダリリージョンに非同期でレプリケートします。プライマリリージョンで障害が発生した場合でも、セカンダリリージョンのデータにアクセスできるため、リージョン障害からの復旧が可能です。GRS は RA-GRS よりもコストが低く、リージョン間レプリケーションを提供する最も経済的なオプションです。

ZRS（zone-redundant storage）は同一リージョン内の複数のアベイラビリティゾーンにデータをレプリケートしますが、リージョン全体の障害には対応できません。

RA-GRS（read-access geo-redundant storage）は GRS の機能に加えて、セカンダリリージョンへの読み取りアクセスも提供しますが、GRS よりもコストが高くなります。

GZRS（geo-zone-redundant storage）は、ZRS と GRS を組み合わせたもので、最も高い可用性を提供しますが、コストも最も高くなります。

| ストレージタイプ | リージョン障害からの保護 | コスト | 読み取りアクセス（セカンダリ） |
|---|---|---|---|
| LRS | なし | 最低 | なし |
| ZRS | なし（ゾーン障害のみ） | 低 | なし |
| GRS | あり | 中 | なし |
| RA-GRS | あり | 高 | あり |
| GZRS | あり | 最高 | なし |

</details>

顧客がデータベースオブジェクトを作成できるようにするソリューションを推奨する必要があります。このソリューションはビジネス目標を満たす必要があります。推奨事項に何を含めるべきですか？（一つ選択）

- A. 各顧客に対して、既存のスキーマに対する db_owner 権限を付与する
- B. 各顧客に対して、追加のスキーマを作成し、新しいスキーマに対する ddl_admin 権限を付与する
- C. 各顧客に対して、既存のスキーマに対する ddl_admin 権限を付与する
- D. 各顧客に対して、追加のスキーマを作成し、新しいスキーマに対する db_writer 権限を付与する

<details>
<summary>解答・解説</summary>

**解答:** B

正解は「各顧客に対して、追加のスキーマを作成し、新しいスキーマに対する ddl_admin 権限を付与する」です。

ddl_admin 権限は、SQL Server の固定データベースロールではなく、スキーマレベルの権限で、Data Definition Language (DDL) コマンドを実行する権限を付与します。これにより、顧客は自分のスキーマ内でテーブル、ビュー、ストアドプロシージャなどのデータベースオブジェクトを作成できます。

各顧客に対して新しいスキーマを作成することで、顧客間のデータ分離が確保され、セキュリティが向上します。各顧客は自分のスキーマ内でのみオブジェクトを作成でき、他の顧客のスキーマには影響を与えません。

Azure SQL Database の権限管理では、最小権限の原則に従い、必要最小限の権限のみを付与することが重要です。ddl_admin 権限と顧客ごとの個別スキーマの組み合わせは、この原則に沿った最適な解決策です。

| 選択肢 | 不適切な理由 |
|---|---|
| 既存のスキーマに対する db_owner 権限を付与 | db_owner は過剰な権限を付与し、セキュリティリスクとなる |
| 既存のスキーマに対する ddl_admin 権限を付与 | 顧客間でスキーマを共有すると、分離性が損なわれる |
| 追加のスキーマに対する db_writer 権限を付与 | db_writer はデータ操作のみの権限で、オブジェクト作成権限がない |

</details>

## 問題 130

Azure SQL Database で Transparent Data Encryption (TDE) を有効化するために必要なセキュリティオブジェクトはどれですか？（一つ選択）

- A. Database Master Key (DMK)
- B. Azure Key Vault
- C. Certificate
- D. Server Master Key (SMK)

<details>
<summary>解答・解説</summary>

**解答:** D

TDE を有効化するためには、Server Master Key (SMK) が必要です。SMK はデータベースサーバーレベルで作成される暗号化キーで、TDE の基盤となるキー階層の最上位に位置します。

TDE のキー階層は以下のようになっています：

| キーの種類 | 役割 | スコープ |
|---|---|---|
| Server Master Key (SMK) | キー階層の最上位 | サーバーレベル |
| Database Encryption Key (DEK) | 実際のデータ暗号化に使用 | データベースレベル |
| Certificate | DEK を保護 | サーバーレベル |

- Azure Key Vault は TDE の代替オプションとして使用できますが、必須ではありません。これは「Bring Your Own Key (BYOK)」シナリオで使用され、TDE キーを Azure Key Vault で管理できます。
- Database Master Key (DMK) はデータベースレベルのキーで、列レベルの暗号化や Always Encrypted 機能に使用されますが、TDE の有効化には直接必要ありません。
- Certificate は TDE で使用されることがありますが、SMK なしでは機能しません。SMK が基盤となり、その上に Certificate と DEK の階層が構築されます。

</details>

## 問題 131

SQL Server または Azure SQL Managed Instance で「URL へのバックアップ」を実行した場合、バックアップファイルはどのように保存されますか？（一つ選択）

- A. Azure Blob Storage のページブロブとして保存される
- B. Azure Blob Storage のブロックブロブとして保存される
- C. Azure Data Lake のファイルとして保存される
- D. Azure File Share の共有ファイルとして保存される

<details>
<summary>解答・解説</summary>

**解答:** B

SQL Server または Azure SQL Managed Instance で「URL へのバックアップ」(BACKUP TO URL) を実行すると、バックアップファイルは Azure Blob Storage のブロックブロブ（Block Blob）として保存されます。

ブロックブロブは大量のデータを効率的にアップロードするために設計されており、最大サイズは約 4.75 TB です。データベースバックアップのような大きなファイルの保存に適しています。

SQL Server や Azure SQL Managed Instance では、バックアップを実行する際に以下の手順が必要です：
1. Azure Storage アカウントの作成
2. コンテナの作成
3. 共有アクセス署名（SAS）または管理 ID 認証の設定
4. SQL Server 資格情報の作成
5. BACKUP TO URL 構文を使用したバックアップの実行

```sql
BACKUP DATABASE [DatabaseName]
TO URL = 'https://storageaccount.blob.core.windows.net/container/backup.bak'
WITH CREDENTIAL = 'MyAzureCredential';
```

| ストレージタイプ | 特徴 | SQL Server バックアップでの使用 |
|---|---|---|
| ブロックブロブ | 大きなファイルのアップロードに最適、最大 4.75TB | URL へのバックアップで使用される |
| ページブロブ | ランダムな読み書きに最適、Azure VM ディスクに使用 | SQL Server バックアップには使用されない |
| Azure Data Lake | 分析ワークロード向け階層型ファイルシステム | SQL Server バックアップには直接対応していない |
| Azure File Share | SMB プロトコルを使用したファイル共有 | URL へのバックアップでは使用されない |

</details>

## 問題 132

Azure SQL データベース sqldb1 があります。Query Store が読み取り専用状態に移行する可能性を最小限に抑える必要があります。どうすべきですか？（一つ選択）

- A. Data Flush Interval の値を倍増します
- B. Statistics Collection Interval の値を半分にします
- C. Query Store の最大サイズを増やします
- D. Statistics Collection Interval の値を倍増します

<details>
<summary>解答・解説</summary>

**解答:** C

> **【2026-04-21 精査注記】** 元の解答は A（Data Flush Interval を倍増）でしたが、**誤りです**。Microsoft の公式ドキュメントによると、Query Store が READ_ONLY になる主因は MAX_STORAGE_SIZE_MB の超過です。最大サイズを増やすことが最も直接的な対策です。Data Flush Interval を大きくすると、サイズチェックの間隔が長くなり、むしろ閾値を超えても気づかない状態が長引くため、READ_ONLY になりやすくなる可能性があります。
> 参照: https://learn.microsoft.com/sql/relational-databases/performance/manage-the-query-store

Query Store が読み取り専用状態に移行する主な理由は、割り当てられたストレージ容量（MAX_STORAGE_SIZE_MB）を超えることです。Query Store の最大サイズを増やすことで、この問題を直接解決できます。

他の選択肢の影響：
- Data Flush Interval の値を倍増すると、ディスクへのサイズチェックが遅くなり、閾値を超えた後も READ_ONLY に移行するまでの時間が長くなる場合があります（リスク軽減の効果はない）。
- Statistics Collection Interval の値を半分にすると、データ収集量が増加し、READ_ONLY になりやすくなります。
- Statistics Collection Interval の値を倍増すると、データ収集量が減少し、若干のリスク軽減効果がありますが、最大サイズを増やすほど直接的ではありません。

| パラメータ | 変更 | 読み取り専用状態への影響 |
|---|---|---|
| Query Store の最大サイズ | 増加 | リスク減少（根本原因への直接対策） |
| Statistics Collection Interval | 倍増 | 若干のリスク減少（データ生成量の減少） |
| Statistics Collection Interval | 半分 | リスク増加（データ生成量の増加） |
| Data Flush Interval | 倍増 | リスク軽減効果なし（サイズチェック間隔が長くなるだけ） |

</details>

## 問題 134

Azure サブスクリプションに論理 SQL サーバー Server1 があります。Server1 のマスターデータベースには User1 というユーザーが含まれています。User1 が Server1 上でデータベースを作成できるようにする必要があります。User1 に割り当てるべきデータベースロールはどれですか？（一つ選択）

- A. dbmanager
- B. db_ddladmin
- C. db_accessadmin
- D. db_owner

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database の論理サーバーレベルでデータベースを作成する権限を付与するには、マスターデータベース内のユーザーに「dbmanager」ロールを割り当てる必要があります。

dbmanager は、Azure SQL Database のサーバーレベルのロールで、ユーザーにデータベースの作成権限を与えます。このロールはマスターデータベース内で割り当てられ、サーバー全体に対する権限を制御します。

他の選択肢について：
- db_ddladmin: データベース内での DDL（Data Definition Language）操作を実行できるデータベースレベルのロールですが、新しいデータベースの作成権限はありません
- db_accessadmin: データベースへのユーザーアクセスを管理するデータベースレベルのロールです

| ロール名 | レベル | 主な権限 | データベース作成権限 |
|---|---|---|---|
| dbmanager | サーバー | データベースの作成と管理 | あり |
| db_ddladmin | データベース | DDL 操作の実行 | なし |
| db_accessadmin | データベース | ユーザーアクセスの管理 | なし |
| db_owner | データベース | データベース内のすべての操作 | なし |

Azure SQL Database では、サーバーレベルの権限とデータベースレベルの権限が明確に分離されています。新しいデータベースを作成するには、サーバーレベルの権限が必要であり、それを提供するのが dbmanager ロールです。

</details>

Azure サブスクリプションに Azure SQL データベースがあります。このデータベースには tablet という名前のテーブルが含まれています。次の Transact-SQL ステートメントを実行しました。データベースに対する分析クエリの実行時間を短縮する必要があります。どの構成を有効にすべきですか？（一つ選択）

- A. BATCH_MODE_ON_ROWSTORE
- B. ROW_MODE_MEMORY_GRANT_FEEDBACK
- C. BATCH_MODE_MEMORY_GRANT_FEEDBACK
- D. BATCH_MODE_ADAPTIVE_JOINS

<details>
<summary>解答・解説</summary>

**解答:** A

BATCH_MODE_ON_ROWSTORE は、列ストアインデックスがなくても行ストアテーブルに対してバッチモード処理を有効にする機能です。分析クエリは通常、大量のデータを処理するため、行単位の処理（Row モード）よりもバッチ単位の処理（Batch モード）の方が効率的です。

バッチモード処理は、複数の行（通常は約 900 行）を一度に処理するため、CPU 使用率を削減し、クエリのパフォーマンスを大幅に向上させることができます。特に分析クエリのような集計操作を含むクエリに効果的です。

他の選択肢について：
- BATCH_MODE_MEMORY_GRANT_FEEDBACK: メモリ割り当ての最適化に役立ちますが、主にバッチモード操作のメモリ使用量を調整するものです
- ROW_MODE_MEMORY_GRANT_FEEDBACK: 行モード操作のメモリ割り当てを最適化しますが、分析クエリの全体的なパフォーマンス向上には BATCH_MODE_ON_ROWSTORE ほど効果的ではありません
- BATCH_MODE_ADAPTIVE_JOINS: 実行時にクエリの結合方法を動的に変更する機能で、特定の結合操作のパフォーマンスを向上させますが、全体的な分析クエリのパフォーマンス向上には限定的です

| 構成オプション | 主な目的 | 分析クエリへの効果 |
|---|---|---|
| BATCH_MODE_ON_ROWSTORE | 行ストアテーブルでのバッチ処理を有効化 | 大量データの処理を高速化、CPU 使用率を削減 |
| BATCH_MODE_MEMORY_GRANT_FEEDBACK | バッチモード操作のメモリ割り当て最適化 | メモリ使用の効率化だが処理速度への直接効果は限定的 |
| ROW_MODE_MEMORY_GRANT_FEEDBACK | 行モード操作のメモリ割り当て最適化 | 行処理の効率化だが大量データ処理には不向き |
| BATCH_MODE_ADAPTIVE_JOINS | 実行時の結合方法の動的最適化 | 特定の結合操作の最適化のみ |

</details>

<details>

**解答:** C

SELECT * FROM sys.dm_exec_requests

- sys.dm_db_resource_stats は有用な DMV ですが、単独ではなく他の DMV と組み合わせて使用するのが効果的です。

| 分析方法 | tempdb 分析の適合性 | リアルタイム性 | 詳細度 |
|---|---|---|---|
| 動的管理ビュー（DMV） | 高 | 高 | 高 |
| Query Performance Insight | 中 | 中 | 中 |
| 自動チューニング推奨事項 | 低 | 低 | 低 |
| sys.dm_db_resource_stats 単独 | 中 | 高 | 中 |

アクティブジオレプリケーションは、この要件を満たす最適な選択肢です。アクティブジオレプリケーションを使用すると、プライマリデータベースから複数のセカンダリデータベースを別のリージョンに作成でき、これらのセカンダリデータベースは読み取りアクセスが可能です。また、プライマリデータベースがあるデータセンターで障害が発生した場合、セカンダリデータベースに手動でフェイルオーバーすることができます。

他の選択肢について：

- フェイルオーバーグループは、自動フェイルオーバー機能を提供しますが、読み取り可能なコピーが 1 つしか保証されないため、「少なくとも 2 つの読み取り可能なコピー」という要件を満たしません
- 自動バックアップと長期保存バックアップは、データの保護には役立ちますが、読み取り可能なデータベースコピーを提供せず、データセンター障害時の即時可用性も保証しません

| 機能 | 読み取り可能なコピー | リージョン間の冗長性 | 自動フェイルオーバー |
|---|---|---|---|
| アクティブジオレプリケーション | 複数可能 | あり | 手動 |
| フェイルオーバーグループ | 1 つ | あり | 自動 |
| 自動バックアップ | なし | なし | なし |
| 長期保存バックアップ | なし | なし | なし |

</details>

## 問題 149

Azure サブスクリプションに Azure SQL Managed Instance の SQLMI1 と Log Analytics ワークスペースの Workspace1 があります。SQLMI1 のパフォーマンスメトリックを収集し、Workspace1 にストリーミングする必要があります。どうすればよいですか？（一つ選択）

- A. SQLMI1 の診断設定を変更する
- B. Azure SQL Analytics を構成して Workspace1 を使用するようにする
- C. SQLMI1 のプライベートエンドポイント接続を作成する
- D. SQLMI1 のコンピューティング + ストレージ設定を変更する

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Managed Instance のパフォーマンスメトリックを Log Analytics ワークスペースに送信するには、診断設定を構成する必要があります。診断設定により、Azure リソースのプラットフォームメトリックとログを特定の宛先（この場合は Log Analytics ワークスペース）に送信できます。

診断設定を構成するには：
1. Azure ポータルで SQL Managed Instance リソースに移動
2. 「監視」セクションの「診断設定」を選択
3. 「診断設定の追加」をクリック
4. 収集するメトリックを選択（SQL Managed Instance のパフォーマンスメトリックを含む）
5. 宛先として「Log Analytics ワークスペースに送信」を選択し、Workspace1 を指定

他の選択肢が不適切な理由：

- Azure SQL Analytics は Log Analytics の一部として機能しますが、メトリックの収集自体は診断設定で行う必要があります
- コンピューティング + ストレージ設定はパフォーマンス特性を変更するものであり、メトリックの収集には関係ありません
- プライベートエンドポイント接続はネットワーク接続に関するもので、メトリックの収集には直接関係ありません

| 選択肢 | 目的 | メトリック収集に適切か |
|---|---|---|
| 診断設定 | リソースのメトリックとログを特定の宛先に送信 | はい |
| Azure SQL Analytics | 収集したデータの分析と可視化 | 部分的（診断設定と併用） |
| コンピューティング + ストレージ設定 | リソースのパフォーマンス特性の変更 | いいえ |
| プライベートエンドポイント接続 | プライベートネットワーク経由のアクセス確保 | いいえ |

</details>

## 問題 150

Azure SQL Database のワークロードをリージョン内の複数のデータセンターに分散させるために選択すべきオプションはどれですか？（一つ選択）

- A. Availability Zones
- B. Availability Sets
- C. Geo-replication
- D. Failover Groups

<details>
<summary>解答・解説</summary>

**解答:** A

Availability Zones は Azure リージョン内の物理的に分離されたデータセンターのグループであり、独立した電源、冷却、ネットワークを備えています。Azure SQL Database で Availability Zones を使用すると、リージョン内の複数のデータセンター間でワークロードを分散させることができます。

他の選択肢について：

- Availability Sets はリージョン内の単一データセンター内での仮想マシンの分散に使用されますが、SQL Database には適用されません
- Geo-replication と Failover Groups は異なるリージョン間でのデータ複製に使用されるもので、リージョン内のデータセンター間の分散には使用されません

Azure SQL Database の Business Critical および Premium サービスレベルでは、Availability Zones を活用して高可用性を実現できます。これにより、データセンターレベルの障害が発生しても、サービスの継続性を確保できます。

| オプション | 分散範囲 | 主な目的 | SQL Database への適用 |
|---|---|---|---|
| Availability Zones | リージョン内の複数データセンター | 同一リージョン内のデータセンター障害からの保護 | 適用可能 |
| Availability Sets | 単一データセンター内 | 単一データセンター内のハードウェア障害からの保護 | SQL Database には非適用 |
| Geo-replication | 異なるリージョン間 | リージョン全体の障害からの保護 | 適用可能 |
| Failover Groups | 異なるリージョン間 | 自動フェイルオーバーを含むリージョン障害からの保護 | 適用可能 |

</details>
