## 問題 51

アプリケーションをデプロイする計画があります。このアプリケーションには Azure SQL Database と Azure Web App が含まれています。アプリケーションには以下の要件があります：

- Web アプリは Azure 仮想ネットワーク上でホストする必要がある
- Azure SQL Database にはプライベート IP アドレスを割り当てる必要がある
- Azure SQL Database は仮想ネットワークからの接続のみを許可する必要がある

要件を満たすソリューションを推奨する必要があります。推奨事項に何を含めるべきですか？（一つ選択）

- A. Azure Private Link
- B. サーバーレベルのファイアウォール
- C. データベースレベルのファイアウォール
- D. ネットワークセキュリティグループ（NSG）

<details>
<summary>解答・解説</summary>

**解答:** A

Azure Private Link は、Azure PaaS サービス（この場合は Azure SQL Database）にプライベート接続を提供するサービスです。これにより、以下の要件を満たすことができます：

1. **プライベート IP アドレスの割り当て**: Azure Private Link を使用すると、Azure SQL Database にプライベート IP アドレスを割り当てることができます。これにより、データベースはパブリックインターネットを経由せずに仮想ネットワーク内からアクセスできるようになります。
2. **仮想ネットワークからの接続のみを許可**: Private Link を使用すると、SQL Database へのアクセスを特定の仮想ネットワークからのみに制限できます。これにより、パブリックエンドポイントを完全に無効化することも可能です。
3. **Web アプリの仮想ネットワーク統合**: Azure Web App は仮想ネットワーク統合機能を使用して、Private Link 経由で SQL Database に安全に接続できます。

他の選択肢が適切でない理由：
- **サーバーレベルのファイアウォールとデータベースレベルのファイアウォール**: これらはパブリック IP アドレスベースのアクセス制御を提供しますが、SQL Database にプライベート IP アドレスを割り当てる機能はありません。
- **ネットワークセキュリティグループ（NSG）**: NSG は仮想ネットワーク内のリソース間のトラフィックを制御しますが、Azure SQL Database にプライベート IP アドレスを割り当てる機能はありません。

| 機能 | Azure Private Link | サーバー/DB ファイアウォール | NSG |
|---|---|---|---|
| プライベート IP アドレスの割り当て | はい | いいえ | いいえ |
| 仮想ネットワークからの接続のみを許可 | はい | 部分的（サービスエンドポイント使用時） | いいえ |
| パブリックエンドポイントの完全無効化 | はい | いいえ | いいえ |
| Web アプリとの仮想ネットワーク統合 | はい | 限定的 | 限定的 |

</details>

## 問題 52

Azure SQL Database に「DBI」という名前のデータベースがあり、その中に「index1」という名前の非クラスタ化インデックスが含まれています。エンドユーザーがこのインデックスを使用するとクエリが遅いと報告しています。インデックスに対して実行されている操作を特定する必要があります。どの動的管理ビューを使用すべきですか？（一つ選択）

- A. sys.dm_db_index_operational_stats
- B. sys.dm_db_index_usage_stats
- C. sys.dm_db_index_physical_stats
- D. sys.dm_exec_query_stats

<details>
<summary>解答・解説</summary>

**解答:** B

`sys.dm_db_index_usage_stats` は、インデックスの使用状況に関する統計情報を提供する動的管理ビューです。このビューを使用すると、インデックスに対する検索操作（seeks）、スキャン操作（scans）、ルックアップ操作（lookups）、および更新操作（updates）の回数を確認できます。これにより、インデックスがどのように使用されているか、またはほとんど使用されていないかを特定できます。

他の選択肢について：
- **sys.dm_db_index_operational_stats** - インデックスの I/O、ロック、ラッチ操作に関する低レベルの情報を提供しますが、インデックスの使用パターンを特定するには最適ではありません。
- **sys.dm_db_index_physical_stats** - インデックスの断片化レベルなど、インデックスの物理的な特性に関する情報を提供しますが、使用状況は表示しません。
- **sys.dm_exec_query_stats** - クエリのパフォーマンス統計を提供しますが、特定のインデックスの使用状況は直接示しません。

| 動的管理ビュー | 主な用途 | インデックス操作の特定に適しているか |
|---|---|---|
| sys.dm_db_index_usage_stats | インデックスの使用状況統計（seeks, scans, lookups, updates） | はい（最適） |
| sys.dm_db_index_operational_stats | インデックスの I/O、ロック、ラッチ操作の詳細 | 部分的 |
| sys.dm_db_index_physical_stats | インデックスの断片化レベルなどの物理的特性 | いいえ |
| sys.dm_exec_query_stats | クエリのパフォーマンス統計 | いいえ |

</details>

## 問題 53

オンプレミスのアプリケーション App1 があり、オンプレミスの Microsoft SQL Server 2016 データベース DB1 にデータを保存しています。複数の Azure リージョンに App1 の追加インスタンスをデプロイする計画があります。各リージョンには App1 と DB1 の個別のインスタンスが配置されます。DB1 の各インスタンスは Azure SQL Data Sync を使用して同期します。デプロイメントに適したデータベースサービスを推奨する必要があります。ソリューションは管理作業を最小限に抑える必要があります。推奨に何を含めるべきですか？（一つ選択）

- A. SQL Server on Azure Virtual Machines
- B. Azure Database for PostgreSQL
- C. Azure SQL Database シングルデータベース
- D. Azure SQL Managed Instance

<details>
<summary>解答・解説</summary>

**解答:** C

Azure SQL Database シングルデータベースが最適な選択です。その理由は：

1. PaaS（Platform as a Service）として提供されるため、インフラストラクチャの管理が不要
2. Azure SQL Data Sync をネイティブにサポート
3. 複数リージョンへの展開が容易
4. 自動バックアップ、パッチ適用、高可用性機能が組み込まれており、管理作業を大幅に削減

他の選択肢の問題点：
- **SQL Server on Azure Virtual Machines**: IaaS ソリューションであり、OS やデータベースエンジンの管理が必要で管理作業が増加
- **Azure Database for PostgreSQL**: SQL Server から PostgreSQL への移行には大幅なアプリケーション変更が必要
- **Azure SQL Managed Instance**: 単一インスタンスのデータベースよりも管理オーバーヘッドが大きい

| サービス | 管理作業 | SQL Data Sync サポート | 移行の容易さ |
|---|---|---|---|
| Azure SQL Database | 最小 | ネイティブ | 容易 |
| Azure SQL Managed Instance | 中程度 | サポート | 容易 |
| SQL Server on Azure VM | 多い | サポート | 最も容易 |
| Azure Database for PostgreSQL | 中程度 | 非対応 | 困難 |

Azure SQL Database シングルデータベースは、管理作業を最小限に抑えながら、複数リージョンでの展開と同期を実現するための最適なソリューションです。

</details>

## 問題 54

2 つの Azure Virtual Machines（VM1 と VM2）上に SQL Server 2019 CU8 のインスタンスが実行されています。VM1 と VM2 にフェイルオーバークラスターインスタンス（FCI）を展開する必要があります。この解決策では、分散ネットワーク名（DNN）とロードバランサーの必要性を排除する必要があります。どのように対応すべきですか？（一つ選択）

- A. VM1 と VM2 を同じ仮想ネットワーク上の単一のサブネットに接続する
- B. VM1 と VM2 を同じ Azure リージョン内の異なる近接配置グループに展開する
- C. VM1 と VM2 を単一の近接配置グループに展開する
- D. VM1 と VM2 を同じ仮想ネットワーク上の異なるサブネットに接続する

<details>
<summary>解答・解説</summary>

**解答:** A

SQL Server フェイルオーバークラスターインスタンス（FCI）を Azure VM に展開する際、分散ネットワーク名（DNN）やロードバランサーを使用せずに構成するには、VM1 と VM2 を同じ仮想ネットワーク上の単一のサブネットに接続する必要があります。

SQL Server 2019 CU8 以降では、Azure 共有ディスク機能を使用して FCI を構成できます。この構成では、同じサブネット上の VM が共有ディスクにアクセスし、従来の Windows Server フェイルオーバークラスタリング（WSFC）の機能を活用できます。

単一サブネット上の VM は、同じ IP アドレス空間内で通信でき、クラスターの仮想 IP アドレス（VIP）を使用してフェイルオーバー時のクライアント接続を維持できます。これにより、DNN やロードバランサーが不要になります。

異なるサブネットを使用する場合は、サブネット間でのフェイルオーバーを処理するために DNN またはロードバランサーが必要になります。また、近接配置グループは VM の物理的な近接性を確保するものですが、それだけでは DNN やロードバランサーの代替にはなりません。

| 構成オプション | DNN が必要 | ロードバランサーが必要 | 説明 |
|---|---|---|---|
| 同一サブネット | 不要 | 不要 | 同じ IP アドレス空間内でクラスター VIP を使用可能 |
| 異なるサブネット | 必要 | 必要 | サブネット間でのフェイルオーバーには追加の仕組みが必要 |
| 近接配置グループのみ | 影響なし | 影響なし | VM の物理的近接性のみを確保する機能 |

</details>

## 問題 55

データベース接続のセキュリティを強化するために、SQL Server 構成でサポートする最も安全な TLS（Transport Layer Security）バージョンはどれですか？（一つ選択）

- A. TLS 1.0
- B. TLS 1.3
- C. TLS 1.1
- D. TLS 1.2

<details>
<summary>解答・解説</summary>

**解答:** D

SQL Server でサポートされている最も安全な TLS バージョンは TLS 1.2 です。TLS（Transport Layer Security）はデータベース接続時のデータ暗号化に使用されるプロトコルで、バージョンが新しいほど一般的にセキュリティが強化されています。

TLS 1.0 と 1.1 は脆弱性が発見されており、多くの組織やコンプライアンス基準（PCI DSS など）では既に非推奨とされています。Microsoft SQL Server では、TLS 1.2 がサポートされている最も安全なバージョンとして推奨されています。

TLS 1.3 は最新のバージョンですが、現時点の SQL Server では正式にサポートされていません。

SQL Server のセキュリティ構成では、レジストリ設定を通じてサポートする TLS バージョンを制御できます。本番環境では、TLS 1.0 と 1.1 を無効にし、TLS 1.2 のみを有効にすることが推奨されるベストプラクティスです。

| TLS バージョン | SQL Server でのサポート状況 | セキュリティレベル | 推奨状況 |
|---|---|---|---|
| TLS 1.0 | サポート | 低（脆弱性あり） | 非推奨 |
| TLS 1.1 | サポート | 中（脆弱性あり） | 非推奨 |
| TLS 1.2 | サポート | 高 | 推奨 |
| TLS 1.3 | SQL Server 2022 以降でサポート（TDS 8.0 使用時） | 最高 | SQL Server 2022 + Windows Server 2022/Windows 11 環境で利用可能 |

> **【2026-04-21 精査注記】** TLS 1.3 は SQL Server 2022 で TDS 8.0 プロトコルを使用する際にサポートされるようになった（Windows Server 2022 または Windows 11 上で動作する場合）。SQL Server 2019 以前は TLS 1.3 未サポートのため、問題の答え（D: TLS 1.2）は SQL Server 2019 以前の環境を前提とする場合に正しい。

</details>

## 問題 56

オンプレミスのデータセンターに Microsoft SQL Server 2019 インスタンスがあります。このインスタンスには 4TB のデータベース DB1 が含まれています。DB1 を Azure SQL Database マネージドインスタンスに移行する計画があります。移行中のダウンタイムとデータ損失を最小限に抑えるには、何を使用すべきですか？（一つ選択）

- A. Azure Database Migration Service
- B. データベース ミラーリング
- C. 分散可用性グループ
- D. Always On 可用性グループ

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database マネージドインスタンスへの移行において、ダウンタイムとデータ損失を最小限に抑えるためには、Azure Database Migration Service（DMS）が最適なソリューションです。

Azure Database Migration Service は、オンプレミスのデータベースを Azure データベースサービスに移行するために設計された、フルマネージドサービスです。特に大規模なデータベース（この場合は 4TB）の移行に適しており、オンライン（ほぼ無停止）移行をサポートしています。オンライン移行モードを選択することで、ソースデータベースが稼働したままデータを移行でき、カットオーバー時のダウンタイムを最小限に抑えることができます。

他の選択肢が適切でない理由：
- **データベース ミラーリング**: SQL Server 間の機能であり、Azure SQL マネージドインスタンスへの移行ツールではありません。また、SQL Server の新しいバージョンでは非推奨となっています。
- **分散可用性グループ**: SQL Server インスタンス間のデータ同期に使用されますが、Azure SQL マネージドインスタンスへの移行ツールとしては設計されていません。
- **Always On 可用性グループ**: SQL Server の高可用性ソリューションですが、Azure SQL マネージドインスタンスへの移行ツールではありません。

| 移行オプション | 用途 | Azure SQL マネージドインスタンスへの移行に適しているか |
|---|---|---|
| Azure Database Migration Service | クラウド移行専用のマネージドサービス | はい（オンライン移行をサポート） |
| データベース ミラーリング | SQL Server 間の高可用性（非推奨） | いいえ |
| 分散可用性グループ | 複数の Availability Group 間のデータ同期 | いいえ |
| Always On 可用性グループ | SQL Server の高可用性ソリューション | いいえ |

> **【2026-04-21 精査注記】** 現在は **Managed Instance link**（SQL Server 2016 SP3 以降対応）がダウンタイム最小化移行の推奨方法となっている。DMS も引き続き有効だが、Managed Instance link はオンライン同期後のカットオーバー時間がより短い。問題作成時点では DMS が主流の選択肢だったため答えは A のままで問題ないが、試験では Managed Instance link を選択肢として出題される可能性もある点に注意。

</details>

## 問題 57

Azure AD テナントと、複数の Azure SQL データベースをホストする論理的な Microsoft SQL サーバー SQL1 があります。Azure Automation を使用して、Azure AD ユーザーにデータベースへの権限を自動的に割り当てる計画があります。必要な Automation アカウントを作成する必要があります。どの 2 つのアカウントを作成すべきですか？（二つ選択）

- A. Azure Active Directory の管理センターから、サービスプリンシパルを作成する
- B. SQL1 上で、データベース内に SQL ユーザーを作成する
- C. Azure Active Directory の管理センターから、SQL1 用のシステム割り当てマネージド ID を作成する
- D. SQL1 上で、SQL ログインを作成する
- E. Azure Active Directory の管理センターから、ユーザー割り当てマネージド ID を作成する

<details>
<summary>解答・解説</summary>

**解答:** A、B

Azure Automation で Azure SQL データベースに対する権限を自動的に割り当てるには、以下の 2 つのアカウントが必要です：

1. **サービスプリンシパル**: Azure AD の管理センターで作成するサービスプリンシパルは、Azure Automation が使用する認証エンティティとして機能します。このサービスプリンシパルには SQL Server に接続し、ユーザー権限を管理するための適切な権限が付与されます。
2. **SQL ユーザー**: SQL1 上のデータベース内に SQL ユーザーを作成することで、Azure AD ユーザーとデータベースのマッピングが可能になります。これにより、Azure AD ユーザーがデータベースにアクセスする際の権限を制御できます。

不正解の選択肢について：
- **SQL ログイン**: サーバーレベルの認証に使用されますが、Azure AD との統合には適していません。
- **ユーザー割り当てマネージド ID**: 有効なオプションですが、この特定のシナリオではサービスプリンシパルの方が一般的です。
- **システム割り当てマネージド ID**: 特定リソースに紐づく ID であり、このシナリオでは不要です。

| アカウントタイプ | 用途 | Azure Automation での役割 |
|---|---|---|
| サービスプリンシパル | Azure AD でのアプリケーション認証 | Automation ランブックの実行アイデンティティ |
| SQL ユーザー | データベースレベルでのアクセス制御 | Azure AD ユーザーとデータベース権限のマッピング |
| SQL ログイン | サーバーレベルでのアクセス制御 | Azure AD 統合には適さない |
| ユーザー割り当てマネージド ID | 複数リソース間で共有可能な ID | 特定のシナリオでは使用可能だが必須ではない |
| システム割り当てマネージド ID | 特定リソースに紐づく ID | このシナリオでは不要 |

</details>

## 問題 58

Azure サブスクリプションに SQL1 という名前の Azure SQL データベースがあります。SQL1 は可用性ゾーンをサポートしていない Azure リージョンにあります。同じリージョン内に SQL1 のセカンダリ レプリカを確保する必要があります。何を使用すべきですか？（一つ選択）

- A. ローカル冗長ストレージ（LRS）
- B. 自動フェールオーバー グループ
- C. アクティブ geo レプリケーション
- D. ゾーン冗長データベース

<details>
<summary>解答・解説</summary>

**解答:** B

正しい答えは「自動フェールオーバー グループ」です。自動フェールオーバー グループは、同じリージョン内または異なるリージョンにセカンダリ レプリカを作成できます。可用性ゾーンをサポートしていないリージョンでも使用可能です。

自動フェールオーバー グループは、Azure SQL データベースの高可用性を確保するための機能で、プライマリ データベースに障害が発生した場合に自動的にセカンダリ データベースにフェールオーバーします。同じリージョン内でも構成可能です。

各選択肢の説明：

| 選択肢 | 説明 | 適用可否 |
|---|---|---|
| ローカル冗長ストレージ（LRS） | Azure Storage の冗長オプションであり、SQL データベースのレプリケーションには適用されない | 不適切 |
| 自動フェールオーバー グループ | 同一リージョンまたは異なるリージョンにセカンダリ レプリカを作成できる | 適切 |
| アクティブ geo レプリケーション | 技術的には同一リージョン内にもセカンダリ レプリカを作成できるが、リージョン障害から保護されない | 技術的には可能だが目的に合わない |
| ゾーン冗長データベース | 可用性ゾーンをサポートするリージョンでのみ使用可能 | 不適切（問題条件を満たさない） |

> **【2026-04-21 精査注記】** アクティブ geo レプリケーションは技術的に同一リージョン内のセカンダリ レプリカ作成をサポートする（Microsoft ドキュメントで確認済み）。ただし「同一リージョン内のセカンダリ」はリージョン障害から保護しないため、"セカンダリ レプリカを確保する" という目的には自動フェールオーバー グループが引き続き正解。解説の「異なるリージョンにセカンダリ レプリカを作成するための機能」という記述は不正確。

</details>

## 問題 59

Azure SQL Database でクエリストアを使用しています。レポートのパラメータと実行プランによってレポートのパフォーマンスが低下していることに気づきました。レポートのパフォーマンスを向上させる必要があります。管理作業を最小限に抑える解決策を選ぶ必要があります。どうすべきですか？（一つ選択）

- A. DBCC FREEPROCCACHE コマンドを実行する
- B. 最も効率的な実行プランを強制する
- C. パラメータスニッフィングを無効にする
- D. スナップショット分離を有効にする

<details>
<summary>解答・解説</summary>

**解答:** B

「最も効率的な実行プランを強制する」が正解です。クエリストアは実行プランの履歴を保存しており、パフォーマンスが良好だった実行プランを特定して強制適用することができます。これにより、パラメータスニッフィングの問題を回避し、一貫したパフォーマンスを確保できます。また、管理作業も最小限で済みます。

他の選択肢について：
- **DBCC FREEPROCCACHE**: プラン・キャッシュをクリアしますが、問題の根本的な解決にはならず、新しいプランが再び非効率になる可能性があります。
- **パラメータスニッフィングを無効にする**: オプティマイザーがパラメータ値に基づいた最適化ができなくなり、全体的なパフォーマンスが低下する可能性があります。
- **スナップショット分離を有効にする**: トランザクションの分離レベルを変更するもので、実行プランの問題解決には直接関係ありません。

| 選択肢 | 管理作業 | 効果 | 適切さ |
|---|---|---|---|
| 最も効率的な実行プランを強制する | 最小限 | 特定の問題に対して直接的 | 最適 |
| DBCC FREEPROCCACHE | 中程度 | 一時的な解決策 | 不十分 |
| パラメータスニッフィングを無効化 | 中程度 | 他のクエリに悪影響の可能性 | 不適切 |
| スナップショット分離を有効化 | 高い | 実行プラン問題に無関係 | 不適切 |

## 問題 67

Azure SQL Database マネージドインスタンスのフェイルオーバーグループに対する災害復旧を計画しています。会社の SLA では、大規模な障害が発生した場合、フェイルオーバーグループ内のデータベースができるだけ早く利用可能になることが求められています。Read/Write フェイルオーバー ポリシーを Automatic に設定しました。この構成の結果として正しいものはどれですか？（二つ選択）

- A. データセンターまたは Azure リージョンの障害が発生した場合、データベースは自動的にフェイルオーバーします
- B. 障害発生時、プライマリ インスタンス内のデータベースは即時にフェイルオーバーします
- C. 障害発生時、個々のデータベースを選択的にフェイルオーバーさせることができます
- D. 障害発生時、データベースがフェイルオーバーするまでの最小遅延は 1 時間です
- E. 障害発生時、各データベースのフェイルオーバーに異なる猶予期間を設定できます

<details>
<summary>解答・解説</summary>

**解答:** A、B

Azure SQL Database マネージドインスタンスのフェイルオーバーグループで Read/Write フェイルオーバーポリシーを Automatic に設定すると、以下の効果があります：

1. **データセンターまたは Azure リージョンの障害が発生した場合、人間の介入なしに自動的にフェイルオーバーが実行されます**。これにより、SLA で要求される「できるだけ早く利用可能になる」という要件を満たします。
2. **自動フェイルオーバーでは、障害検出後、プライマリ インスタンス内のデータベースは即時にフェイルオーバープロセスを開始します**。手動操作を待つ必要がないため、復旧時間が短縮されます。

- **個々のデータベースを選択的にフェイルオーバーさせることはできません**。フェイルオーバーグループ内のすべてのデータベースが一緒にフェイルオーバーします。
- **フェイルオーバーの最小遅延は 1 時間ではありません**。自動フェイルオーバーは障害検出後すぐに開始されます。
- **各データベースに異なる猶予期間を設定することはできません**。フェイルオーバーグループの設定は、グループ内のすべてのデータベースに適用されます。

| フェイルオーバーポリシー | 特徴 | 人間の介入 | フェイルオーバー速度 |
|---|---|---|---|

**解答:** A

- B. 推定実行プラン（Estimated Execution Plan）
- C. ライブ クエリ統計（Live Query Statistics）
- D. 表示実行プラン（Display Execution Plan）

<details>
<summary>解答・解説</summary>

**解答:** B

SQL Server では、クエリが実行される際に実行プランが生成されますが、プランキャッシュに格納されるのは「推定実行プラン」です。

推定実行プランは、クエリオプティマイザーがクエリを分析して最適な実行方法を決定する際に生成されるプランで、実際のデータにアクセスせずに統計情報のみを使用して作成されます。このプランがプランキャッシュに保存され、同様のクエリが再実行される際に再利用されることでパフォーマンスが向上します。

|---|---|---|
| 推定実行プラン | 統計情報のみを使用して生成され、実際のデータにアクセスしない | はい |
| 実際の実行プラン | クエリ実行時に生成され、実際のデータアクセスと処理に基づく | いいえ |
| ライブ クエリ統計 | クエリ実行中にリアルタイムで進行状況を表示する | いいえ |
| 表示実行プラン | クライアントツールで表示用に整形された実行プラン | いいえ |

</details>

1. インデックスの再設計
2. ホットスポットとなるテーブルのパーティション分割
3. 該当するデータベースのファイル配置の最適化
4. アプリケーションコードの見直し

| 待機タイプ | 主な原因 | 適切な解決策 | 不適切な解決策 |
|---|---|---|---|
| PAGELATCH_UP | 同一ページへの同時アクセス競合 | インデックス再設計、パーティション分割 | データファイルの自動拡張設定変更 |

## 問題 76

Azure SQL Database の db1 に Sales という名前のテーブルがあります。Sales テーブルに変更追跡（Change Tracking）を実装する必要があります。最初に何をすべきですか？（一つ選択）

- A. db1 で Change Tracking を有効にする
- B. master データベースでマスターキーを作成する
- C. Sales テーブルに対して Change Tracking を有効にする
- D. db1 でデータベースレベルのトリガーを作成する

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database で変更追跡（Change Tracking）を実装するには、まず最初にデータベースレベルで Change Tracking を有効にする必要があります。その後、個々のテーブルに対して Change Tracking を有効にすることができます。

正しい手順は以下の通りです：
1. データベースレベルで Change Tracking を有効にする
2. 対象テーブル（この場合は Sales）に対して Change Tracking を有効にする

データベースレベルで Change Tracking を有効にするには、以下のような T-SQL を実行します：

```sql
ALTER DATABASE db1
SET CHANGE_TRACKING = ON
(CHANGE_RETENTION = 2 DAYS, AUTO_CLEANUP = ON)
```

```sql
ALTER TABLE Sales
ENABLE CHANGE_TRACKING
```

マスターキーの作成やデータベースレベルのトリガー作成は、Change Tracking の実装に直接関係ありません。

| 変更追跡の実装手順 | 説明 |

- A. sys.dm_tran_locks
- B. sys.dm_exec_requests
- C. sys.dm_os_waiting_tasks
- D. sys.dm_exec_sessions

<details>
<summary>解答・解説</summary>

**解答:** B

Azure SQL Managed Instance でブロッキングを引き起こしているセッションを特定するための適切な動的管理ビュー（DMV）を選択する必要があります。

`sys.dm_exec_requests` は現在実行中のリクエストに関する情報を提供し、各リクエストのブロッキング情報も含まれています。特に重要なのは、この DMV の `blocking_session_id` 列で、現在のリクエストをブロックしているセッションの ID を確認できます。

他の選択肢について：
- `sys.dm_tran_locks`: トランザクションによって保持されているロックに関する情報を提供しますが、直接ブロッキングセッションを特定するには追加の結合が必要です。
- `sys.dm_os_waiting_tasks`: 待機中のタスクに関する情報を提供しますが、ブロッキングセッションを直接特定するには適していません。

| 動的管理ビュー | 主な用途 | ブロッキング情報 |
|---|---|---|
| sys.dm_exec_requests | 現在実行中のリクエスト情報 | blocking_session_id 列でブロッキングセッションを直接特定可能 |
| sys.dm_tran_locks | ロック情報 | request_status='WAIT' のロックを確認可能だが、直接ブロッカーは特定できない |
| sys.dm_os_waiting_tasks | 待機中のタスク | blocking_session_id 情報はあるが低レベル情報が多い |
| sys.dm_exec_sessions | セッション情報 | ブロッキング情報なし |

| event_file | 詳細なイベント分析と永続化 | ディスク上のファイル | すべてのイベント詳細 |

## 問題 86

Azure SQL Database に保存されている秘密鍵がパスワードで暗号化された証明書があります。この秘密鍵のパスワードを変更する必要があります。どの Transact-SQL 文を実行すべきですか？（一つ選択）

- A. ALTER CERTIFICATE WITH PRIVATE KEY (DECRYPTION BY PASSWORD = '古いパスワード', ENCRYPTION BY PASSWORD = '新しいパスワード')
- B. CREATE CERTIFICATE FROM FILE WITH PRIVATE KEY (DECRYPTION BY PASSWORD = '古いパスワード', ENCRYPTION BY PASSWORD = '新しいパスワード')
- C. BACKUP CERTIFICATE TO FILE WITH PRIVATE KEY (DECRYPTION BY PASSWORD = '古いパスワード', ENCRYPTION BY PASSWORD = '新しいパスワード')
- D. ALTER MASTER KEY REGENERATE WITH ENCRYPTION BY PASSWORD = '新しいパスワード'

<details>
<summary>解答・解説</summary>

**解答:** A

正しい構文は以下のとおりです：

```sql
ALTER CERTIFICATE certificate_name

);
```

| コマンド | 目的 | 秘密鍵パスワード変更に適切か |

**解答:** A

Always On 可用性グループ（AG）のリスナーは、クライアントがプライマリ レプリカに接続するためのエンドポイントとして機能します。Azure 環境では、このリスナーが正しく機能するためにはロードバランサーの構成が必要です。

ロードバランサーは、クライアント接続をプライマリ レプリカに転送する役割を担います。フェールオーバーが発生した場合、ロードバランサーは自動的に新しいプライマリ レプリカにトラフィックをリダイレクトします。

他の選択肢が適切でない理由：
- **ネットワーク セキュリティ グループ（NSG）**: トラフィックのフィルタリングを行いますが、リスナーの機能には直接関係しません。
- **仮想ネットワーク ゲートウェイ**: VNet 間や オンプレミスとの接続に使用されますが、AG リスナーの機能とは無関係です。
- **ネットワーク インターフェイス カード（NIC）**: VM のネットワーク接続に使用されますが、AG リスナーの動作に直接必要なコンポーネントではありません。

| コンポーネント | AG リスナーとの関係 | 主な機能 |
|---|---|---|
| ロードバランサー | 必須 | クライアント接続をプライマリ レプリカに転送、フェールオーバー時の自動リダイレクト |
| NSG | 補助的 | ネットワークトラフィックのフィルタリング |
| 仮想ネットワーク ゲートウェイ | 関連性低 | オンプレミスと Azure 間の接続 |
| NIC | 基盤要素 | 各 VM のネットワーク接続 |

</details>

## 問題 97

自動チューニングによって更新されたプランのステータスを表示する DMV はどれですか？（一つ選択）

- A. sys.dm_db_tuning_recommendations
- B. sys.query_store_query_text
- C. sys.dm_db_automatic_tuning_options
- D. sys.query_store_plan

<details>
<summary>解答・解説</summary>

**解答:** A

sys.dm_db_tuning_recommendations DMV（動的管理ビュー）は、Azure SQL Database や SQL Server 2017 以降で自動チューニング機能によって生成された推奨事項の詳細を表示します。このビューには、自動チューニングによって更新されたプランのステータスや、実装された推奨事項の詳細が含まれています。

他の選択肢について：
- **sys.query_store_query_text**: Query Store に保存されているクエリのテキストを表示するためのビューであり、自動チューニングのステータスは表示しません。
- **sys.dm_db_automatic_tuning_options**: データベースで利用可能な自動チューニングオプションとその現在の構成状態を表示しますが、プランの更新ステータスは表示しません。
- **sys.query_store_plan**: Query Store に保存されている実行プランに関する情報を表示しますが、自動チューニングによる更新ステータスは直接表示しません。

| DMV 名 | 主な用途 |
|---|---|
| sys.dm_db_tuning_recommendations | 自動チューニングによる推奨事項とプラン更新のステータスを表示 |
| sys.dm_db_automatic_tuning_options | 自動チューニング機能の設定状態を表示 |
| sys.query_store_query | Query Store に保存されているクエリの情報を表示 |
| sys.query_store_plan | Query Store に保存されている実行プランの情報を表示 |
| sys.query_store_query_text | Query Store に保存されているクエリのテキストを表示 |

</details>

## 問題 98

SQL Server データファイルを Azure VM で使用する場合、どのタイプのストレージを使用すべきですか？（一つ選択）

- A. Azure Files
- B. Azure Blob Storage
- C. Azure Managed Disks
- D. Azure Table Storage

<details>
<summary>解答・解説</summary>

**解答:** C

SQL Server データファイルを Azure VM で使用する場合、最適なストレージ選択肢は Azure Managed Disks です。

Azure Managed Disks は、Azure VM 用に設計された永続的なブロックストレージであり、SQL Server のデータファイルに最適な選択肢です。以下の理由から SQL Server のワークロードに適しています：

1. 高いパフォーマンス - Premium SSD や Ultra Disk などのオプションにより SQL Server の I/O 要件を満たす高いスループットと低レイテンシを提供します
2. 高可用性 - 99.999% の可用性を提供
3. スケーラビリティ - ディスクサイズを動的に変更可能
4. 簡易管理 - バックアップやスナップショットの管理が容易

他の選択肢が適切でない理由：
- **Azure Blob Storage**: オブジェクトストレージであり、ブロックレベルのアクセスを必要とする SQL Server データファイルには適していません。
- **Azure Files**: SMB プロトコルを使用するファイル共有サービスで、SQL Server の本番環境のパフォーマンス要件を満たせない場合があります。
- **Azure Table Storage**: NoSQL データストアであり、SQL Server データファイルの保存には使用できません。

| ストレージタイプ | 用途 | SQL Server データファイルへの適合性 |
|---|---|---|
| Azure Managed Disks | ブロックレベルストレージ、VM 用 | 高（推奨） |
| Azure Blob Storage | オブジェクトストレージ | 低 |
| Azure Files | SMB ファイル共有 | 中（特定のシナリオのみ） |
| Azure Table Storage | NoSQL データストア | 不適合 |

</details>

## 問題 99

Azure SQL Managed Instance を使用しており、SQL Agent Job の電子メール通知を有効にする必要があります。どのような対応をすべきですか？（一つ選択）

- A. Database Mail を構成し、SQL Agent の通知プロファイルを設定する
- B. msdb データベースで sp_set_sqlagent_properties ストアドプロシージャを実行する
- C. Azure Portal から SQL Agent の通知設定を構成する
- D. Agent XPs オプションを有効にする

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Managed Instance で SQL Agent Job の電子メール通知を有効にするには、以下の手順が必要です：

1. Database Mail を構成する
2. SQL Agent の通知プロファイルを設定する

Azure SQL Managed Instance では、SQL Server のオンプレミス版と同様に、ジョブ通知を送信するために Database Mail 機能を使用します。SQL Agent は Database Mail を使用してメール通知を送信するため、最初に Database Mail を構成し、次に SQL Agent がそれを使用するように設定する必要があります。

他の選択肢が不適切な理由：
- **sp_set_sqlagent_properties**: 正しいストアドプロシージャですが、これだけでは不十分です。Database Mail の構成が先に必要です。
- **Azure Portal から直接 SQL Agent の通知設定を構成することはできません**: T-SQL コマンドを使用して構成する必要があります。
- **Agent XPs オプション**: SQL Managed Instance ではデフォルトで有効になっているため、別途有効化する必要はありません。

| 設定手順 | 説明 | 必要性 |
|---|---|---|
| Database Mail の構成 | メールプロファイルとアカウントを作成し、SMTP サーバー設定を構成 | 必須 |
| SQL Agent 通知プロファイルの設定 | sp_set_sqlagent_properties を使用して SQLAgentMailProfile を設定 | 必須 |
| Agent XPs の有効化 | SQL Managed Instance ではデフォルトで有効 | 不要 |
| Azure Portal での設定 | SQL Managed Instance では Portal からの直接設定は不可 | 不可能 |

</details>

## 問題 100

Azure Logic App を作成して、Azure SQL Database の「db1」にある「Sales」テーブルにデータを挿入しています。Logic App は「行の挿入」アクションを使用していますが、Sales テーブルにトリガーがあるため、実行のたびにエラーが発生します。Sales テーブルに影響を与えずに Logic App を修正する必要があります。どの 3 つのアクションを実行すべきですか？（三つ選択）

- A. Sales テーブルに行を挿入するストアドプロシージャを db1 に作成する
- B. Logic App から「行の挿入」アクションを削除する
- C. Logic App に「ストアドプロシージャの実行」アクションを作成する
- D. Logic App に「アイテムが作成されたとき」トリガーを追加する
- E. Sales テーブルからトリガーを削除する

<details>
<summary>解答・解説</summary>

**解答:** A、B、C

この問題では、Azure Logic App が Sales テーブルに直接データを挿入しようとしてエラーが発生しています。テーブルにトリガーがあるため問題が起きていますが、テーブル側は変更せずに Logic App 側で対応する必要があります。

正しい解決策は以下の 3 ステップです：

1. **Sales テーブルに行を挿入するストアドプロシージャを db1 に作成する** - これにより、トリガーを考慮した適切なデータ挿入ロジックを実装できます
2. **Logic App から「行の挿入」アクションを削除する** - 問題の原因となっている直接挿入アクションを取り除きます
3. **Logic App に「ストアドプロシージャの実行」アクションを作成する** - 作成したストアドプロシージャを呼び出すように変更します

「アイテムが作成されたとき」トリガーを追加することは、Logic App の起動条件を変更するだけで、Sales テーブルへの挿入エラーは解決しません。また、Sales テーブルからトリガーを削除するという選択肢は、問題の条件「Sales テーブルに影響を与えずに」という要件に反します。

| アプローチ | 説明 | Sales テーブルへの影響 | 問題解決 |
|---|---|---|---|
| ストアドプロシージャ作成 | テーブル挿入ロジックを SQL に移動 | なし | はい |
| 行の挿入アクション削除 | 問題のあるアクションを除去 | なし | 部分的 |
| ストアドプロシージャ実行アクション追加 | 新しい挿入方法を提供 | なし | はい |

</details>
