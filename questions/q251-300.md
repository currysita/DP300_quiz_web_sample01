## 問題 251

Microsoft SQL Server 2019 インスタンスがあります。バックアップメンテナンスソリューションを構成しています。データベースを任意の時点に復元できるようにする必要があります。どの 2 種類のバックアップを含める必要がありますか？（二つ選択）

- A. 差分バックアップ
- B. フルデータベースバックアップ
- C. ファイルグループバックアップ
- D. トランザクションログバックアップ
- E. 部分バックアップ

<details>
<summary>解答・解説</summary>

**解答:** B、D

任意の時点（point in time）へのリストアを実現するには、フルデータベースバックアップとトランザクションログバックアップの組み合わせが必要です。

フルデータベースバックアップはデータベース全体のスナップショットを提供し、リカバリの基点となります。トランザクションログバックアップは、フルバックアップ以降に発生したすべてのトランザクションを記録しており、これを使用して特定の時点までデータベースを復元できます。

他の選択肢が適切でない理由：

- 差分バックアップは最後のフルバックアップ以降に変更されたデータのみをバックアップしますが、point-in-time リカバリには不十分です
- ファイルグループバックアップはデータベースの特定部分のみをバックアップするため、完全なリカバリには不十分です
- 部分バックアップはデータベースの一部のみをバックアップするため、完全なリカバリには不十分です

| バックアップの種類 | 説明 | Point-in-Time リカバリに必要か |
|---|---|---|
| フルデータベース | データベース全体のスナップショット | はい（基点として必須） |
| トランザクションログ | データベースに対するすべての変更を記録 | はい（時点指定に必須） |
| 差分 | 最後のフルバックアップ以降の変更のみ | いいえ（補助的） |
| ファイルグループ | 特定のファイルグループのみ | いいえ（部分的） |
| 部分 | 読み取り専用ファイルグループを除外可能 | いいえ（部分的） |

</details>

## 問題 252

Azure Virtual Machines 上の SQL Server インスタンスに、DB1 という名前のデータベースがあります。DB1 に対して Azure SQL Data Sync を実装する予定です。どの分離レベルを構成すべきですか？（一つ選択）

- A. REPEATABLE READ
- B. READ COMMITTED
- C. SNAPSHOT
- D. SERIALIZABLE

<details>
<summary>解答・解説</summary>

**解答:** C

Azure SQL Data Sync は、複数のデータベース間でデータを同期するためのサービスです。Data Sync を効率的に実行するには、適切なトランザクション分離レベルを設定する必要があります。

SNAPSHOT は、Azure SQL Data Sync に推奨される分離レベルです。この分離レベルでは、トランザクションの開始時点でのデータのスナップショットを使用し、他のトランザクションによる変更を見ることはありません。これにより、同期プロセス中のブロッキングが最小限に抑えられ、データの整合性を維持しながら効率的な同期が可能になります。

Azure SQL Data Sync を実装する際は、同期パフォーマンスを最適化し、データの整合性を確保するために SNAPSHOT 分離レベルを使用することが推奨されています。これにより、同期プロセス中のブロッキングを回避しながら、一貫したデータビューを維持できます。

| 分離レベル | 特徴 | Data Sync への適合性 |
|---|---|---|
| READ UNCOMMITTED | 最も低い分離レベル、ダーティリードが発生する可能性あり | 同期に不適切（データ整合性の問題） |
| READ COMMITTED | コミットされたデータのみ読み取り可能 | 同期中にブロッキングが発生する可能性あり |
| REPEATABLE READ | 読み取りデータの一貫性を保証 | ブロッキングが増加し、同期パフォーマンスに影響 |
| SERIALIZABLE | 最も厳格な分離レベル | 同期中に深刻なブロッキングが発生し、パフォーマンスが低下 |
| SNAPSHOT | 読み取り操作がブロックされない | Data Sync に最適（ブロッキングなしでデータ整合性を維持） |

</details>

## 問題 253

Azure Virtual Machines 上の SQL Server インスタンスをデプロイしました。自動パッチ適用を有効にする必要があります。どの 2 つのコマンドレットを実行すべきですか？（二つ選択）

- A. Set-AzVMSqlServerExtension
- B. New-AzVMSqlServerAutoPatchingConfig
- C. Register-AzResourceProvider
- D. New-AzVMConfig
- E. Enable-AzVMPatchOrchestration

<details>
<summary>解答・解説</summary>

**解答:** A、B

Azure VM 上の SQL Server で自動パッチ適用を有効にするには、以下の 2 つのコマンドレットを組み合わせて使用します：

1. **New-AzVMSqlServerAutoPatchingConfig** - このコマンドレットは自動パッチ適用の設定を構成します。パッチ適用のスケジュール、メンテナンス時間、パッチカテゴリなどを指定できます。
2. **Set-AzVMSqlServerExtension** - このコマンドレットは、SQL Server IaaS Agent 拡張機能を設定し、前のステップで作成した自動パッチ適用の構成を適用します。

不正解の選択肢について：

- **New-AzVMConfig** - 新しい VM 構成オブジェクトを作成するためのコマンドレットですが、既存の VM の自動パッチ適用を有効にする直接的な手段ではありません。
- **Register-AzResourceProvider** - Azure リソースプロバイダーを登録するためのコマンドレットで、SQL Server の自動パッチ適用とは関係ありません。
- **Enable-AzVMPatchOrchestration** - 実際には存在しないコマンドレットです。Azure VM のパッチ管理には別のアプローチが使用されます。

| コマンドレット | 目的 | 自動パッチ適用との関連 |
|---|---|---|
| New-AzVMSqlServerAutoPatchingConfig | 自動パッチ適用の設定を構成 | 直接関連（必要） |
| Set-AzVMSqlServerExtension | SQL Server IaaS Agent 拡張機能を設定 | 直接関連（必要） |
| New-AzVMConfig | 新しい VM 構成オブジェクトを作成 | 間接的な関連 |
| Register-AzResourceProvider | Azure リソースプロバイダーを登録 | 関連なし |
| Enable-AzVMPatchOrchestration | 存在しないコマンドレット | 該当なし |

SQL Server IaaS Agent 拡張機能は、自動パッチ適用、自動バックアップ、Azure Key Vault との統合など、Azure VM 上の SQL Server に特化した管理機能を提供します。

</details>

## 問題 254

SQL Server on Azure Virtual Machine（SQL1）のメトリクスを監視し、Kusto クエリ言語を使用してクエリを実行する必要があります。管理作業を最小限に抑える解決策が必要です。メトリクスはどこに保存すべきですか？（一つ選択）

- A. Azure Blob Storage アカウント
- B. Log Analytics ワークスペース
- C. Azure Event Hubs
- D. Azure Monitor メトリクスデータベース

<details>
<summary>解答・解説</summary>

**解答:** B

SQL Server on Azure Virtual Machine のメトリクスを監視し、Kusto クエリ言語（KQL）を使用してクエリを実行するには、Log Analytics ワークスペースが最適な選択肢です。

Log Analytics ワークスペースは Azure Monitor の一部であり、以下の理由から最適な選択となります：

1. ネイティブな Kusto クエリ言語（KQL）サポート - Log Analytics は KQL を使用してデータをクエリするように設計されています
2. 管理作業の最小化 - 設定が簡単で、Azure Portal から直接管理できます
3. SQL Server 用の統合監視ソリューション - SQL VM 用の監視エージェントと直接統合されています
4. データの保持とクエリのパフォーマンス - メトリクスデータの長期保存と高速なクエリ実行をサポートします

他の選択肢が適切でない理由：

- Azure Blob Storage はデータの保存には適していますが、Kusto クエリ言語を直接サポートしていません
- Azure Event Hubs はデータストリーミングのためのサービスであり、クエリ機能が組み込まれていません
- Azure Monitor メトリクスデータベースは存在しますが、KQL ではなく別のクエリ言語を使用します

| サービス | Kusto クエリ言語サポート | 管理の容易さ | SQL Server との統合 | 主な用途 |
|---|---|---|---|---|
| Log Analytics ワークスペース | ネイティブサポート | 高い（最小限の管理作業） | 直接統合 | ログとメトリクスの分析 |
| Azure Blob Storage | サポートなし | 中程度 | 追加設定が必要 | データ保存 |
| Azure Event Hubs | サポートなし | 中程度 | 追加設定が必要 | データストリーミング |
| Azure Monitor メトリクス | 限定的 | 高い | 部分的 | リアルタイムメトリクス |

SQL Server on Azure VM のメトリクスを Log Analytics ワークスペースに送信するには、Azure Monitor エージェントをインストールし、適切なデータ収集ルールを設定します。これにより、KQL を使用してメトリクスデータに対して複雑なクエリを実行できるようになります。

</details>

## 問題 255

データ移行プロセスにおいて、セルフホステッド統合ランタイム（Self-hosted Integration Runtime）の主な役割は何ですか？（一つ選択）

- A. ソースとターゲットに接続し、データ移行を容易にする
- B. 移行されたデータを一時的にキャッシュして転送速度を向上させる
- C. データ移行プロセス中にデータの変換と正規化を実行する
- D. 移行されたデータの整合性を検証するためのチェックサムを生成する

<details>
<summary>解答・解説</summary>

**解答:** A

セルフホステッド統合ランタイム（Self-hosted Integration Runtime）は、Azure Data Factory や Azure Synapse Analytics などのサービスで使用される重要なコンポーネントです。その主な役割は、プライベートネットワーク内のデータソースとクラウドデータサービスの間の橋渡しをすることです。

セルフホステッド統合ランタイムの主な機能：

1. 接続性の提供：オンプレミスやプライベートネットワーク内のデータソースへの安全な接続を確立します
2. データ移動の実行：ソースからターゲットへのデータ転送を処理します
3. アクティビティのディスパッチ：データ変換などのアクティビティを実行します

他の選択肢が不正解である理由：

- データのキャッシュは統合ランタイムの主な役割ではありません
- データ変換は可能ですが、主な役割は接続と移行の実行です
- チェックサム生成は統合ランタイムの標準機能ではありません

| 統合ランタイムの種類 | 主な用途 | ネットワーク環境 |
|---|---|---|
| Azure 統合ランタイム | パブリックにアクセス可能なデータソース | クラウド内 |
| セルフホステッド統合ランタイム | プライベートネットワーク内のデータソース | オンプレミス/VNet |
| Azure-SSIS 統合ランタイム | SSIS パッケージの実行 | クラウド内 |

セルフホステッド統合ランタイムは、ファイアウォールの内側やプライベートネットワーク内のデータソースにアクセスする必要がある場合に特に重要です。これにより、セキュリティを維持しながらハイブリッドデータ統合シナリオが可能になります。

</details>

## 問題 256

複数のテナントのデータを Azure SQL Database エラスティックプールを使用して保存するアプリケーションがあります。各テナントには独自のデータベースがあります。アプリケーションの認証ソリューションを構成する必要があります。ソリューションは次の要件を満たす必要があります：ユーザーが自分のテナントのデータベースにのみアクセスできるようにする、データベースを専用インスタンスに移動できるようにする、管理作業を最小限に抑える。ソリューションに何を含めるべきですか？（一つ選択）

- A. Azure Active Directory グループ
- B. 包含データベース ユーザー
- C. データベース ロール
- D. サーバー ロール

<details>
<summary>解答・解説</summary>

**解答:** B

この問題では、マルチテナント環境での Azure SQL Database の認証方法について問われています。

包含データベース ユーザー（Contained Database Users）は、この要件に最適な選択肢です。その理由は：

1. テナント分離 - 包含データベース ユーザーはデータベース内に存在し、特定のデータベースにのみアクセス権を持ちます。これにより、ユーザーは自分のテナントのデータベースにのみアクセスできます。
2. データベース移行の容易さ - 包含データベース ユーザーはデータベース自体に含まれているため、データベースを別のサーバーやインスタンスに移動しても認証情報が維持されます。これは「データベースを専用インスタンスに移動できる」という要件を満たします。
3. 管理作業の軽減 - 包含データベース ユーザーを使用すると、サーバーレベルのログインを作成・管理する必要がなく、データベースレベルでユーザー管理ができるため管理作業が軽減されます。

他の選択肢が適切でない理由：

- Azure Active Directory グループ：テナント分離は可能ですが、データベース移行時に追加設定が必要になる場合があります。
- データベース ロール：ロールはアクセス権の管理に使用されますが、認証メカニズム自体ではありません。
- サーバー ロール：サーバーレベルでの権限管理であり、データベースごとのテナント分離には適していません。

| 認証方法 | テナント分離 | データベース移行 | 管理作業 |
|---|---|---|---|
| 包含データベース ユーザー | 各 DB に閉じている | 容易（DB 内に含まれる） | 最小限 |
| Azure AD グループ | 可能 | 追加設定が必要 | 中程度 |
| データベース ロール | 権限管理のみ | 権限の再設定が必要 | 中程度 |
| サーバー ロール | サーバーレベル | 再設定が必要 | 高い |

</details>

## 問題 257

データベース管理者として、インデックスのメンテナンスを自動化したいと考えています。どのプラットフォームが自動インデックス管理機能をネイティブにサポートしていますか？（一つ選択）

- A. Azure SQL Database
- B. SQL Server on-premises
- C. Azure SQL Managed Instance
- D. SQL Server in an Azure VM

<details>
<summary>解答・解説</summary>

**解答:** A

Azure SQL Database は、自動インデックス管理機能である「自動チューニング」（Automatic Tuning）をネイティブにサポートしています。この機能は、ワークロードパターンを継続的に監視し、インデックスの作成や削除を自動的に行うことでパフォーマンスを最適化します。

Azure SQL Database の自動チューニング機能には以下の特徴があります：

- インデックスの作成推奨（CREATE INDEX recommendations）
- 未使用インデックスの削除推奨（DROP INDEX recommendations）
- クエリ実行プランの強制（FORCE LAST GOOD PLAN）

他の選択肢については：

- SQL Server on-premises：手動でのインデックス管理が必要です。Query Store と組み合わせて使用することで推奨事項は得られますが、自動実行はされません。
- Azure SQL Managed Instance：Azure SQL Database と同様の機能を持ちますが、一部の自動チューニング機能に制限があります。
- SQL Server in an Azure VM：基本的に on-premises と同じで、自動インデックス管理機能はネイティブには含まれていません。

| プラットフォーム | 自動インデックス管理 | 備考 |
|---|---|---|
| Azure SQL Database | あり | 完全な自動チューニング機能をサポート |
| SQL Server on-premises | なし | 手動管理が必要 |
| Azure SQL Managed Instance | 部分的 | 一部の自動チューニング機能に制限あり |
| SQL Server in an Azure VM | なし | 手動管理が必要 |

Azure SQL Database の自動チューニング機能は、DBA の作業負荷を軽減し、データベースのパフォーマンスを継続的に最適化するのに役立ちます。

</details>

## 問題 258

Azure SQL データベースの可用性戦略を推奨する必要があります。この戦略は以下の要件を満たす必要があります：クライアントアプリケーションが接続文字列を変更する必要のないフェイルオーバーをサポートする、データベースをセカンダリ Azure リージョンにレプリケートする、セカンダリリージョンへのフェイルオーバーをサポートする。推奨事項に何を含めるべきですか？（一つ選択）

- A. フェイルオーバーグループ
- B. geo-レプリケーション
- C. トランザクションレプリケーション
- D. 可用性ゾーン

<details>
<summary>解答・解説</summary>

**解答:** A

フェイルオーバーグループは、Azure SQL Database の可用性を確保するための機能で、以下の理由から正解です：

1. フェイルオーバーグループは、クライアントアプリケーションが接続文字列を変更することなくフェイルオーバーを処理できます。これは、フェイルオーバーグループがリスナーエンドポイントを提供し、フェイルオーバー時に自動的に新しいプライマリにリダイレクトするためです。
2. フェイルオーバーグループは、データベースをセカンダリ Azure リージョンにレプリケートする機能を提供します。
3. フェイルオーバーグループは、セカンダリリージョンへの自動または手動フェイルオーバーをサポートしています。

他の選択肢が不適切な理由：

- geo-レプリケーション：基本的なクロスリージョンレプリケーションを提供しますが、フェイルオーバーグループとは異なり、クライアント接続文字列の自動リダイレクトを提供しません。フェイルオーバー時にアプリケーション側で接続文字列の変更が必要です。
- トランザクションレプリケーション：主にデータ配布のためのものであり、高可用性ソリューションとしては設計されていません。クロスリージョンフェイルオーバーの自動管理機能がありません。
- 可用性ゾーン：単一リージョン内のデータセンター障害に対する保護を提供しますが、リージョン間レプリケーションやクロスリージョンフェイルオーバーは提供しません。

| 機能 | クライアント接続文字列の維持 | クロスリージョンレプリケーション | 自動フェイルオーバー |
|---|---|---|---|
| フェイルオーバーグループ | はい | はい | はい |
| geo-レプリケーション | いいえ | はい | いいえ（手動のみ） |
| トランザクションレプリケーション | いいえ | はい（ただし目的が異なる） | いいえ |
| 可用性ゾーン | はい（リージョン内） | いいえ | はい（リージョン内のみ） |

</details>

## 問題 259

Azure 環境において、最も低いレイテンシーを提供するストレージの種類はどれですか？（一つ選択）

- A. Premium SSD
- B. Standard HDD
- C. Ultra SSD
- D. Standard SSD

<details>
<summary>解答・解説</summary>

**解答:** C

Ultra SSD は、Azure 内で最も低いレイテンシーと高い IOPS を提供するストレージオプションです。これは特に、SAP HANA、SQL Server、Oracle、その他の高性能データベースなど、ミッションクリティカルなワークロードに最適です。Ultra SSD は 1 ミリ秒未満のレイテンシーを実現できます。

他のオプションと比較すると：

- Premium SSD は、Ultra SSD の次に高性能ですが、レイテンシーは Ultra SSD より高くなります。
- Standard SSD は、中程度のパフォーマンスを提供し、コスト効率が良いオプションです。
- Standard HDD は最も低コストですが、最も高いレイテンシーを持ちます。

| ディスクタイプ | レイテンシー | IOPS | スループット | ユースケース |
|---|---|---|---|---|
| Ultra SSD | 最低（<1ms） | 最高 | 最高 | ミッションクリティカルなワークロード、高性能データベース |
| Premium SSD | 低（<5ms） | 高 | 高 | 本番環境のアプリケーション、パフォーマンス重視のワークロード |
| Standard SSD | 中（<10ms） | 中 | 中 | Web サーバー、軽量アプリケーション |

- E. DB1 のプロパティから Active Geo-Replication を有効にする

<details>
<summary>解答・解説</summary>

**解答:** B、C

1. **Server2 という名前の新しい Azure SQL Database サーバーを作成する（C）**：フェールオーバーグループにはセカンダリサーバーが必要です。Server1（プライマリ）とは別のリージョンに Server2 を作成します。

| Server2 の作成（C） | 必要 | フェールオーバーグループにはセカンダリサーバーが必要 |
| 自動バックアップ保持期間の構成（D） | 不要 | リージョン停止対応には不十分 |

</details>

## 問題 289

Azure SQL データベース DB1 があります。20 個の CHAR(400) 型の列を持つ Table1 というテーブルがあり、行圧縮が有効になっています。データベース監査中にどのフィールドも 150 文字以上含まれていないことがわかりました。Table1 にページ圧縮を適用できるようにする必要があります。どうすべきですか？（一つ選択）

- A. 列をスパースとして構成する
- B. 列の型を varchar(200) に変更する
- C. 列の型を nvarchar(MAX) に変更する
- D. 列の型を varchar(150) に変更する

<details>
<summary>解答・解説</summary>

**解答:** B

CHAR データ型は固定長であり、指定した長さ分のストレージを常に使用します。この場合、実際のデータが 150 文字以下であっても、各列は 400 文字分のスペースを占有しています。これは非効率的であり、ページ圧縮の効果を制限します。

ページ圧縮を効果的に適用するためには、固定長データ型よりも可変長データ型の方が圧縮効果が高くなります。正解は「列の型を varchar(200) に変更する」です。これにより：

- 固定長 CHAR(400) から可変長 varchar(200) に変更することで、実際に使用されているデータ長（150 文字以下）に基づいてストレージが割り当てられます
- 200 という上限は、現在の最大使用量（150 文字）を十分にカバーしています
- MAX 指定は不要であり、むしろ大きなオーバーヘッドを生じる可能性があります

| データ型 | 特性 | ページ圧縮への影響 |
|---|---|---|
| CHAR(400) | 固定長、常に 400 文字分のスペースを使用 | 非効率的、圧縮効果が低い |
| varchar(200) | 可変長、実際のデータ長に基づくスペース使用 | 効率的、圧縮効果が高い |
| varchar/nvarchar(MAX) | 2GB までの可変長、追加のオーバーヘッドあり | 小さなデータには過剰、非効率的 |
| スパース列 | NULL 値の格納に最適化、非 NULL 値にはオーバーヘッド | ページ圧縮問題の直接的な解決策ではない |

スパース列は NULL 値の多い列に適していますが、この問題ではデータが実際に存在し、ただ長さが短いという状況なので適切ではありません。また、nvarchar は Unicode データ用であり、この状況では必要ないと考えられます。

</details>

## 問題 292

データベース管理者に通知するソリューションを実装する必要があります。このソリューションは監視要件を満たす必要があります。どのように実装すべきですか？（一つ選択）

- A. 静的しきい値を持つ Azure Monitor アラートルールを作成し、そのアラートルールをアクショングループに割り当てる
- B. QueryStoreRuntimeStatistics をログに記録し、Azure Event Hub にストリーミングする診断設定を追加する
- C. 動的しきい値を持つ Azure Monitor アラートルールを作成し、そのアラートルールをアクショングループに割り当てる
- D. SQL Server エージェントジョブを作成し、パフォーマンスカウンターをモニタリングする

<details>
<summary>解答・解説</summary>

**解答:** A

正解は「静的しきい値を持つ Azure Monitor アラートルールを作成し、そのアラートルールをアクショングループに割り当てる」です。Azure Monitor アラートルールを使用すると、特定のメトリックがしきい値を超えたときに通知を送信できます。静的しきい値は、明確な基準値に基づいてアラートをトリガーするため、予測可能な監視要件に適しています。アクショングループを使用することで、アラートが発生した際に管理者への通知方法（メール、SMS、音声通話など）を定義できます。

- B. `BACKUP DATABASE DB1 TO URL='https://storageaccount.blob.core.windows.net/container/DB1.bak' WITH (MIRROR_BACKUP, STATS=5, COMPRESSION);`
- C. `BACKUP DATABASE DB1 TO URL='https://storageaccount.blob.core.windows.net/container/DB1.bak' WITH (SECONDARY_ONLY, STATS=5, COMPRESSION);`
- D. `BACKUP DATABASE DB1 TO URL='https://storageaccount.blob.core.windows.net/container/DB1.bak' WITH (NoInit, STATS=5, COMPRESSION);`

<details>
<summary>解答・解説</summary>

**解答:** A

Always On 可用性グループ環境でセカンダリレプリカ（この場合は SQL2）からバックアップを実行する場合、COPY_ONLY オプションを使用する必要があります。

- SECONDARY_ONLY というオプションは存在しません
- NoInit オプションは既存のバックアップメディアに追加するためのオプションであり、セカンダリレプリカからバックアップを取る際の要件とは関係ありません

| バックアップオプション | 説明 | セカンダリレプリカでの使用 |
|---|---|---|
| COPY_ONLY | バックアップチェーンに影響を与えないバックアップを作成 | 推奨 |
| MIRROR_BACKUP | ミラーバックアップセットを作成 | 不適切 |
| SECONDARY_ONLY | 存在しないオプション | 不適切 |
| NoInit | 既存メディアへの追加 | 不適切 |

</details>

## 問題 296

次の Azure Data Factory パイプラインがあります：「Ingest Data from System1」、「Ingest Data from System2」、「Populate Dimensions」、「Populate Facts」。「Ingest Data from System1」と「Ingest Data from System2」には依存関係がありません。「Populate Dimensions」は「Ingest Data from System1」と「Ingest Data from System2」の後に実行する必要があります。「Populate Facts」は「Populate Dimensions」パイプラインの後に実行する必要があります。すべてのパイプラインは 8 時間ごとに実行する必要があります。パイプラインの実行をスケジュールするには何をすべきですか？（一つ選択）

- A. すべての 4 つのパイプラインにイベントトリガーを追加する
- B. 親パイプラインを作成して 4 つのパイプラインを含め、スケジュールトリガーを使用する
- C. すべての 4 つのパイプラインにスケジュールトリガーを追加する
- D. 親パイプラインを作成して 4 つのパイプラインを含め、イベントトリガーを使用する

<details>
<summary>解答・解説</summary>

**解答:** B

正解は「親パイプラインを作成して 4 つのパイプラインを含め、スケジュールトリガーを使用する」です。

理由：
1. 依存関係のあるパイプラインを実行する場合、親パイプラインを作成して子パイプラインとして実行順序を制御するのが最適な方法です
2. 8 時間ごとの定期実行が必要なため、スケジュールトリガーが適切です

他の選択肢が不適切な理由：
- 「すべての 4 つのパイプラインにスケジュールトリガーを追加する」：各パイプラインに個別にスケジュールトリガーを設定すると、依存関係の管理ができません
- 「すべての 4 つのパイプラインにイベントトリガーを追加する」：イベントトリガーは外部イベント（BLOB の作成など）に反応するもので、定期実行には適していません
- 「親パイプラインを作成してイベントトリガーを使用する」：親パイプライン構造は正しいですが、定期実行にはイベントトリガーではなくスケジュールトリガーが適切です

| トリガータイプ | 用途 | 依存関係管理 | 定期実行 |
|---|---|---|---|
| スケジュールトリガー | 時間ベースの実行 | ×（単独では不可） | ○ |
| イベントトリガー | 外部イベント発生時の実行 | × （単独では不可） | × |
| 親パイプライン＋スケジュールトリガー | 複雑なワークフロー実行 | ○ | ○ |

Azure Data Factory では、複雑な依存関係を持つパイプラインを管理する場合、親パイプラインを作成し、その中で実行順序を制御することが推奨されています。この方法により、依存関係の管理と定期実行の両方の要件を満たすことができます。

</details>

Azure 仮想マシン上の SQL Server に、DB1 という名前のデータベースがあります。クエリ 1178902 の各実行の所要時間（ミリ秒）を示すプラン概要を確認しています。クエリが最短時間で実行されるプランを使用するようにするには、どうすればよいですか？（一つ選択）

- A. プラン 1220917 のクエリ実行プランを強制する
- B. パラメータスニッフィングを無効にする
- C. DBCC FREEPROCCACHE コマンドを実行する
- D. プラン 1221065 のクエリ実行プランを強制する

<details>
<summary>解答・解説</summary>

**解答:** A

プラン 1220917 を強制することが正解です。これは、問題文の「最短時間で実行される」という要件を満たすためです。プラン概要から、このプランが最も短い実行時間を持つことが示されていると想定されます。

SQL Server では、特定の実行プランを強制するには、クエリストアやプランガイドを使用します。これにより、SQL Server のクエリオプティマイザーが通常選択するプランではなく、指定したプランを使用するよう強制できます。SQL Server のクエリストアを使用して特定のプランを強制するには、以下のようなコマンドを使用します：

```sql
EXEC sp_query_store_force_plan @query_id = 1178902, @plan_id = 1220917;
```

- パラメータスニッフィングを無効にすると、最適なプランが選択されない可能性があります
- DBCC FREEPROCCACHE はキャッシュされたすべてのプランを削除するため、最適なプランも含めて削除されてしまいます
- プラン 1221065 は、プラン 1220917 よりも実行時間が長いと想定されます

| 選択肢 | 説明 | 適切か |
|---|---|---|
| プラン 1220917 を強制 | 最も効率的なプランを明示的に選択する | はい |
| パラメータスニッフィング無効化 | パラメータ値に基づくプラン最適化を防ぐ | いいえ |
| DBCC FREEPROCCACHE | すべてのキャッシュプランを削除し再コンパイルを強制 | いいえ |
| プラン 1221065 を強制 | 最適でないプランを選択する | いいえ |

</details>

## 問題 300

リージョン障害が発生した場合に備えて、Azure SQL Database の高可用性を構成する必要があります。どの 2 つの機能を使用できますか？（二つ選択）

- A. 自動バックアップ
- B. アクティブ geo レプリケーション
- C. 自動フェールオーバー グループ
- D. 長期バックアップ保持
- E. Azure Site Recovery

<details>
<summary>解答・解説</summary>

**解答:** B、C

リージョン障害（地域的な停止）に対する Azure SQL Database の高可用性を確保するには、地理的に分散したレプリケーション機能が必要です。

**アクティブ geo レプリケーション**は、別のリージョンに読み取り可能なセカンダリデータベースを作成し、非同期レプリケーションを使用してプライマリデータベースと同期します。リージョン障害が発生した場合、手動でセカンダリにフェールオーバーすることができます。

**自動フェールオーバー グループ**は、複数のデータベースを一括して別リージョンにレプリケーションし、グループ単位で自動フェールオーバーを提供します。リージョン障害時に自動的にフェールオーバーするため、手動介入なしで高可用性を実現できます。

不正解の選択肢について：
- **自動バックアップ**：リージョン障害からの復旧には使用できません。同一リージョン内でのバックアップのみです
- **長期バックアップ保持**：障害発生時の即時フェールオーバーではなく、長期保存用のバックアップ機能です
- **Azure Site Recovery**：SQL Database ではなく、主に VM やオンプレミスワークロードのディザスタリカバリに使用されます

| 機能 | リージョン障害対応 | フェールオーバー方式 | 用途 |
|---|---|---|---|
| アクティブ geo レプリケーション | ✓ | 手動 | 個別データベースの地理冗長化 |
| 自動フェールオーバー グループ | ✓ | 自動 | 複数データベースの一括地理冗長化 |
| 自動バックアップ | ✗ | N/A | 同一リージョン内のバックアップ |
| 長期バックアップ保持 | ✗ | N/A | 長期保存用バックアップ |
| Azure Site Recovery | ✗ | 手動 | VM・オンプレミス向けディザスタリカバリ |

</details>
